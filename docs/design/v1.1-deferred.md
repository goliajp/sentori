# v1.1 deferred work — 4 features blocked on infra

Three from the original v0.9 roadmap can't be ship-and-Insight-test
without infrastructure changes that don't fit in this autorun:

## #4 — JS Profiling (Hermes sampler)

**Why deferred**: Hermes ships a sampling profiler
(`HermesAPI.startSamplingProfiler` / `dumpSampledTrace`) but accessing
it from JS needs a private RN bridge that varies per RN minor. A
pure-JS `setInterval(scrapeStack, 50ms)` polyfill would only capture
idle-tick stacks, not the busy frames that are usually the perf bug —
worse than useless.

**Current substitute**: `sentori.measureFn(name, fn)` (v0.9.0 #14)
gives the host explicit op-timing for code blocks they suspect.

**v1.1 path**:
1. Vendor Hermes' `HermesAPI.h` header per RN minor matrix (0.76 /
   0.78 / 0.79). Symlink in `ios/` and `android/jni/`.
2. iOS: Swift wrapper around `hermes::HermesRuntime::startSamplingProfiler`
   exposed through `SentoriModule`.
3. Android: JNI bridge to the same Hermes runtime instance.
4. Server: parse Hermes `.cpuprofile` JSON format into a flame-graph
   payload (function → self time / total time tree). Reuse existing
   spans table — add `op = 'sentori.profile'` with `data.flame_graph`
   blob.
5. Dashboard: flame-graph view (horizontal bars, click to drill, top
   "5 self-time hotspots" sidebar). Reuse the existing
   `FrameSourceDrawer` for click-through to source.

Effort: L. Major risk = RN minor matrix breaks across releases —
ship behind a `capture.profile.hermes: true` flag with explicit
RN-version pin in docs.

## #2 — Session Replay (wireframe mode)

**Why deferred**: XL native work. Walking the iOS / Android view
tree at 1 Hz, serializing `{kind, rect, text|maskedText, bgColor,
isImage}` per node, gzipping as NDJSON attachment, then rendering
SVG in the dashboard. Each piece is 200-500 LOC of native code that
needs real-device verification against multiple RN versions.

**Current substitute**: `screenshot` (v0.7.3+) gives a single image
at crash time; `stateSnapshot` (v0.9.2 +S2 + +S2-VIEW) gives the
redux/zustand state diff timeline; `sessionTrail` (v0.7+) gives the
breadcrumb timeline. Together these cover ~70% of replay's use cases
in error-triage workflows.

**v1.1 path**:
1. iOS: `UIView.snapshotView(afterScreenUpdates:)` is too expensive at
   1 Hz; walk the hierarchy + serialize per-node dictionary instead.
   Background thread, mainQ for actual read.
2. Android: identical via `View.getRootView()` + reflection on
   `TextView` etc.
3. Mask consultation: same `registerMaskQuery` consumer registry
   shipped in v0.7.3 — text nodes inside any `nativeID ∈ masked` get
   `text: "***"`.
4. Storage: gzipped NDJSON per session as a single `replay` attachment
   (D5 decision). 60 s ring on-error + continuous at
   `replaysSessionSampleRate`.
5. Dashboard: an `<svg>` viewer that re-rasterizes wireframes
   browser-side. Plays at 1 Hz; scrubbable.

Effort: XL (3+ weeks). Highest impact post-v1.0 — the dashboard
literally has no UI playback today.

## #15 — Unified Mobile Issue View

**Why deferred**: depends on #2 + #4 being complete. Without
wireframe replay and flame-graph profile, "unified view" reduces to
"same multi-tab issue detail page". The 'At a glance' enhancement
that *would* be valuable today is already partially present (Culprit
section above tabs + Mobile Vitals data shows in Stack tag chips).

**Current state**: Issue detail page is dense enough — Stack /
Events / Breadcrumbs / Tags / Activity / User reports tabs +
Culprit section + Attachment gallery + Context pane all on one
scrollable surface. Operator gets a complete picture without
switching pages.

**v1.1 path**: Single-page layout with header (release · bundle ·
device · location · culprit commit) → 4-column row (stack | replay
strip | vitals strip | state time-travel) → 4-column row (trace
waterfall | breadcrumbs | profile flame | feedback). Built on top
of #1 / #2 / #4 / +S2-VIEW / +S3 — all the data sources need to
exist before the layout has anything to show.

Effort: M. Reasonable evening's polish once the data is in place.

## #7 — Android NDK crash capture (real breakpad)

**v0.9.5 ships a stub** (`SentoriNativeOrigin.looksNative`): events
that throw `UnsatisfiedLinkError` / have JNI / .so frames get a
`native_signal: true` tag so the dashboard can split them out. But
pure SIGSEGV in a third-party `.so` that doesn't bubble up as a
Java exception still terminates the process invisibly.

**v1.1 path**:
1. Vendor a tiny `crashpad` build (Chromium's, ~200 KB stripped).
2. Add `CMakeLists.txt` + Gradle `externalNativeBuild` block in the
   SDK's `android/build.gradle`.
3. JNI binding: `Sentori.installNativeCrashHandler()` registers
   crashpad handler before any host C++ libs load.
4. Minidump → `<filesDir>/sentori/pending/<uuid>.dmp`.
5. Reuse the existing drain-on-next-launch path that handles Java
   crashes today — extend to upload minidumps as a new event-shape.
6. Server: `dump_syms` symbolicator (the breakpad project ships one)
   wired in front of the existing dSYM/Proguard symbolicator. New
   admin endpoint for uploading stripped `.so` debug symbol files
   (same shape as `/dsyms` / `/mappings`).

Effort: L. Self-contained from sentori's side; the integration with
host apps is "add `crashpad` aar to your dependencies" — same UX as
crashlytics has used for years.
