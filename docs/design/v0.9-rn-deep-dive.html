Sentori v0.9 — React Native 深度设计

把 Sentori React Native SDK 从"可接受的 v0.7 替代品"推到"RN 领域最好"。立场鲜明，必要时反 Sentry。这份文档分三层：对标行业现状、列出我们要做但他们不会做的原创功能、把这些塞进可执行的 v0.9 → v1.0 路线图，最后让维护者拍 7 个分叉决策。

03

差距分析 / Gap Analysis

已在的位置 · 必须打的仗 · 不打的仗

已并跑 入场资格已有

  • 崩溃 + 面包屑 + 会话 + 无崩溃率。
  • react-navigation Tracing + fetch/XHR Tracing + 分布式 traceparent。
  • Mask 注册表 — 手术刀级，消费者拥有。优于 Sentry 的"遮全部"默认。
  • Source Map + dSYM + Proguard 自动化 + CLI 钩子。
  • 自部署 — docker compose up 完胜 Sentry 的 22 容器参考栈。
  • 包体 — 没塞 Replay / Profiler。(这是特性，不是 bug。)
  • 一行接入 + 扁平命名空间。

明确落后 必须补齐才能上桌

  • 无移动 Vitals — 冷启动直方图、慢/冻帧、TTID/TTFD 全无。客户会逐项对比。
  • 无 Session Replay — 所有正经竞品都有；Dashboard 完全没有播放能力。
  • 无 JS Profiling — Sentry 给火焰图；我们说"看 trace"。
  • 无 NDK — 带 C++ 依赖（图像、ML、编解码）的 RN 应用对我们就是黑洞。
  • 无 Velocity 告警 — 告警只有规则式，没有回归式。
  • 新架构保真度 — Sentry 也有缺口；我们这片是 unknown territory。

可弯道超车 设计宪法帮我们赢

  • Wireframe Replay（PostHog 流派）。Schema 简：不要每会话 MB 级存储。隐私故事好讲。在 Linear 密度下渲染更舒服。
  • Moments / 流失追踪（Embrace 有；Sentry 没）。一行 API sentori.startMoment('checkout').end() → 漏斗 UI。
  • TurboModule 异常修复。补 RCTTurboModule.mm 吞异常那个洞。我们做得到是因为只有一个 SDK + 一个运行时模型。
  • EAS Update / CodePush 感知。每条事件带 bundle id；崩溃高峰自动建议回滚。Sentry release health 是 binary-build 中心。
  • RN-shaped Velocity 告警。"Issue X 在 bundle r/123（灰度 20%）上 30 分钟内激增 5×" → Slack/Linear。
  • 单页 Issue → Replay → Trace → Profile。Sentry 四件都有但 join 松。Schema 简让它们落在同一行，UI 密让它们落在同一屏。

不竞争 明确放弃

  • JS+native 全栈火焰图。挑 JS 线程就够。
  • 通用 OTel 接入面。让 Datadog / Embrace 打这场仗。
  • 巨型集成生态。Slack / Linear / Jira / GitHub 够了。
  • Web SDK 功能追赶 — 这版搁置。RN-first 就是 RN-first。
  • 像素级 Raster Replay。宪法决定的反 Sentry 立场。Wireframe 是我们押的方向。
04

原创创新 / Sentori-Original

竞品 = 0
为什么有这一章 路线图（下一节）讲的是对标行业 + 补齐 + 个别弯道超车。但 sentori 不应止步于"做得跟 Sentry 一样好"。这一节是原创视角：从我们的角度发明的、竞品 0 或半解决的能力。每一项都瞄着"做完之后 Sentori 是市面上唯一一家"。 工程上不全在 v0.9 范围内，但应当被认作 sentori 的产品定义而不是 nice-to-have。
S1 AI 语义聚合 + 根因建议 原创 · 竞品 0 L

Sentry 至今仍用 hash 几行 stack 做 fingerprint。我们站在 LLM 工具链上，自然做语义级聚合：对 stack + 关键 breadcrumbs + 源码上下文做嵌入，"看似无关的 5 条 issue 其实是同一根因" 自动归类。每个新 issue 顶部直接出"建议修复 diff"。

// server-side only, dashboard 顶部新加一块面板:
┌────────────────────────────────────────┐
│ 建议根因: parseRoute 在 deep-link 处 │
│ 未防御 null nextScreen.params       │
│ 语义同类: #1812 #1547 #1209          │
│ 建议 patch: [+12 -4] view diff      │
└────────────────────────────────────────┘
触及 服务端 LLM 调用 · dashboard 差异化 Sentry 的 root cause 是 paid Premium
// 没人做 · 我们做完是市面唯一
S2 State 时光机 原创 · 竞品 0 L

Sentry Replay 只有 UI 截图，没 state。我们记 Redux / Zustand / Context state 快照，与 breadcrumb timeline 同步。点击崩溃前 8 秒任意 breadcrumb → 看当时 整个 app state。这是 RN 调试真正的痛点 — "崩溃时购物车里到底有什么"。

sentori.bindState({
  redux: store,          // 可选
  zustand: useStore,      // 可选
});

// 或手动:
sentori.recordState({ user, cart });
// 每次 breadcrumb fire → diff vs 上次 → 入 ring
触及 SDK state 适配器 · server attachment · dashboard 时光机 viewer
// Sentry replay 只有 UI · 我们有 state
S3 罪魁祸首定位 + Revert PR 原创 · 竞品 0 M

给定 issue 的 throw line + first_seen 时间窗 + git history，算出最可能引入此 bug 的 commit。Issue Detail 顶部出一行"likely culprit · commit a3f7b · @alice · 2h before first crash"，旁边按钮 "生成 revert PR" 走 GitHub 集成。

// dashboard issue detail header 新增:
[likely culprit]
  a3f7b91 @alice · 2h before first crash
  "refactor: simplify deep-link parser"
  [open commit] [generate revert PR]
触及 服务端 git 同步 + blame · GitHub 集成
// 没有竞品做这个 · 仅靠 sentori 服务端就能算
S4 崩溃前哨 (Pre-crash Sentinel) 原创 · 竞品 0 M

所有竞品都是 reactive — 崩了才报。我们做 predictive：客户端订阅内存压力 / 帧预算 / 存储 / 网络抖动，模式识别到"即将崩溃"信号在 crash 之前几秒/几分钟主动报。dashboard 上是一类新事件 kind: nearCrash，独立 timeline 标记。

sentori.init({
  capture: {
    preCrashSentinel: true,
    sentinelChannels: [
      'memory-pressure',
      'frame-budget-overrun',
      'storage-low',
      'oom-warning',
    ],
  },
});
触及 SDK iOS/Android 系统 hook · server 新事件类型 · dashboard 标记
// 所有竞品 = 事后归档 · 我们 = 事前预警
S5 生产事件复刻测试 (Repro-as-Code) 原创 · 竞品 0 M

Issue Detail 顶部一个按钮 "导出为 Jest 测试"。Server 把 breadcrumb sequence + 错误 trigger + state（来自 S2）+ mock 网络响应一起编排成一份 tests/__repros__/issue-N.test.ts。RN 工程师下载、本地跑、断点、修。从 crash 到本地复现 = 30 秒。

// dashboard 输出:
// tests/__repros__/issue-1812.test.ts
test('issue-1812: deep link with null params', async () => {
  mockFetch('/api/route', { status: 200 });
  await act(() => Linking.openURL('app://deep'));
  // expect: parseRoute throws TypeError
});
触及 服务端测试生成 · dashboard 导出 · RN 测试 boilerplate
// 行业第一个 · RN 调试痛点直接命中
S6 隐私实验室 + 合规评分 原创 · 半解决 M

Sentry 有 scrubber，但没有给你打分。我们扫所有入库 event → 找疑似 PII (email / phone / address / cc-like 数字)。每个 release 一个 Privacy Score (0–100) + GDPR / CCPA risk tag。Top "本周 PII 泄漏 surface" 列表 → 一键加 mask 规则。

// dashboard 新 module: Privacy
// release v1.2.3 privacy score
┌──────────────────────────────────────┐
│ score: 68 / 100  risk: MEDIUM     │
│ leaks:                              │
│   12 events 含 email-like                  │
│   3  events 含 phone-like                    │
│   1  breadcrumb 含 cc-like                    │
│ [add mask rule] [export GDPR report]│
└──────────────────────────────────────┘
触及 服务端 PII 扫描 cron · dashboard Privacy 模块
// 给合规团队一个 SaaS-ready dashboard
S7 实时调试流 (Live Debug Stream) 原创 · 竞品 0 L

Sentry Spotlight 只在本地开发能用。我们做 生产级 live debug：dev 在 dashboard 选一个 user → 开 live session → SDK 拿到 ephemeral flag → 进入"无批量、无采样"模式 → 该 user 接下来的所有 event 实时流到 dashboard timeline。10 分钟超时自动关。

// 客户端无新 API,
// 只要 init 时 setUser 过即可:
sentori.setUser({ id: 'u-123' });

// dashboard:
// 找到 u-123 → [start live debug]
// → server 下发 flag
// → SDK 进入 live mode
// → 接下来的所有 events 实时 stream
触及 SDK control channel · server SSE · dashboard live viewer
// 行业第一个 prod-grade 的 live debug
合计 7 项原创 = "做完之后只有 Sentori 一家"

S1 + S2 + S3 + S5 在一起的形态非常有意思 — issue 出来 → AI 推根因 (S1) → 时光机看 crash 时的 state (S2) → blame 出罪魁 commit (S3) → 一键导成本地测试 (S5)。这是一条 sentry 完全做不到的工作流，因为他们 schema 厚、产品分裂。我们 schema 简 + UI 密 + 数据全在同一行，做这条工作流是自然衍生。这是我们真正的 moat。

#1 移动 Vitals B 并跑 L

没有冷启动 + TTID/TTFD + 慢/冻帧 → 进不了 Sentry / Embrace / Datadog 的对比表。客户最常引用的"试了你们工具，发现没 X"缺口。

sentori.init({
  capture: { mobileVitals: true },
});

// 自动无法判断时手动标 TTFD:
const h = sentori.markTimeToFullDisplay('Home');
h.end();
触及 SDK · 原生 iOS+And · 服务端 · dashboard
#3 启动崩溃循环防护 A 差异化 M

Bugsnag 唯一拥有。OTA 更新有 bug 会让 RN 应用陷入不可恢复启动崩溃死循环。便宜建造，触发时戏剧化效果强。

sentori.init({
  capture: {
    launchCrashGuard: {
      enabled: true,
      onLaunchCrashDetected: (info) => ({
        action: 'rollback',
        toBundle: info.lastSafeBundle,
      }),
    },
  },
});
触及 原生 SDK · server
#6 Moments / 流失追踪 A 差异化 M

Embrace 标志性能力；Sentry 没有。Schema 简（一个 span + abandoned 标志位）+ UI 密（漏斗图）。一行 API。

const m = sentori.startMoment('checkout', {
  properties: { cartValue: 42 },
});
m.checkpoint('payment-submitted');
m.end();      // 或 .fail() / .abandon()
触及 SDK · server (span field) · dashboard
#10 EAS Update / CodePush 感知 A 差异化 M

每个生产 RN 应用都发 OTA。Crashlytics 不处理；Sentry release health 是 binary-build 中心。直接的胜利。

sentori.init({
  release: '1.2.3+456',   // store binary
  // 自动从 expo-updates 探测:
  // bundle: { id: 'r/123', deployedAt }
});
触及 SDK · server · dashboard
#5 Velocity 告警 (RN-shaped) B 并跑 M

不是"错误率 > X" — 而是"这条 issue 比 10 分钟前命中更多用户了"。纯服务端能力。

# 项目 velocity 策略
new_issue_in_release:
  notify: @oncall via slack
issue_velocity_3x_in_30m:
  notify: @oncall via slack
issue_velocity_5x_in_30m:
  page: @oncall via pagerduty
触及 server cron · 现有集成
#13 Feature Flag 维度 B 并跑 S

Bugsnag 一个小 API，带来巨大过滤能力。tag 不能区分"实验维度"与"环境状态"。

sentori.setFeatureFlag('checkout-v2', 'variant-a');
sentori.clearFeatureFlag('checkout-v2');
触及 SDK · server (索引字段) · dashboard
#11 GraphQL 操作自动命名 C 锦上添花 S

每个 RN 应用用 Apollo/urql/Relay。所有 GraphQL 请求当前 breadcrumb 是 POST /graphql（无用）。自动提取 operationName。

sentori.init({
  capture: {
    network: { graphql: true },
  },
});
// → breadcrumb: graphql/UpdateCart
触及 SDK 网络 handler · mask 复用
#12 Rage Tap / 连点 C 锦上添花 S

Sentry 8.10 加的。小但向评估者表明"这个 SDK 懂 RN"。与 Replay (#2) 配合更强。

// 在 app root 处:
<sentori.RageTapCapture>
  {children}
</sentori.RageTapCapture>
// 自动: 800ms 内 ≥3 次 → ui.multiClick breadcrumb
触及 SDK gesture 桩
v0.9.1 弯道超车 + sentori 原创首发 · ~5 周 · 6 项
#2 Session Replay (wireframe 模式) A 差异化 XL

最大可信度缺口。Wireframe 而非 raster 是弯道超车：存储更便宜、GDPR 下更易讲、Linear 密度下渲染更舒服。PostHog 已证明模型可行；错误追踪赛道无人触及。Sentry 的 raster: 40 KB 包 + $0.003/replay 存储成本。

sentori.init({
  capture: {
    replay: {
      mode: 'wireframe',          // 'wireframe' | 'off'
      sampleSessionRate: 0.05,    // 连续 stream
      sampleOnErrorRate: 1.0,     // 60s 环形缓冲
      maskText: true,
      maskImages: true,
    },
  },
});
触及 原生 view 树遍历 · server attachment · dashboard SVG renderer 砍掉若赶进度 连续 stream → 仅 on-error 60 秒
#8 TurboModule 异常保真 A 差异化 M

RN 0.76+ 默认新架构。RCTTurboModule.mm 把 NSException 包装成通用 JSError。Sentry 也没修。"市面唯一保真" claim 可写到 marketing 上。

// 无客户面 API.
// iOS: 拦 ObjCTurboModule::performVoidMethodInvocation
// And: TurboModule.invokeMethod 反射代理
// coerceError 给 JS event 接上原生 stack
触及 仅原生 风险 每个 RN minor 可能破

06

决策 / Decisions

7 个分叉 · 维护者拍板 · 路线图随之重排
D1 Replay 形态
A · 仅 wireframe 反 Sentry 立场。更轻、隐私更易讲、发布更快。
B · wireframe + raster 像素级可选。原生渲染复杂度高。
客户若一定要 raster — 不是我们目标客户。如真需要，做付费模块独立 ship。
D2 Profiling
A · Hermes 采样 相比"除 Sentry 外"所有竞品形成差异化。on-slow-frame 默认。
B · 不做 省 L 工作量。失去性能 bug 对比。
on-slow-frame 模式确保常态下生产采样开销基本为零。always-on profiling 是陷阱。
D3 启动崩溃回调
A · 阻塞 ≤200ms Bugsnag 模型。在 JS 跑之前选 rollback / reset / continue。
B · fire-and-forget 更安全。客户在启动后再调 sentori.rollback()。
这个功能的本意就是不让坏 JS 跑起来。不阻塞 = 功能价值减半。硬 200ms 超时 → 默认 continue。
D4 新架构立场
A · 激进 RN 0.77+ 优先。自己的原生模块用 TurboModule。
B · 兼容优先 支持 0.71+ 全功能。开发负担重，速度慢。
2026 新启 RN 应用都在新架构上。Sentry 有存量要拖；我们没有。跟 RN core 同步走。
D5 Replay 存储
A · attachment blob 每会话 gzip NDJSON。复用现 attachment 系统。零迁移。
B · replay_frames 表 帧级查询能力。运维更重。破"部署轻"。
Schema 简。在 event 行保留 frame_count + duration 供过滤即可。不预建查询表。
D6 SaaS 定价角度
A · 免费 replay，按保留期付费 Replay 事件免费；> 7 天保留为付费档。
B · 按 replay 计费 买家熟悉。Sentry 形态。
"我们不是 Sentry" 的故事。Wireframe 便宜 → 卖便宜。与 D1 技术选择对齐。
D7 SDK API 命名空间
A · 扁平 sentori.startMoment · sentori.recordState · sentori.setFeatureFlag …
B · 按域名嵌套 sentori.perf.startMoment · sentori.flags.set …
"一行接入"意味着一个命名空间。结构化的代价 = 放弃宪法原则，代价太高。

风险 & 接下来 / Risks & Next

5 风险 · 4 意外发现 · 1 CTA

路线图级别的风险

  1. 包体膨胀。每个新功能都是 JS + 原生。做到 #15 SDK 可能翻倍。缓解：每项功能必须可通过 capture.* 关闭，tree-shakeable。v0.9.0 完后审一遍。
  2. 新架构跑步机。RN minor 版本每 ~2 个季度破一次 TurboModule 内部。0.76 / 0.78 / 0.79 跑 CI 矩阵；pin known-good。
  3. dogfood 饱和。Insight 一个客户撑 v0.9，v1.0 没有第二 canary。在 v1.0 之前需要排上第二个 RN dogfood 项目。
  4. Dashboard 密度上限。UI 密 + 4 种新数据类型 + 7 项原创 → 可能变 "Bloomberg Terminal"。#15 统一 issue 视图非谈判。
  5. 服务端范围悄悄膨胀。一半功能触及服务端。"RN 深度设计"标题掩盖了这同时是服务端路线图。明确承认它。
