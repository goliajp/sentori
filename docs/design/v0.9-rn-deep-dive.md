# v0.9 — RN Deep Dive

> Roadmap design for the Sentori React Native SDK: moving from "credible v0.7 alternative" to "best-in-class for RN".
>
> Status: design draft. Author: research pass 2026-05.
> Anchor versions during research: `@sentry/react-native` 8.11.x · `@embrace-io/react-native` 6.5.x · `@bugsnag/react-native` 8.3.x · `@datadog/mobile-react-native` (post-2026 major) · `posthog-react-native-session-replay` 3.2+ · React Native 0.77–0.79 (New Architecture default).

This doc looks at where the RN observability market is in mid-2026, where Sentori already has parity, where it is plainly behind, and where it can leapfrog by leaning on the project's design constitution: **schema 简, UI 密, 部署轻, 一行接入**. The output is an ordered v0.9 → v1.0 feature backlog plus the forks the user needs to call.

The doc is opinionated. Sentori does not have the headcount to clone Sentry; the win condition is being the RN-shaped product Sentry has to retrofit into a generalist platform. Every recommendation below is filtered through that.

---

## 1. Competitor landscape (mid-2026)

### 1.1 Sentry React Native (`@sentry/react-native` 8.11.x)

The default reference. Eight years of RN-specific bugfixes layered on a generalist platform. As of 8.11 the surface is:

- **Crash**: JS, iOS NSException, Android JVM/NDK via sentry-cocoa / sentry-android wrappers. New Architecture (Fabric/TurboModules) has known rough edges — `RCTTurboModule.mm` swallows NSExceptions and re-throws them as Hermes `JSError`, which surfaces in their pipeline as `__pthread_kill` or noisy Hermes frames ([GH discussion 276](https://github.com/reactwg/react-native-new-architecture/discussions/276)). Even Sentry doesn't have this fully solved.
- **Replay**: `Sentry.mobileReplayIntegration()`. Native-driven. iOS walks the view hierarchy and rasterizes. Android offers two strategies — `PixelCopy` (cheaper, less mask-reliable) and `Canvas` (custom redraw, always masks). Default frame rate 1 fps, only when something changes. `replaysSessionSampleRate` (continuous) + `replaysOnErrorSampleRate` (60 s ring, flushed on error). Bundle cost ≈ 40 KB compressed on Android if not excluded ([Sentry replay overhead](https://docs.sentry.io/platforms/react-native/session-replay/performance-overhead/)). All text/images/webviews masked by default; Canvas strategy *always* masks regardless of override.
- **Profiling**: Hermes JS-thread sampling + platform profilers (Swift/ObjC + Kotlin/Java). Gated behind `profilesSampleRate` *relative to* `tracesSampleRate`. UI Profiling experimental ([Sentry RN profiling](https://docs.sentry.io/platforms/react-native/profiling/)).
- **Tracing**: app-start, react-navigation v4+, react-native-navigation, XHR/fetch, touch+gesture interactions, slow/frozen frames, JS loop-stall, React Profiler component renders. TTID auto on react-navigation v6+; TTFD requires manual `Sentry.startTimeToFullDisplaySpan` + `.end()`. Known fragility on TabNavigator route preloading ([sentry-rn#3786](https://github.com/getsentry/sentry-react-native/issues/3786)).
- **Mobile Vitals**: slow frames (>16.67 ms), frozen frames (>700 ms), frame delay hang time, cold/warm start. Aggregated dashboard-side.
- **User feedback**: 8.9+ ships `GlobalErrorBoundary` and a `feedbackIntegration()` form (`showFeedbackForm`). When the form opens it grabs the last 30 s of replay and bundles it ([Sentry user feedback](https://docs.sentry.io/platforms/react-native/user-feedback/)).
- **Rage tap / multi-click**: 8.10 added `ui.multiClick` breadcrumbs that surface on the replay timeline.
- **Source maps**: dedicated `@sentry/expo-upload-sourcemaps` (8.9+) plus environment-driven `SENTRY_RELEASE` / `SENTRY_DIST`.
- **Offline**: platform-side caching with retry.

**Killer feature**: replay + profiling + traces stitched in one issue page.
**Friction**: bundle weight (replay alone ≈ 40 KB compressed; full SDK considerably more), pricing opacity (50 replays/month free then $0.003/replay scaling), and the New Architecture exception handling gap they haven't shipped a fix for.

### 1.2 Embrace.io (`@embrace-io/react-native` 6.5)

The product most explicitly designed for RN. As of v6 the SDK is fully [built on OpenTelemetry](https://embrace.io/blog/embrace-react-native-sdk-now-fully-built-on-opentelemetry/).

- Packages split: `react-native`, `react-native-redux`, `react-native-navigation`, `react-native-tracer-provider`, `react-native-otlp`.
- **"Moments"** — stopwatch + abandonment tracking. Same primitive that powers app-launch perf measurement; you can wrap a user action ("add to cart") and Embrace tells you both p50/p95 and how many sessions *abandoned* mid-moment.
- **Session Play-by-play** — every error, network call, screen change is laid out on a single horizontal timeline. Not a video replay; a structured timeline.
- Tracking: navigation, component mount/unmount, Redux actions, orientation, render errors, unhandled promise rejections, OTLP export, network span forwarding to backend.
- Session metadata API for user identification & tagging.

**Killer feature**: moments + abandonment is something Sentry genuinely doesn't have a direct analog for. Session timeline is denser than Sentry's session-replay-or-nothing.
**Friction**: closed-source backend, enterprise sales motion, no self-host story. Pricing not public.

### 1.3 Bugsnag (`@bugsnag/react-native` 8.3.x)

The conservative, stable competitor.

- **Launch crash detection** — if the app crashes during startup, Bugsnag flags it specifically; the SDK exposes a hook so you can disable risky features / reset state on next launch to break the crash loop. This is a real RN pain point that Sentory, Sentori, and Crashlytics don't directly handle.
- **Stability scores** based on crash-free sessions.
- **Feature flags** as report dimensions (which experiment was active when this crashed?).
- Performance Spans (newly added 2026): native+JS app-start coverage.
- Native side via `bugsnag-android@6.12.1` + `bugsnag-cocoa@6.32.2` — early capture before JS bridge init.

**Killer feature**: launch-crash loop detection + feature-flag dimensionality. Both are signal-dense, low-overhead, and uncommon.
**Friction**: dashboard feels dated; no replay; perf spans are catching up to Sentry but not leading.

### 1.4 Firebase Crashlytics + `@react-native-firebase/crashlytics`

The default for "I just want crashes, free".

- **Velocity alerts** (2026 update): triggers when a single issue affects a percentage of users in 30 minutes; thresholds now require both min user count and min impact percentage; ANR velocity supported.
- Native and JS error capture, unhandled promise rejections.
- Cheap (free at moderate scale), Google-grade infra.

**Killer feature**: velocity alerting is genuinely good and Sentori's alerting story is thin.
**Friction**: locked into Firebase project shape, no perf/tracing in the same pane, can't self-host, no source-map automation for RN beyond what the user wires by hand. Worst integration friction of the bunch.

### 1.5 Datadog RUM React Native

- OpenTelemetry, distributed trace header propagation, 20% default resource trace sample rate.
- 2026 major release added an `@datadog/mobile-react-native-openfeature` package — local feature flag eval + automatic RUM exposure tracking.
- Strong on backend-trace ↔ mobile-trace correlation through `firstPartyHosts`.

**Killer feature**: backend trace correlation when the customer is already on Datadog APM.
**Friction**: priced for enterprises; install path is heavy; SDK is a wrapper around the generic mobile SDK and feels less RN-native than Sentry or Embrace.

### 1.6 Instabug → Luciq (archived Jan 2026)

The repo was [archived 2026-01-28](https://github.com/Instabug/Instabug-React-Native) and rebranded to Luciq. Historically the user-feedback-first product: shake-to-report, session replay (full screenshot stream), console logs, network captures, user steps. Still relevant as a feature reference even though the lineage is interrupted.

### 1.7 PostHog (`posthog-react-native-session-replay` 3.2+)

Replay-first, analytics-first; not crash-first.

- **Wireframe replay**: view hierarchy → JSON → HTML wireframe at render time. Doesn't look like the real UI but is cheap and PII-safe by default.
- Optional screenshot mode (explicit opt-in, masking required).
- Captures touches, network, logs.
- Android API 26+, iOS 13+, Expo Go not supported.

**Killer feature**: wireframe replay's PII story is fundamentally easier to explain to compliance than rasterized screenshots.
**Friction**: product analytics platform, not error platform — error/crash UX is thin.

## 3. Gap analysis + differentiation thesis

### 3.1 Where we already have parity

- Crash + breadcrumbs + sessions + crash-free rate.
- react-navigation tracing + fetch/XHR tracing.
- Mask registry — actually *better* than Sentry's "mask everything" default for teams that want to be surgical.
- Source maps + dSYM + Proguard.
- Self-host story (`docker compose up` beats Sentry's 22-container reference stack hands-down).
- Bundle size (no replay/profiler embedded today — this is a feature, not a bug).
- One-line init.

### 3.2 Where we're plainly behind (must close to be credible)

These hurt in head-to-head sales / dogfood:

1. **No mobile vitals** — Sentry, Embrace, Datadog all show cold-start histograms, slow/frozen frames, TTID/TTFD. Customers compare. We currently can't.
2. **No session replay in any form** — every serious competitor has one. The dashboard literally has no playback.
3. **No JS profiling** — when an Insight performance bug comes in we say "look at the trace"; Sentry says "look at the flame graph".
4. **No NDK** — RN apps with native C++ deps (image processing, ML, codec) crash invisibly for us today.
5. **No velocity alerts** — Crashlytics, Embrace, Datadog all wake people up; Sentori's alerting is rule-based, not regression-based.
6. **New Architecture exception fidelity** — even Sentry has gaps here, but ours are unknown territory. Every RN app shipping in 2026 is on the New Architecture.

### 3.3 Where we can leapfrog (the design constitution working for us)

This is the interesting part — places where being late + small + opinionated wins:

- **Wireframe replay (PostHog-style) instead of rasterized stream.** Schema 简: no megabytes-per-session storage. Privacy story is fundamentally easier to defend. Renders in dense Linear-style UI better than fuzzy 1-fps screenshots. We can ship this in a fraction of the time it would take to clone Sentry's native replay stacks.
- **Moments / abandonment.** Embrace has this; Sentry doesn't. A `sentori.startMoment('checkout').end()` API is one line and dashboard-renders as a funnel — playing directly to UI 密.
- **TurboModule exception hardening.** Patch the `RCTTurboModule.mm` swallowed-NSException gap properly. We can do this because we own one SDK and one runtime model; Sentry has to ship it across all of mobile.
- **EAS Update / CodePush awareness.** Tag every event with the OTA bundle id; show crash-spike-by-bundle-id in the dashboard; auto-correlate "deployed update X 12 minutes ago, crash rate 4×'d, suggest rollback". Crashlytics doesn't bundle this; Sentry's release health is binary-build-centric.
- **Velocity alerts that are RN-shaped.** Don't just port Crashlytics. "Issue X spiked 5× in the last 30 min on bundle `r/123` (rollout 20%)" → one-tap channel out to Slack/Linear (we already have those integrations).
- **A single-binary issue→replay→trace→profile join.** Sentry has all four but the join is loose (separate products with cross-links). With Sentori's schema 简, all four live on the same `event` row. UI 密 → one screen, not three tabs hidden behind menus.
- **No-config feedback widget.** Sentry shipped one in 8.9; Embrace has one; we have an API but no widget. Ship a one-import RN component with a shake-to-open default.

### 3.4 Where we explicitly should NOT compete

- **Full feature-parity profiler** with Sentry's flame graphs across JS+native. Pick JS thread only.
- **Generic OTel surface** for non-RN consumers — let competitors fight that battle.
- **Massive ecosystem of integrations** (Datadog has hundreds). Pick the ones an RN team actually uses: Slack, Linear, Jira (later), GitHub.
- **Web SDK feature catch-up** in this doc. See [§7](#7-deferred-and-out-of-scope).
- **Replay-as-pixel-perfect-video**. We're explicitly anti-Sentry on this — wireframe is the bet.

## 4. v0.9 → v1.0 roadmap

15 features. Ranked by composite priority (impact × differentiation ÷ effort).

Effort: S ≤ 1 day · M 1–3 days · L 1–2 weeks · XL 3+ weeks.
Grade: 🅐 differentiator, 🅑 parity, 🅒 nice-to-have.

### #1 — Mobile Vitals (cold start + TTID/TTFD + slow/frozen frames) 🅑 — Effort: L

**Why now.** Without these we can't sit at the same table as Sentry, Embrace, Datadog in an RN evaluation. It's the most-cited "I tried your tool and it didn't have X" gap from the customer side.

**API sketch.**

```ts
sentori.init({
  token, release,
  capture: {
    mobileVitals: true, // default true
  },
});

// Manual TTFD when auto can't tell:
const handle = sentori.markTimeToFullDisplay('Home');
// ...data loaded, image rendered...
handle.end();
```

Auto: `useTraceNavigation` already wraps react-navigation. Extend it to emit `nav.ttid` from `navigation.dispatch` to first frame after `componentDidMount` of the new route. App-start: iOS `mach_absolute_time` between `applicationDidFinishLaunching` and first JS bridge ready; Android `Process.getStartElapsedRealtime()` to first `ReactRootView.onAttachedToWindow`. Slow/frozen frames: hook `CADisplayLink` (iOS) + `Choreographer.FrameCallback` (Android), count frames > 16.67 ms (slow) and > 700 ms (frozen) per route span.

**Technical approach.** SDK: 1 new iOS native class + 1 Android. JS side: extend `navigation.ts` to start/end TTID spans automatically. Server: add `vitals` field to event/trace; aggregation buckets per route per release in a new `mobile_vitals` table or by extending `span_attrs` index. Dashboard: new "Mobile" tab under each release with p50/p75/p95 cold-start + frame chart.

**Deps.** None.

**Risks.** Auto-TTID is fragile under TabNavigator preloading (Sentry has open issues). Mitigation: ship auto + always offer manual override; document.

**Skip if rushed.** Auto-TTFD. Ship auto-TTID and manual TTFD only.

### #2 — Session Replay (wireframe mode) 🅐 — Effort: XL

**Why now.** Largest credibility gap. Wireframe rather than raster is the leapfrog: cheaper to store, easier to ship under GDPR, denser to render in UI. PostHog proved the model works; nobody in the *error-tracking* segment has shipped it. Sentry's raster replay is fundamentally heavier (40 KB compressed bundle, $0.003/replay storage cost).

**API sketch.**

```ts
sentori.init({
  capture: {
    replay: {
      mode: 'wireframe',           // 'wireframe' | 'off' (no raster in v0.9)
      sampleSessionRate: 0.05,     // continuous wireframe stream
      sampleOnErrorRate: 1.0,      // 60 s rolling buffer flushed on capture
      maskText: true,              // default
      maskImages: true,            // default
    },
  },
});
```

Dashboard route: `/issues/:id/replay` — left side timeline (already partly built for session-trail), right side wireframe re-render. Scrubbable.

**Technical approach.**

- iOS: `UIView.snapshotView(afterScreenUpdates:)` is too expensive at frequency; instead walk the view hierarchy and serialize `{kind, rect, text|maskedText, bgColor, isImage}` per node into newline-delimited JSON. ≈ 1 Hz on background thread.
- Android: identical via `View.getRootView()` + reflection on `TextView` etc.
- Server: new `replay_frames` table partitioned by `event_id`. Or — given schema 简 — store as a single attachment blob per session with newline-delimited JSON, gzipped. Probably the latter.
- Dashboard: an `<svg>`-based renderer that re-rasterizes wireframes browser-side. Linear-style aesthetic falls out naturally because we're rendering rectangles + monospace text.
- Buffer mode: ring buffer of last 60 s of frames in JS memory; on `captureException` (or `feedback`) flush to server.

**Deps.** Mask registry (already shipped). Native module bridge (already there).

**Risks.** XL effort, the biggest one in this list. Risk of bikeshedding wireframe fidelity vs raster. Risk of mask-leak on first ship (a `<TextInput>` we missed). Phase the ship: alpha behind flag, dogfood Insight, then default to off until Phase 50.

**Skip if rushed.** Continuous stream — only ship the on-error 60 s buffer. That alone closes 70% of the gap.

### #3 — Launch-crash loop guard 🅐 — Effort: M

**Why now.** Bugsnag-only feature today. RN apps with bad OTA updates or migration bugs ship into launch-crash death-spirals that are invisible if you can't tell "crashed before JS bridge initialized" from "crashed during normal use". Cheap to build, dramatic when it fires.

**API sketch.**

```ts
sentori.init({
  capture: {
    launchCrashGuard: {
      enabled: true,
      onLaunchCrashDetected: (info) => {
        // info.crashedBundle: 'r/123'
        // info.consecutiveCount: 2
        // info.lastSafeBundle: 'r/120'
        // Return action:
        return { action: 'rollback', toBundle: info.lastSafeBundle };
        //  or: { action: 'reset', clearKeys: ['user', 'cache'] };
        //  or: { action: 'continue' };
      },
    },
  },
});
```

**Technical approach.**

- Native: write `launch_marker.json` (epoch + release + bundle id) at *very* early stage of native init. Write `launch_completed.json` after JS bridge first event. If the next launch finds marker present but no completed sibling → previous launch crashed before completion. Increment counter. ≥ 2 → fire callback.
- Integrates with EAS Update (#10): if launchCrashGuard's `info.crashedBundle` is an OTA bundle and `lastSafeBundle` exists, action `rollback` triggers `expo-updates.reloadAsync(lastSafeBundle)`.
- Server gets `launch.consecutiveCount` on the next event after recovery, surfaces as a per-issue "launch-crash" badge in dashboard.

**Deps.** None for v0 (full power needs #10).

**Risks.** Side-channels — if the user wants the callback async, we have to block native init until they answer. Mitigation: callback must return synchronously within 200 ms or we default to `continue`.

**Skip if rushed.** Skip the callback API; just *detect* and tag events with `launch_crash_count`, dashboard surfaces it. Cuts effort to S.

### #4 — JS-thread profiling (Hermes sampler) 🅐 — Effort: L

**Why now.** Customers ask for it after they look at Sentry. With Hermes, sampling is cheap and we don't need native CPU profiling to deliver the headline value (95% of RN perf bugs are JS-thread bugs). Frame the product differently from Sentry: don't compete on flame-graph beauty, compete on **"this slow render → this exact function call"** auto-correlation.

**API sketch.**

```ts
sentori.init({
  capture: {
    profile: {
      mode: 'on-slow-frame',  // 'always' | 'on-slow-frame' | 'off'
      sampleHz: 100,
    },
  },
});
```

Dashboard: when an issue has slow frames *and* a profile attached, the issue detail shows the call stack distribution of the frames during the slow render. Single click "go to function in source viewer".

**Technical approach.** Hermes ships a sampling profiler (`HermesAPI.startSamplingProfiler` / `dumpSampledTrace`). Wrap it from the SDK; on slow-frame detection (#1), grab the last N seconds of samples, dedupe by stack, attach to the next event. Server: parse Hermes profile format into a flame graph payload. Dashboard: existing stack viewer extended with horizontal-bar profile view.

**Deps.** Mobile vitals (#1) to know when to *trigger* the on-slow-frame mode.

**Risks.** Hermes profiler API surface differs across RN versions. Mitigation: ship behind feature flag, test on 0.76 / 0.78 / 0.79.

**Skip if rushed.** Skip flame-graph rendering; ship just the raw `samples[]` attachment and dashboard "download .cpuprofile" button. ChromeDevTools opens it.

### #5 — Velocity alerts (RN-shaped) 🅑 — Effort: M

**Why now.** Crashlytics has this; we don't. It's the alerting feature users actually want — not "alert me when error rate > X" but "alert me when *this issue* is suddenly hitting more users than it was 10 minutes ago".

**API sketch.** Server + dashboard only, no SDK change. Dashboard adds a "Velocity" tab on each issue + project-level "Velocity alerts" config.

```yaml
# project velocity policy (editable in dashboard)
new_issue_in_release: notify @oncall via slack
issue_velocity_3x_in_30m: notify @oncall via slack
issue_velocity_5x_in_30m: page @oncall via pagerduty
issue_first_seen_after_release: notify #releases
```

**Technical approach.** Server: a job runs every 5 min, computes (count_30m_now / count_30m_prev) per issue, filters above thresholds. Reuse existing Slack/Linear integration. Add bundle-id awareness (#10) so the alert message says "spiked since OTA bundle `r/123` shipped".

**Deps.** Helped by #10.

**Risks.** False positives during low-traffic windows. Mitigation: require min absolute count (e.g. ≥ 20 events in window) before alert fires.

**Skip if rushed.** Just ship new-issue-in-release alert. M → S.

### #6 — Moments / abandonment 🅐 — Effort: M

**Why now.** Embrace's headline differentiator that Sentry doesn't have. Plays directly to schema 简 (it's literally `span_start` + `span_end` + abandoned bit) and UI 密 (renders as a funnel).

**API sketch.**

```ts
const m = sentori.startMoment('checkout', { properties: { cartValue: 42 } });
// ...
m.checkpoint('payment-submitted');
// ...
m.end();         // or m.fail('declined') / m.abandon()
```

If app backgrounds for > 30 s before `.end()`, auto-`abandon`.

Dashboard: new "Moments" view, lists each moment with p50/p95 duration + abandonment rate, segmentable by release / device / user.

**Technical approach.** SDK: a thin layer over the existing span system. Server: span already has `status` field; add `abandoned` value and a new `moment_name` indexed column on spans. Dashboard: SQL aggregation.

**Deps.** None.

**Risks.** Abandonment heuristic (background-without-end) misses force-killed apps. Mitigation: also count "app launched fresh without prior moment-end" as abandonment for active moments.

**Skip if rushed.** Skip checkpoints, ship `startMoment` + `end` + `abandon` only. M → S.

### #7 — Android NDK crash capture 🅑 — Effort: L

**Why now.** Big RN apps with native deps (image processing, ML, codecs) lose ≈ 10-20% of crashes to us today. We can't credibly say "RN-first" if we miss the C++ side.

**API sketch.** No customer-facing API. Auto on, opt-out.

**Technical approach.** Bundle `breakpad` or `crashpad` mini-dumps. On `SIGSEGV`/`SIGABRT`/`SIGBUS`/`SIGILL`, write minidump to `<filesDir>/sentori/pending/`. Drain-on-next-launch path already exists for Java crashes; extend to upload minidumps. Server: a new symbolicator pipeline using `breakpad`'s `dump_syms` for `.so` files (uploaded same way as dSYMs today).

**Deps.** None.

**Risks.** Signal-handler reentrancy bugs are nasty; risk of corrupting the legitimate crash path. Mitigation: chain-call to any existing handler, write minidump to pre-allocated buffer, no malloc in signal context.

**Skip if rushed.** Capture signal + tag the event with `native_signal`; skip the minidump entirely. Customer at least knows it was a native crash, even if they can't see the stack. L → M.

### #8 — TurboModule / Fabric exception fidelity 🅐 — Effort: M

**Why now.** New Architecture is default in RN 0.76+. Currently every NSException thrown from a TurboModule gets wrapped into a generic `JSError` by `RCTTurboModule.mm` and surfaces in our pipeline as either a Hermes crash or `__pthread_kill` ([sentry-rn known issue](https://github.com/reactwg/react-native-new-architecture/discussions/276)). Even Sentry hasn't fixed this. Fixing it is high-signal, low-volume, and very technical-marketing-friendly ("Sentori is the only RN SDK that preserves TurboModule exception types in 2026").

**API sketch.** No customer-facing API.

**Technical approach.** Native iOS: swizzle / patch `ObjCTurboModule::performVoidMethodInvocation` to capture the NSException + native call stack *before* RN wraps it. Stash in a per-thread holder. When the JS-side error then arrives, our existing `coerceError` looks for that holder and attaches the native stack to the JS event.

Android: equivalent via `TurboModule.invokeMethod` interception (Reflection + dynamic proxy on the TurboModule registry).

**Deps.** None.

**Risks.** Tied to RN internals — risk of breaking on every RN minor. Mitigation: dispatch table by RN version + clear fallback to current behavior.

**Skip if rushed.** Ship iOS only — Android side is harder. M → S.

### #9 — User feedback widget (`<sentori.FeedbackButton />`) 🅑 — Effort: M

**Why now.** We have `sendUserFeedback()` but no widget. Sentry just shipped one in 8.9; Embrace has one. One-import drop-in. Plays to 一行接入.

**API sketch.**

```tsx
import { FeedbackButton } from '@goliapkg/sentori-react-native';

<FeedbackButton
  trigger="shake"           // 'shake' | 'manual' | 'fab'
  attachScreenshot
  attachReplayBuffer        // ties to #2's 60 s buffer
/>
```

When opened: prompt + free-text + email field + optional screenshot. Submit fires `sendUserFeedback` and bundles the replay buffer if available.

**Technical approach.** Pure JS RN component. Shake detection via `react-native-shake` peer dep or our own `accelerometer` integration. Screenshot already shipped. Replay buffer already in #2.

**Deps.** Helped by #2 (replay) but not required.

**Risks.** Aesthetic risk — every feedback widget looks default. Use design-tokens.md, render in Linear-ish style.

**Skip if rushed.** Skip shake detection, ship `trigger="manual"` only (consumer calls `feedbackRef.current.open()`). M → S.

### #10 — EAS Update / CodePush awareness 🅐 — Effort: M

**Why now.** Every Insight-class production RN app ships OTAs. Crashlytics does not handle this; Sentry's release health is binary-build-centric. Direct win.

**API sketch.**

```ts
  release: '1.2.3+456',     // store binary
  // auto-detected from expo-updates if installed:
  // bundle: { id: 'r/123', deployedAt: '2026-05-15T...' }

```

Auto-detected via `Updates.updateId` from `expo-updates` if peer dep is installed; otherwise consumer can pass `bundle: { id, deployedAt }` manually for CodePush.

Dashboard: filter all views (issues, traces, vitals) by bundle id. New "Releases" page splits store binary into nested bundle list; per-bundle crash-free %.

**Technical approach.** SDK: optional `expo-updates` peer import. Server: add `bundle_id` to events / spans (already easily-indexed). Dashboard: filter wiring. Velocity alerts (#5) gets bundle-id awareness.

**Deps.** None for SDK side. Velocity (#5) consumes it.

**Risks.** Two competing OTA worlds (Expo EAS + CodePush self-hosted post-App-Center). Detection logic has to handle both.

**Skip if rushed.** Ship as a manual `bundle` field; skip auto-detect. M → S.

### #11 — GraphQL operation auto-naming 🅒 — Effort: S

**Why now.** Every RN app uses GraphQL via Apollo / urql / Relay. Currently all GraphQL requests are breadcrumbs as `POST /graphql` (useless). Auto-extract `operationName` from the body and turn it into a span.

**API sketch.** Auto if enabled.

```ts
sentori.init({
  capture: {
    network: {
      graphql: true,   // default true; parses operationName
    },
  },
});
```

**Technical approach.** In `handlers/network.ts`, on POST to URLs matching `*graphql*` or content-type `application/graphql`, parse body's `operationName` field. Span becomes `graphql/UpdateCart` instead of `POST /graphql`. Body kept under `gql.operation`. Variable values redacted via mask.

**Deps.** Mask registry.

**Risks.** Parsing user-typed JSON in the network hot path. Mitigation: only check small bodies (< 8 KB), short-circuit on parse failure.

**Skip if rushed.** None — this is already small.

## 5. Decisions

The roadmap hides forks. The user picks; the picks shape every "S vs M" estimate above.

### D1. Replay: wireframe-only or wireframe + raster?

- **Option A — Wireframe only (recommendation).** Ship #2 as wireframe-only. Anti-Sentry stance, lighter, better privacy story, faster to ship.
- **Option B — Wireframe primary + opt-in raster.** Adds significant native rendering complexity for 1-fps screenshots. Larger SDK, harder ATT/privacy story, but matches Sentry on the "I want pixel-perfect" question that compliance teams sometimes ask.

**Recommendation: A.** It's the constitutional choice. If a customer demands raster they're not our customer. If we ever need raster, ship as a separate paid module.

### D2. Profiling: Hermes sampler or skip entirely?

- **Option A — Hermes sampler (#4) (recommendation).** L effort, real differentiation vs everyone-but-Sentry.
- **Option B — Skip profiling, double down on traces + slow-frame breadcrumbs.** Saves L effort. Loses the perf-bug evaluation comparison.

**Recommendation: A**, *but* gated behind `on-slow-frame` mode by default so the JS thread isn't sampled in steady state. Production overhead must be effectively zero in the common case.

### D3. Launch-crash callback: blocking or fire-and-forget?

- **Option A — Synchronous callback (≤200 ms) (recommendation).** Lets the customer pick rollback / reset / continue *before* JS runs. Bugsnag's model. Powerful but risky.
- **Option B — Fire after JS boots.** Safer. Customer can't actually break the loop on the same launch; they can call `sentori.rollback()` once they're up.

**Recommendation: A.** Blocking but with a hard 200 ms timeout → default `continue`. The whole point is to *not* run the broken JS. If we're not willing to block, the feature is worth half as much.

### D4. New Architecture support: aggressive or compatibility-first?

- **Option A — Aggressive (recommendation).** Target RN 0.77+ first; older arch is best-effort. Lean on TurboModules for our own native modules (better perf, cleaner code).
- **Option B — Compatibility-first.** Support 0.71+ at parity. More dev effort, lower velocity.

**Recommendation: A.** Every greenfield RN app in 2026 is on the New Architecture. Sentory has to support both because of their installed base; Sentori has no installed base to drag.

### D5. Replay storage: per-event blob attachment or first-class table?

- **Option A — Gzipped NDJSON attachment per session (recommendation).** Schema 简: zero new tables. Reuse the attachment subsystem we already have. ~50 KB / minute typical.
- **Option B — First-class `replay_frames` partitioned table.** Better for querying ("show me all replays where TextInput X was tapped"). Heavier ops, more migrations, breaks 部署轻.

**Recommendation: A**, with the caveat that we keep frame-level metadata (frame count, duration) on the event row for filtering. If we ever need frame-level querying, table comes later. Don't pre-build.

### D6. Pricing / SaaS positioning angle

Not technical but informs scope. Sentry's replay is $0.003/replay. Embrace is "talk to sales". We will be self-host-friendly forever (constitution). For SaaS, do we:

- **Option A — Free replay, paid retention (recommendation).** Replay events are free; data retention beyond 7 days is the paid tier. Plays to 部署轻 (our marginal cost on a wireframe is tiny).
- **Option B — Sentry-like per-replay pricing.** Familiar to buyers, easier to forecast revenue.

**Recommendation: A.** It's the simpler, more defensible "we are not Sentry" story. Aligns with the wireframe technical choice — wireframes are cheap, so price them cheap.

### D7. New SDK API: extend `sentori.*` or create `sentori.mobile.*`?

15 new APIs ahead. The current namespace is flat (`sentori.init`, `sentori.captureException`, ...). Adding `sentori.startMoment`, `sentori.markTimeToFullDisplay`, `sentori.setFeatureFlag`, `sentori.measureFn`, `<sentori.FeedbackButton />`, `<sentori.RageTapCapture />` ... starts to crowd.

- **Option A — Stay flat (recommendation).** `sentori.startMoment` etc. One-line autocomplete remains the killer feature. 一行接入 means *one global namespace*.
- **Option B — Nest by domain.** `sentori.perf.startMoment`, `sentori.flags.set`. Cleaner taxonomy but breaks autocomplete intuition.

**Recommendation: A.** Stay flat. The price of organization is dropping the constitutional one-namespace principle, and that price is too high.

## 6. Risks the roadmap collectively carries

Independent of any single feature:

1. **Bundle size creep.** Every new feature is JS + native code. By #15 we could double SDK size. Mitigation: make every new feature opt-in via `capture.*` flags, tree-shakeable. Re-audit after v0.9.0 ships.
2. **New-Architecture treadmill.** RN minor versions break TurboModule internals every ~2 quarters. Mitigation: CI matrix for 0.76, 0.78, 0.79; pin known-good and document.
3. **Dogfood saturation.** Insight is one app. Once it's on v0.9, we have no second canary for v1.0. Mitigation: ship the v0.9 SDK to one or two other RN dogfood projects (`marketing/` if it has an RN demo? Talk to user about lining up).
4. **Dashboard density limit.** UI 密 + 4 new data types risks "Bloomberg Terminal" backfire. Mitigation: #15 is non-negotiable; ship the unified mobile view, not just more tabs.
5. **Server scope quietly expanding.** Half the features above touch the server. The "RN deep-dive" framing hides that this is also a server roadmap. Acknowledge it. Don't let v0.9 hit week 8 because the server team is doing flame-graph parsing.

---

## 7. Deferred / out of scope

- **Web SDK parity** (`@goliapkg/sentori-react`, `-vue`, `-svelte`, `-solid`). Tracked separately. They benefit from server-side improvements (#5 velocity, #13 flags, #10 OTA-equivalents) automatically. Replay and mobile vitals don't apply.
- **Generic OTel ingestion** for non-Sentori-SDK clients. Embrace + Datadog have it; not our segment.
- **Datadog-style backend trace correlation** (`firstPartyHosts` magic). Possible later via our existing distributed-trace header. Not a v0.9 priority.
- **Pixel-perfect raster replay.** Constitutional choice — see D1. Probably never.
- **Generic mobile SDK** (Swift / Kotlin standalone). Out of scope; we are RN-shaped.
