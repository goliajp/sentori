## What you need to do (admin access required)

### 1. Environment variable

Update `SENTORI_BASE_URL` on the production lx64 server. Edit
the `.env` (or docker-compose env section) that ships with
`/apps/sentori/docker-compose.yml`:

```
- SENTORI_BASE_URL=https://app.sentori.golia.jp   # OLD
+ SENTORI_BASE_URL=https://sentori.golia.jp       # NEW
```

This is consumed by:
- OAuth callback URLs sent to Google / GitHub
- Verification + invite + transfer + reset-password email links
- Integration OAuth callbacks (Linear, GitHub-issues, Jira, etc.)

Restart the server container after editing:

```bash
ssh lx64
cd /apps/sentori
docker compose --env-file .env -f docker-compose.yml up -d server
```

### 2. Caddyfile on the origin

Copy the Caddyfile block from [single-domain-routing.md §"What
devops needs to do"](./single-domain-routing.md) into your
devops repo's Caddy config (origin server). Replace any existing
`sentori.golia.jp { ... }` block. Then:

```bash
ssh lx64
sudo caddy reload --config /etc/caddy/Caddyfile
```

Verify:

```bash
curl -sI https://sentori.golia.jp/             # marketing
curl -sI https://sentori.golia.jp/docs/        # docs
curl -sI https://sentori.golia.jp/main         # SPA (returns SPA index.html)
curl -sI https://sentori.golia.jp/admin/api/me # backend (401 unauth, expected)
curl -sI https://ingest.sentori.golia.jp/v1/events # ingest (405 GET-not-allowed, expected)
```

### 3. OAuth provider configurations (Google + GitHub)

Update the registered redirect URIs in each provider's OAuth app
admin panel. Only the **host** changed in v2.4 (`app.sentori.*`
→ `sentori.*`); the **path stays `/auth/oauth/<p>/callback`**.
The `/auth/*` → `/api/auth/*` Caddy rewrite has been in place
since v1.0 and carries over unchanged to the consolidated host —
do NOT add `/api/` to the registered URI, providers do strict
match against what the server sends them.

#### Google Cloud Console

1. Go to <https://console.cloud.google.com/apis/credentials>
2. Open the Sentori OAuth 2.0 Client.
3. Authorized redirect URIs — add:

   ```
   https://sentori.golia.jp/auth/oauth/google/callback
   ```

4. (Optional) Remove the old `https://app.sentori.golia.jp/auth/oauth/google/callback`
   after verifying the new one works.

#### GitHub OAuth App

1. <https://github.com/settings/developers> → Sentori OAuth App
2. Authorization callback URL:

   ```
   https://sentori.golia.jp/auth/oauth/github/callback
   ```

3. Save.

### 4. Per-integration OAuth callbacks

These also live behind `SENTORI_BASE_URL` and trigger when an
**org operator** clicks "Connect Linear/GitHub/Jira" in
Integrations. Path:

```
${SENTORI_BASE_URL}/admin/api/integrations/<kind>/callback
```

Where `<kind>` is `linear`, `github`, `gitlab`, or `jira`. After
the env-var change above, the URL becomes:

```
https://sentori.golia.jp/admin/api/integrations/linear/callback
```

Update the registered redirect URIs in each:

- **Linear** — <https://linear.app/sentori-or-yours/settings/api> → OAuth Applications
- **GitHub** — second OAuth App for issue integration (separate
  from user-login one above)
- **GitLab** — application config
- **Jira / Atlassian** — Atlassian Developer Console

If these aren't currently used (Slack via webhook URL doesn't
need this) you can skip and configure on first activation.

### 5. (Optional) Add redirects from old subdomains

In the same Caddyfile, add:

```caddyfile
app.sentori.golia.jp {
    redir https://sentori.golia.jp/main{uri} permanent
}
docs.sentori.golia.jp {
    redir https://sentori.golia.jp/docs{uri} permanent
}
```

Existing bookmarks / saved URLs follow the 301 silently.

### 6. DNS

No changes needed. All four hosts (`sentori`, `app.sentori`,
`docs.sentori`, `ingest.sentori`) already point at the lx64
origin per `goliajp/devops/zones.yaml`. Caddy auto-issues TLS
certs via ACME on first request.

## Rollback

If anything goes wrong:

1. `git revert` the consolidation commits (last two on main).
2. Revert `SENTORI_BASE_URL` back to `https://app.sentori.golia.jp`.
3. Revert Caddyfile to the old `app.* / docs.* / sentori.*`
   three-block layout.
4. OAuth provider URLs can stay updated — adding a new redirect
   URI doesn't break the old one. So you can revert without
   needing to touch Google Cloud Console again.

No data migration in either direction — just routing config.
