# Sentori v2.29 — development plan

Status: **in progress 2026-06-10 — interactive actions wire passthrough.** Tenth version of Push Phase 2. Brings notification action buttons into the wire so any platform can render them. iOS-side category registration stays a host-app concern (Apple's NSUNotificationCategory must be registered at app launch — server can't dictate it); we surface the actions in a `sentori_actions` custom-data field for the host to react to.

Owner: claude + takagi
Date: 2026-06-10
References:

- iOS `UNNotificationCategory` / `UNNotificationAction` — registered in host AppDelegate
- Android `RemoteInput` / FCM `notification.click_action`

## NEVER rule

- **Zero wire break.** `actions` is an additive optional field.
- **No host-app forced wiring.** Customers who don't pass `actions` see exactly v2.28 behaviour.
- **No silent action invocation.** v2.29 surfaces actions through wire; the host invokes them via existing `onTap` / native callbacks. Server-side ack of action choice waits for v2.30+.

## L2 — Work units

| WU | Acceptance |
|---|---|
| **W1** Wire field | `NativeOptions` gains `actions: Vec<PushAction>` where `PushAction = { id: String, title: String, isTextInput?: bool, isDestructive?: bool }`. All but `id` + `title` optional. |
| **W2** Provider passthrough | APNs: keeps `aps.category` (legacy), emits `sentori_actions` as JSON array under custom data — host AppDelegate can read on tap. FCM: emits `sentori_actions` under `data` (FCM stringifies the JSON). WebPush: same. |
| **W3** Tests | APNs payload includes `sentori_actions` when actions present; omits when absent. FCM stringifies the array. |
| **W4** Preflight + close | `bun run preflight` green. ROADMAP v2.29 row. `git merge --no-ff feature/push-v2.29 → develop`. Push. |

## What v2.29 does NOT do

- **No native UNNotificationCategory generation.** Apple requires categories at app launch; we can't dictate from the server. Documented.
- **No Android channel registration.** v2.30 covers Channel API.
- **No action-response ack endpoint.** v2.30 / v2.31 will add `POST /v1/push/sends/:id/action` when needed.
