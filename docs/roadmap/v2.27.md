## NEVER rule

- **No events-table schema change.** The events table is partitioned and migrating an indexed column across many partitions is operationally risky. v2.27 reads existing `payload->'breadcrumbs'` JSONB. If query cost becomes a problem we can add a dedicated index in a later version.
- **Zero customer-side change.** No SDK update needed. Dashboard reads existing data.
- **Per-project isolation.** The query scopes to the calling project's events only.
- **Graceful degrade.** Pre-v2.25 sends carry no `_sentori.msgId` -> their downstream-impact card shows "n/a — pre-correlation push" instead of zeros.

## What v2.27 does NOT do

- **No materialized view.** Live-query `events_partitioned` directly. Materialized refresh adds operational complexity (interval, reindex) that we don't need until prod query load proves it.
- **No reverse-attribution endpoint.** "Which pushes reached the user behind issue X?" is one query away but doesn't fit on the existing Issue detail page without a UI redesign. Deferred to a later version focused on Issue-side surfacing.
- **No campaign / template / audience slice UI.** v2.25 wrote those columns but the cohort-comparison view requires its own dashboard module. Deferred to a "Push BI" module probably v2.30+.
- **No backfill of pre-v2.25 push_origin tagging.** Older breadcrumbs simply have no msgId; correlation degrades to "n/a" for those sends.

## Risk + mitigation

- **JSONB scan cost.** Partition pruning by `received_at` between `sent_at` and `sent_at + 24h` keeps the row count bounded to one (rare two) partitions. Then a sequential scan over `payload->'breadcrumbs'`. For a typical project pushing < 10k events/day, the scan is well under a second. If cost becomes visible we add a GIN index on `payload->'breadcrumbs'`.
- **24 h window.** Most push-driven user behaviour happens within hours. 24 h is a safe outer bound; if real prod data justifies different windows we expose `window=12h` / `48h` as a query param later.
- **Multi-tenant ambiguity.** Query joins on `project_id` first, so no cross-project leakage.
