## NEVER rule

- **Zero break of v2.7/v2.8 wire shape.** `/v1/push/*` routes + `device_tokens.native_token` shape (hex APNs token) stay byte-compatible. The server's `ApnsProvider` already handles real APNs deliveries â€” v2.9 just hands it real tokens from real devices.
- **Opt-in default.** Same as web: `sentori.push.register()` is host-triggered. Sentori SDK init does NOT request the OS permission. iOS users see the prompt only when the host calls register.
- **Performance iron rule applies.** APNs registration runs on a background dispatch queue. UNUserNotificationCenter delegate callbacks must not block main > 5 ms. Buffer + JS-side drain â€” never a synchronous bridge call from the delegate.
- **No host AppDelegate code required for the common path.** Method swizzle the standard `application:didRegisterForRemoteNotificationsWithDeviceToken:` and `application:didFailToRegisterForRemoteNotificationsWithError:` selectors so the SDK works in zero-config Expo + RN apps. Document the explicit-forward stub for hosts that have disabled swizzling.

## What v2.9 does NOT do

- **No Android.** v2.10 lands the FCM module. iOS today; Android one release later.
- **No Expo config plugin.** v2.11 adds the plugin that auto-injects `aps-environment` entitlement + `UIBackgroundModes`. Until then, recipe walks the manual setup.
- **No iOS rich notifications / Notification Service Extension.** Notification mutation / image attachment is a v2.12+ topic and lives outside the bridge.
- **No quiet hours / category management.** Host owns those; UNUserNotificationCenter exposes them directly.
- **No live activity / Dynamic Island.** Separate iOS API; deferred indefinitely.

## Host-side requirements

For the recipe (v2.9):

1. **Info.plist**
   ```xml
   <key>UIBackgroundModes</key>
   <array>
     <string>remote-notification</string>
   </array>
   ```
2. **Entitlements** (Xcode â†’ Target â†’ Signing & Capabilities â†’ Push Notifications)
   - Adds `aps-environment` entitlement automatically (`development` or `production` based on signing).
3. **AppDelegate** â€” no code required; SDK method-swizzles. To opt out, set Info.plist `Sentori.disableAppDelegateSwizzle = YES` and forward `application:didRegisterForRemoteNotificationsWithDeviceToken:` to `SentoriPushNotifications.shared.handleRegisteredToken(_)` manually.

## Verification recipe (manual smoke after deploy)

```bash
# 1. Upload APNs credentials to Sentori (one-time per project; the
#    project's APNs key id + team id + bundle id go in `config`; the
#    .p8 PEM is the encrypted secret).
curl -X PUT "https://sentori.golia.jp/admin/api/projects/<id>/push/credentials" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d @- <<EOF
{
  "provider": "apns",
  "config": { "key_id": "...", "team_id": "...", "bundle_id": "com.example.app", "env_default": "sandbox" },
  "secret": { "p8": "-----BEGIN PRIVATE KEY-----\n.../-----END PRIVATE KEY-----" }
}
EOF

# 2. In the React Native app: build + run on a real device (simulator
#    doesn't receive real APNs).
#    Host app calls:
#    await sentori.push.register({ onMessage: (m) => console.log(m) })
#    â†’ permission prompt â†’ ipt_* handle returned.

# 3. Send a push (server-side):
curl -X POST "https://ingest.sentori.golia.jp/v1/push/send" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d '{ "to": "ipt_<handle>", "title": "Hi", "body": "from sentori", "options": { "priority": "high" } }'

# 4. Within 30 s the device receives the push. Foreground â†’ the
#    onMessage handler fires; background â†’ standard iOS banner.
```
