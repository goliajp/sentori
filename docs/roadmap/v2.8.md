## NEVER rule

- **Zero break of v2.7 wire shape.** `/v1/push/*` routes + Sentori-native message shape + Expo-compat endpoints stay byte-compatible. v2.7 customers using raw REST keep working.
- **Opt-in default holds.** SDK config `init({ push: { enabled: false } })`. Host app must call `sentori.push.registerWeb({...})` explicitly to request the Notification permission.
- **No new system crypto dep.** Use the pure-Rust crates already in tree (`aes-gcm`, `hkdf`, `jsonwebtoken`, `rand`, `sha2`) plus one new pure-Rust addition: `p256` for the ECDH key-agreement. Stay off OpenSSL.
- **No web service worker auto-cache.** The SDK's Service Worker template only handles `push` and `notificationclick` events; it never touches the cache API or fetch interception. Anything beyond payload delivery is host-app territory.

## What v2.8 does NOT do

- **No RN iOS push.** v2.9 ships the iOS native module. v2.8 customers wanting RN push wait one phase or hit `/v1/push/tokens` from custom native code.
- **No RN Android push.** v2.10 lands the FCM Android native module.
- **No Expo plugin / dashboard module.** v2.11.
- **No HCM / MiPush.** v2.12.
- **No Service Worker for offline data sync / fetch interception.** The template only handles `push` + `notificationclick`. Hosts that want a full PWA Service Worker register their own and forward the push event to Sentori's listener (recipe documents the merge pattern).
- **No `push.registerWeb()` auto-call on `initSentori()`.** Permission prompts are host UX.

## Verification recipe (manual smoke after deploy)

```bash
# 1. Generate a VAPID key pair locally (one-time per project)
openssl ecparam -genkey -name prime256v1 -noout -out vapid_private.pem
openssl ec -in vapid_private.pem -pubout -out vapid_public.pem
PUB_B64=$(openssl ec -in vapid_public.pem -pubin -outform DER 2>/dev/null \
    | tail -c 65 | base64 | tr '+/' '-_' | tr -d '=')

# 2. Upload to Sentori
curl -X PUT "https://sentori.golia.jp/admin/api/projects/<id>/push/credentials" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d "{ \"provider\": \"webpush\",
          \"config\": { \"vapidPublic\": \"$PUB_B64\", \"contact\": \"mailto:dev@example.com\" },
          \"secret\": { \"vapidPrivate\": \"$(cat vapid_private.pem | base64 | tr -d '\n')\" } }"

# 3. In the browser, with `sentori.push.registerWeb({ vapidPublicKey: '<PUB_B64>' })` already invoked
#    the SDK POSTs to /v1/push/tokens and returns an ipt_*.

# 4. Send a push:
curl -X POST "https://ingest.sentori.golia.jp/v1/push/send" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d '{ "to": "ipt_<handle>", "title": "Hi", "body": "from sentori" }'
# → { "tickets": [{ "id": "send_...", "status": "queued" }] }

# 5. ≤30 s later: cron picks up, encrypts payload, POSTs to subscription endpoint,
#    browser SW receives the push, host app handler fires.

curl "https://ingest.sentori.golia.jp/v1/push/receipts/send_..." \
    -H "Authorization: Bearer st_admin_<token>"
# → { "ticket": { "status": "sent", "providerOutcome": "WP_201", ... } }
```
