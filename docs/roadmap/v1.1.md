## L1 — Roadmap statement

Sentori in v1.0 covers **debug** (errors, replay, traces). Analytics chunk A landed live-presence + Audience as proof. v1.1 expands sentori to three coordinated product legs while raising the engineering bar across the platform:

> **v1.1 turns sentori from "debug platform with two side experiments" into "debug + product analytics + security posture — built to one engineering standard."**

The standard half is non-negotiable: every chunk applies `architecture-standards.md` (transport tiers, cache hierarchy, structured errors, observability, Valkey discipline, code quality discipline). Without that, the three legs drift apart into separate-feeling product surfaces.

## L2 — Version boundary

Fixed at this line; no scope creep into v1.1. Anything below the cut is v1.2+.

### In v1.1

**Foundation chunks (apply to everything else)**:
- **F1** Server-side correlation id middleware + `X-Sentori-Correlation-Id` header on every response
- **F2** Structured error body across every existing endpoint (`{code, message, hint, docUrl, correlationId, layer}`)
- **F3** Dashboard react-query L2 persistence — MMKV/localStorage adapter, all queries opt-in via a single config
- **F4** Self-tracing — sentori's own dashboard + SDK report perf metrics back through `/v1/metrics:batch`
- **F5** Public error-code documentation generator — Rust enum + doc comments → markdown pages

**Analytics chunks (chunk A already shipped)**:
- **B** `sentori.track(name, props)` API + auto-pageview emission + server `events.kind='track'` partition
- **C** Hourly rollup table + scheduler + `Audience` dashboard tab (DAU/WAU/MAU + dimension splits + cohort retention)
- **D** `Behavior` dashboard tab (top routes, dwell, drop-off) + per-user timeline drill-down

**Security posture chunks**:
- **S1** `installId` SDK module (Keychain on iOS + Keystore on Android) + server-side IP→ASN lookup + signal collection
- **S2** `sentori.reportSecurity()` API + `events.kind='security'` partition + `Pin anomaly` dashboard tab with geo map
- **S3** Trust-scoring engine (weighted sum) + `sentori.queryTrustScore()` API + L1/L2 caching + SSE backplane + `Posture` dashboard tab
- **S4** Federation linkage table + cross-project trace view + signal explorer (operator weight tuning)

**Polish chunks** (run after the others, in parallel where possible):
- **P1** Valkey discipline sweep (rename non-compliant keys, verify TTLs, atomic Lua refactors where flame-graphed hot)
- **P2** Server code-quality sweep (unwrap removal, module size enforcement, `#[non_exhaustive]` annotations)
- **P3** Client typed-query-keys + atom state isolation sweep
- **P4** Property tests for the scoring function, dedup, rollup math
- **P5** Error-code doc pages on the marketing site (uses F5's generator)

### Explicitly NOT in v1.1 (deferred to v1.2+)

- WASM modules (replay reconstruction, ASN lookup, span-tree assembly). Land as opt-in optimisation per measurement.
- WS bidi transport (T3). Score-stream uses SSE in v1.1; WS only when a feature truly needs server→client *invalidation*.
- Funnels / cohort builders for Analytics. (Live + Audience + Behavior + User detail is enough surface area for v1.1.)
- Step-up challenge orchestration. v1.1 returns the score; host SDK picks the UX.
- Adversarial detection / bot clustering for Security.
- Web/JS SDK parity for analytics + security. RN-first per `project_direction`.
- Cohort retention deep dive UI (basic matrix only).

## L3b — Cold plan (per-chunk shape, no step-level)

Each entry: **what it does / depends on / verify gate**. No step list — that's L3a per chunk when it goes hot.

### Foundation phase (do first, blocks everything that wants the standards applied)

**F1 — Correlation id middleware**
- Server tower layer generates uuid-v7 per request, threads through `tracing::info_span`, sets `X-Sentori-Correlation-Id` response header.
- Dependencies: none.
- Verify: `curl -v` against any endpoint sees the header; logs grep-able by id.

**F2 — Error shape sweep**
- Every existing handler's error path returns the structured body. Server defines a per-module `Error` enum, `IntoResponse` converts uniformly.
- Dashboard `AdminApiError` type carries the new fields; UI banners render `hint` + copyable `correlationId`.
- SDK exception classes per feature, carrying `code` + `correlationId`.
- Dependencies: F1.
- Verify: every 4xx/5xx returns the spec shape; existing integration tests updated to assert.

**F3 — react-query L2 persistence**
- Single `persistQueryClient` adapter at dashboard root; MMKV-shaped storage for RN dashboard (later), localStorage for web today.
- Each query opts in via meta: `{ persist: true, persistKey: 'live:<proj>' }`.
- Eviction: LRU per scope, 5 MB cap, debounced 1 s writes.
- Dependencies: none (front-end only).
- Verify: cold reload of dashboard paints from cache before network; size cap holds across heavy use.

**F4 — Self-tracing**
- Server: every handler spans (already partly there); ensure `correlation_id`, `project_id`, `duration_ms` populated.
- Dashboard: react-query observer publishes `dashboard.query.duration_ms` per call to `/v1/metrics:batch`.
- SDK: `sdk.heartbeat.duration_ms`, `sdk.transport.batch_size`, `sdk.cache.hit_rate` reported through the existing metrics path.
- `/admin/api/self-test` returns ingest p50/p95 + valkey/db rt + last cert poll + last digest run.
- Dependencies: F1.
- Verify: self-test endpoint returns green within 200 ms cold; dashboard "platform health" strip on Overview renders amber if any tier degrades.

**F5 — Error-code doc generator**
- Build-time tool reads `Error` enums from server, emits one md file per code to `docs/errors/<code>.md`.
- Dependencies: F2.
- Verify: `bun docs:errors` produces N pages; CI gate fails if any `code` introduced without doc comment.

### Analytics phase

**B — `sentori.track()` API**
- SDK: `sentori.track(name, props)`; auto-pageview when navigation span fires.
- Server: `events_kind = 'track'`; partition Postgres table, same ingest path.
- Dashboard: surfaces in Audience under a future "Custom events" tab (lit by chunk C); for chunk B alone, lands a probe panel inside debug Events tab.
- Dependencies: F1, F2 (errors), F3 (cache for the probe panel).
- Verify: a `track` event from rn-example shows up on the dashboard within 5 s; payload preserves caller-provided props.

**C — Hourly rollup + Audience tab**
- Server: new `event_rollups_hourly` table, scheduler runs every 5 min (lag tolerance 10 min), reads since last watermark.
- Dashboard: `Audience` tab on the existing audience module entry — DAU/WAU/MAU sparkline, stacked-area by device/release/country, cohort retention matrix.
- Dependencies: B (provides track + pageview data), F3, F4.
- Verify: 1 week of synthetic traffic produces a believable DAU/WAU/MAU; cohort retention math sanity-checked against a hand-computed dataset.

**D — Behavior + User detail**
- Dashboard: `Behavior` tab — top routes / dwell / drop-off Sankey. `User detail` page — given user id, render session timeline (nav, custom events, errors, replay attachments).
- Server: query helpers aggregating navigation breadcrumbs + spans; cached 30 s.
- Dependencies: B, C.
- Verify: clicking a user in the Audience country breakdown opens that user's timeline with their last 24 h of events.

### Security phase

**S1 — installId + ASN**
- SDK: install-id module persisted to Keychain (iOS) / Keystore (Android). First-launch UUID, opaque to host.
- Server: extend geoip pipeline with IP→ASN lookup; new event field `asn`.
- Dependencies: F1, F2.
- Verify: rn-example fresh install reports installId; reinstall preserves it on iOS; ASN appears on incoming events.

**S2 — `reportSecurity()` + Pin map**
- SDK: `sentori.reportSecurity(kind, data)` ingest API; convenience helper `reportPinMismatch({ expected, observed, serverName })`.
- Server: `events_kind = 'security'` partition.
- Dashboard: `Pin anomaly` tab — geo map of mismatches in the last 24 h, click → list of affected users.
- Dependencies: S1, F2, F3.
- Verify: reportSecurity from rn-example puts a pin on the map; cluster of >3 mismatches in same ASN highlights as suspicious.

**S3 — Trust scoring + Posture**
- Server: scoring engine (initial 6-signal weighted sum), `trust_profiles` table, `trust:<...>` Valkey hot cache.
- SDK: `sentori.queryTrustScore()` with L1 + L2 (MMKV/Keychain) cache; SSE backplane `/v1/security/score:stream` for push updates.
- Dashboard: `Posture` tab — users sorted by lowest current score; click → security_events timeline for that user.
- Dependencies: S1, S2, F4.
- Verify: SDK call returns < 1 ms when cached; new pin-mismatch event drops the score within 1 s via SSE push; weight changes propagate to live score within 30 s.

**S4 — Federation + signal explorer**
- Server: `user_federation_links` table, query-time join for cross-project view.
- SDK: `sentori.linkFederatedIdentity({ provider, subject, userId })`.
- Dashboard: `Trust signal explorer` — for each weight, distribution + a slider that lets the operator simulate the score against historical traffic; cross-project view tied to federated id.
- Dependencies: S3, F5 (errors doc surface).
- Verify: same Google account linked across two projects shows merged timeline; weight slider produces a re-computed score within 200 ms via WASM (or fall back to server-side compute).

### Polish phase (parallel with feature phases where possible)

**P1 — Valkey discipline sweep**
- Audit every existing Valkey usage. Rename keys to spec, add missing TTLs, refactor any non-atomic multi-step write to Lua.
- Verify: linter check (a small ad-hoc script) confirms every `ZADD`/`SET`/`HSET` has a matching `EXPIRE` within the same async function or behind a Lua wrapper.

**P2 — Server code-quality sweep**
- Remove `unwrap()` from handler/hot paths.
- Add `#[non_exhaustive]` to public enums/structs that we expect to extend.
- Split modules > 800 lines.
- Verify: `cargo clippy --deny warnings` passes.

**P3 — Client typed-query-keys + state isolation**
- Convert all `useQuery(['some-string', id, ...])` to typed key helpers.
- Audit cross-module state imports; route through react-query or defined atoms.
- Verify: tsc passes with stricter eslint rule denying string-array query keys.

**P4 — Property tests**
- `proptest` over scoring weighted-sum invariants (monotonicity in trusted-signal increments, no NaN propagation).
- `proptest` over rollup math (sum-of-hourly == day-total).
- Property test over the v2 replay diff round-trip (already covered structurally, formalise).
- Verify: tests run < 30 s in CI; introduce a couple of intentional bugs and confirm they're caught.

**P5 — Error-code doc pages**
- Use F5 generator to scaffold; flesh out each page with 200-word explainer + remediation steps.
- Verify: every `code` in F2's taxonomy has a published page.

## L4 — Triggers

What advances v1.1 from one phase to the next:

**Trigger to start any feature chunk (B / C / D / S1 / S2 / S3 / S4)**: all foundation chunks F1/F2/F3/F4 are deployed and verified in prod. F5 can lag — error doc gen is a "should-have" for v1.1 ship, not a blocker for individual chunks.

**Trigger to start S3**: S1 + S2 + at least 24 hours of real signal flowing from rn-example (sanity that the data shape we'll score against actually looks like we think). Without that, scoring engine ships untestable.

**Trigger to start D**: C deployed and 1 week of hourly rollups built up (so the Behavior page has data to render). Without lag, D ships against empty Audience and the demo is misleading.

**Trigger to start S4**: S3's scoring engine accepts at least one real `reportSecurity` from prod traffic. Federation is the cross-project view of scored entities; not useful if scoring isn't producing anything.

**Trigger to start P1-P4 (polish)**: at least 50 % of v1.1 feature chunks shipped. Polish before substance is premature optimisation; polish after substance lets us see what actually grew non-compliant.

**Trigger to bump from v1.1 to v1.2**: every L2 chunk above either shipped, explicitly cut (with a doc PR moving it to v1.2 list), or is in flight with a clear ETA < 1 week. Plus 1 week soak in prod with no major regression on any v1.1-introduced surface.

## Suggested cadence

If we ship one chunk every 2-3 days (single IC at sentori's existing pace), v1.1 fully lands in **8–10 weeks**. Faster if foundation + parallel chunks compress to a week each.

Sequence proposal (numbered = ship order; bracketed pairs = parallel):

```
1. F1 (correlation id)              ~2 days
2. F2 (error shape sweep)           ~5 days
3. [F3, F4]                         ~3 days each
4. F5 (err-doc generator)           ~3 days
5. [B, S1]                          ~4 days each
6. [C, S2]                          ~6 days each
7. [D, S3]                          ~6 days each
8. S4 + [P1..P5 sweep]              ~5 days
   ───────────────────
   v1.1 ship                        ~8 weeks @ steady cadence
```

## Decisions needed to start (L4 → L3a)

When you OK the v1.1 boundary, pick **which chunk goes hot first**. Recommended starting points (small + foundation-ish):

- **F1** if you want immediate "professional polish" everywhere (correlation id visible in every error from day 1).
- **F3** if you want the dashboard's "first paint < 20 ms" demo right away — visible UX win, no server change.
- **B + S1** parallel if you want to widen the data-shape coverage before doing dashboard work.

I lean F1 first (every other chunk benefits from it) then F3 (UX win, blocks nothing) then split into [B, S1] parallel. But the boundary is yours to pick.
