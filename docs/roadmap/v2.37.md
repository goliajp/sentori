# Sentori v2.37 — development plan

Status: **in progress 2026-06-10 — adaptive throttle (#1) + global per-provider circuit-breaker (#3).** Closes two of the three holes identified in the Insight team's review of Sentori's push under APNs flood conditions. The third hole (429 rate as a separate axis from invalid_rate) lands trivially as a by-product.

## Background

The Insight team flagged that Sentori's queue could pile up and APNs would reject. v2.20–v2.35 handled most failure modes (token cache, persistent HTTP/2, retry jitter, per-provider rate cap, quarantine, SKIP LOCKED), but three gaps remained:

1. **No 429-driven L1 dial-back.** Smart retry respects `Retry-After` per message, but the outgoing rate to the provider stays at L1 cap. APNs telling us "slow down" was respected message-by-message, not system-wide.
2. **No global per-provider circuit-breaker.** Per-(project, provider) quarantine handles project-localised failure, but if APNs is having a bad time globally, we keep hammering at L1 rate for every healthy-looking project.
3. **`should_auto_throttle` only `tracing::warn!`.** v2.23 noted the threshold but never wired the actual throttle action.

## NEVER rule

- Zero wire / SDK / dashboard change.
- Throttle decays automatically — no "stuck in throttled state" footgun.
- Local healthy projects on a globally-unhealthy provider still see throttled L1 — that's the point. The protection is for APNs's downstream relationship, not for per-project SLA.
- Single-instance behaviour preserved; per-process state, no shared store.

## L2 — Work units

| WU | Acceptance |
|---|---|
| **W1** `HealthState::should_global_throttle(kind)` | Aggregates `Counts` across ALL projects for `kind`. Returns `true` when `(rate_limited + other_transient + timeout) / total >= 0.20` AND `total >= 50`. The 50-sample minimum stops a few unlucky retries from tripping a global. |
| **W2** `TokenBucket` accepts a throttle factor | `set_throttle(factor: f64, until: Instant)` — multiplies effective `refill_per_sec` by `factor` until `until`. After expiry, factor resets to 1.0. Stored as `Mutex<Option<(f64, Instant)>>`. Tests with `MockClock`. |
| **W3** `RateLimiter::throttle_provider` + `throttle_project` | Public methods that look up the bucket and call `set_throttle`. Default factor `0.5`, default window `5 min`. |
| **W4** dispatch_cron wires the trigger | After `apply_outcome`'s `health.record`: if `note_transient_failure` returned `true` (quarantine just tripped for project), call `rate_limiter.throttle_project(...)`. Independently, if `should_global_throttle(kind)` returns true, call `rate_limiter.throttle_provider(kind, 0.5, 5min)` and emit a `tracing::warn!`. |
| **W5** `should_auto_throttle` wires throttle_project too | v2.23's per-(project, kind) auto-throttle decision was a no-op. Now it dials L2 down to 50% for 5 min. |
| **W6** Tests | TokenBucket set_throttle decay; RateLimiter provider throttle dials L1 refill; global throttle triggers per HealthState aggregation; both expire after window. |
| **W7** Preflight + close | green; merge develop. |

## What v2.37 does NOT do

- **No new endpoint to query throttle state.** Dashboard surface for v2.37.x.
- **No admin override** ("force release throttle"). v2.37.x.
- **No 429-specific finer-grained adaptation** (currently aggregated with other transients). If prod data shows 429-only treatment is needed, separate it later.
- **No persistence across restarts.** Throttle is in-memory, server restart resets — matches QuarantineState semantics.
