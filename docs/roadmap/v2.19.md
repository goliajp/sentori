## Why

The Push series (v2.7 → v2.12) shipped a production-grade send pipeline: APNs / FCM v1 / Web Push (RFC 8291 self-implemented) / HCM / MiPush, dispatcher cron mirroring `webhook_dispatch`, idempotency, retry schedule, bad-token streak auto-revoke. Operationally we shipped a single dashboard surface for it: credential upsert/delete.

That leaves every other "is my push working?" question without a UI answer:

- Which projects in my org have credentials configured?
- Are my credentials still valid, or did the FCM service-account key get rotated?
- Did my last send go through?
- How many tokens are dead?
- What's in my queue right now?
- A specific send failed — what was the vendor response on each retry?
- Can I re-fire that one?

v2.19 answers all of them. Verbatim from the user's framing: *"完整地管理和监控 push 服务，哪个项目注册了这个服务，权限配置是否绿色可用，看到推送成功与否，重发状态等等"*.

## Decisions frozen

1. **No quota-burning verify for MiPush** — the `/v3/message/regid` POST is the only auth surface and it consumes a quota slot per call. Sending throwaway pushes on every dashboard refresh is unacceptable. We surface "unverified" with a tooltip explaining why; operators know the cred is right when real sends succeed.
2. **Two `Providers` registries (cron + handler) instead of one shared** — passing `Arc<Providers>` through `ServerConfig → router::build → AppState → main` was the alternative; chosen for clarity over a small extra cache (FCM tokens last 3600s, so worst case one extra OAuth mint per restart). Revisit if oauth-mint cost shows up in profiling.
3. **Retry button enabled only when device still present + not revoked** — re-queueing a send to a revoked device would just immediately fail; we hard-block in the handler too (the UI disable is a hint, not the gate).
4. **Page size 50 default, 200 max** for devices + sends list. Operators triaging an incident sometimes scroll fast; 200 keeps that responsive without OOM-ing the table component.
5. **Stats refetch interval 30s** for overview / sends; **10s** for the per-send detail view (an in-flight retry should land visibly within one tick).
6. **Per-row verify is on-demand**, not auto-refresh — clicking the Verify button runs the network call. Auto-pinging on render would amplify outbound traffic by N projects × M providers every time the user lands on the tab. Cached for the session via `qc.setQueryData` so tab-hops don't re-fire.

## What didn't ship in v2.19

- **Send-from-dashboard button** — composing a notification payload in the UI (title / body / target token / data) is a separate UX problem. Operators use the backend integrator path for now (or `curl` against `/v1/push/send`). Lands as v2.20+ if demand surfaces.
- **Bulk re-queue** — "retry all failed sends in the last hour" is operationally tempting but easy to misuse against revoked-token storms. v2.19 ships single-send retry only; bulk would need a confirmation gate + dry-run.
- **Per-token revoke from UI** — read-only on the Devices tab. SDKs revoke via the `DELETE /v1/push/tokens/:handle` public endpoint; admin revoke is a separate authorization conversation.
- **Time-series chart** — KPI cards show current values, not trend. Reuses the existing chart story when one lands; v2.19 keeps the module text-and-table.
