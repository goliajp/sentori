## NEVER rule

- **Zero wire-shape break.** Same routes, same DB schema. Health state is in-memory only.
- **Zero customer-side change required.** Update server image, inherit.
- **Per-(project, provider) isolation absolute.** One project's high invalid_rate cannot affect another project's auto-throttle decision.
- **Auto-throttle MUST be reversible.** When the invalid_rate drops back under the threshold, the throttle releases automatically. No "stuck in throttled state" footgun.

## L2 — Work units

| WU | Acceptance |
|---|---|
| **W1** `HealthState` module | `server/src/push/health.rs` — per-(`project_id`, `ProviderKind`) rolling counters. Five outcomes tracked: `Sent` / `InvalidToken` / `RateLimited` (429) / `Timeout` / `OtherTransient`. Window: 5 min wall-clock, bucketed at 60 s. `record(project, kind, kind_of_outcome)` per-call O(1). Query: `invalid_rate(project, kind)` returns the fraction of sends in the window that were `InvalidToken`. `Clock` seam reused from `token_cache`. |
| **W2** Auto-throttle hook | `HealthState::should_auto_throttle(project, kind) -> bool` true when `invalid_rate >= 0.10` (10 %) AND total sends in window `>= 20` (avoid throttling on small-N noise). When true, `RateLimiter::throttle(project, kind, duration)` halves the L1 + L2 refill rate for that target for `THROTTLE_DURATION` (default 5 min). Throttle auto-lifts when window passes. Re-entrancy: if invalid_rate still high, throttle re-fires. |
| **W3** dispatch_cron feed | After `apply_outcome`'s SendOutcome classification, also call `health.record(project, kind, outcome)`. On `InvalidToken` outcome, additionally check `should_auto_throttle`; if true, log `tracing::warn!` with the rate + win-size + emit one-shot throttle effect (no log spam — throttle is idempotent). |
| **W4** Stale-token soft eviction | `dispatch_cron::sweep_once` SQL `WHERE` clause adds `AND d.last_seen_at > now() - interval '90 days'`. A token whose last `/v1/push/tokens` register/refresh is > 90 d gets silently skipped — almost certainly OS-revoked, sending wastes a slot and a delivery_log row. Operators can still see the rows via the dashboard; we just won't dispatch. |
| **W5** Tests | Unit tests for HealthState: record + invalid_rate calculation; rolling window drops old entries; per-(project, kind) isolation; auto-throttle threshold + minimum-N; throttle decision toggles when rate drops back. |
| **W6** Preflight + close | `bun run preflight` green. ROADMAP v2.23 row added. `git merge --no-ff feature/push-v2.23 → develop`; push. |

## What v2.23 does NOT do

- **No dashboard UI surface.** The "safety margin to blacklist" gauge in the dashboard comes in v2.24 (Send inspector + Receipt API), which is the natural batch for new admin endpoints + UI cards. v2.23 surfaces via `tracing::warn!` only.
- **No DB-backed counters.** v2.23 health state is process-local; multi-instance share is v2.38.
- **No admin endpoint to manually override the throttle.** Future v2.24+ alongside the dashboard.
- **No 429-rate auto-throttle.** Only `InvalidToken` triggers; 429 is provider-side rate hint and is already respected by smart retry (v2.20). Adding a 429-driven L1 dial-back is straightforward but adds API-surface complexity; defer until we have prod data showing it's needed.

## Risk + mitigation

- **Threshold tuning.** 10 % invalid + N >= 20 is conservative — most healthy apps run well under 1 % invalid on a steady-state basis (clients re-register on rotation, OS-revoked tokens get cleaned out via bad_streak in v2.7). Prod logs will tell us if 10 % is too tight or too loose; tune in v2.24+.
- **Throttle hides real problems.** A project consistently at 10 % invalid stays auto-throttled and shows in logs but doesn't surface in a UI yet. Mitigated by the dashboard gauge in v2.24.
- **Stale-token eviction sliding edge.** A token at exactly 90 d 1 ms gets dropped; on next register from the device it comes back. Acceptable.
