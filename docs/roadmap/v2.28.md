# Sentori v2.28 — development plan

Status: **in progress 2026-06-10 — rich media (iOS NSE-ready + Android BigPicture).** Ninth version of Push Phase 2. Closes the RN-richer-rendering side of the push roadmap: payloads can carry an image URL; FCM Android auto-renders BigPicture (zero device-side work); iOS requires a Notification Service Extension which the Expo plugin scaffolds, with a one-time Xcode wiring step documented in the recipe.

Owner: claude + takagi
Date: 2026-06-10
References:

- `docs/design/push-architecture.md` — RN-richer rendering
- APNs: [`mutable-content`](https://developer.apple.com/documentation/usernotifications/modifying-content-in-newly-delivered-notifications) is how a Notification Service Extension takes over a push between delivery and display
- FCM: `message.notification.image` is the standard field, auto-renders BigPicture on Android

## NEVER rule

- **Zero wire-shape break.** New optional `richMedia` field is additive; legacy payloads omit it and behave exactly as v2.27.
- **Zero host-app code change to receive (FCM/Android).** FCM auto-renders the image when `notification.image` is set.
- **iOS NSE is fully optional.** A host that never adds the NSE target still gets text-only delivery — the `mutable-content` flag is harmless; the OS just doesn't have a service extension to invoke.
- **Provider-friendly preserved.** Adding `mutable-content` doesn't trip APNs's abuse heuristics; FCM's `notification.image` is a documented field.

## What v2.28 does NOT do

- **No auto Xcode target injection.** That is v2.28.1's job — needs `withXcodeProject` pbxproj fiddling, separate audit. Customers manually add the NSE target the first time; subsequent prebuilds keep the source in sync.
- **No iOS RN ad-hoc rendering.** Once the NSE is in place + image attached, iOS draws the rich notification with no JS-side work. We don't intercept on the JS side.
- **No video / audio attachments.** v2.28 ships image only. Other attachment types deferred.
- **No custom NSE override.** Customers using their own NSE override our `sentori_attachment_url` — that's the union behavior; no conflict.

## Risk + mitigation

- **`mutable-content: 1` without NSE installed.** Harmless — the OS just delivers the notification without invoking an extension. No degradation vs the v2.27 path.
- **CDN failure for the image URL.** NSE downloads with a 5 s timeout; on failure, the OS delivers the text-only notification. Customer-facing: a missed image, not a missed notification.
- **Expo prebuild overwrites NSE source.** `withDangerousMod` is idempotent — it overwrites `ios/SentoriNSE/SentoriNotificationServiceExtension.swift` on every prebuild. Customers who modify their copy will lose changes on prebuild. Documented in the recipe.
