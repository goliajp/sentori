## Why

`@expo/config-plugins` was a standalone npm package on the old `9.x` /
`10.x` line through Expo SDK 53. With Expo SDK 54 (May 2025) and
onward, Expo moved every first-party `@expo/*` package + `expo-*`
package into version-with-SDK mode вЂ” config-plugins now ships as
`~55.0.x` (Expo 55, Aug 2025) and `~56.0.x` (Expo 56, Nov 2025).

Our `sdk/expo/package.json` was still pinning
`"@expo/config-plugins": "^9 || ^10"` вЂ” effectively `@goliapkg/sentori-expo`
was uninstallable on any current Expo SDK. Same shape on
`sdk/react-native` for `expo-modules-core` (was `">=2.0"`, now ships
as 55.x / 56.x).

The runtime code is fine вЂ” every config-plugin API we use has been
stable since `@expo/config-plugins@4`. This release is a metadata
fix.

## Support window after v2.17

- **Expo 55** (Aug 2025, React Native 0.81)
- **Expo 56** (Nov 2025, React Native 0.82) вЂ” current

Older Expo apps stay on `@goliapkg/sentori-expo@5.x` /
`@goliapkg/sentori-react-native@2.x` вЂ” those tags keep the v2.7вЂ“v2.12
Push setup the same way the published 5.x line did.

Next Expo SDK (57+) will need its own bump when it ships.

## What v2.17 does NOT do

- **No runtime code change.** `app.plugin.js`, every native module
  (Swift / Kotlin), every JS surface stays byte-for-byte the same.
- **No SDK API change.** `sentori.push.*` / `sentori.captureException`
  / everything else customer-facing is identical.
- **No publish step.** v2.17 lands the changesets + dep bumps; the
  `bunx changeset version` + `bunx changeset publish` round runs in a
  separate step (operator-triggered per the existing release runbook).
