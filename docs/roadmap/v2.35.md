# Sentori v2.35 — development plan

Status: **in progress 2026-06-10 — queue model upgrade: SKIP LOCKED for safe multi-worker.** Sixteenth and final version of Push Phase 2. Replaces the implicit "single-worker assumption" sweep query with a Postgres 9.5+ `FOR UPDATE SKIP LOCKED` claim semantics, so multiple dispatch_cron instances (or future Valkey-backed workers) can sweep concurrently without double-dispatching.

## NEVER rule

- Zero wire / SDK / dashboard change.
- Single-instance behaviour identical to v2.34 — `SKIP LOCKED` on a single transaction is a no-op for the only worker.
- No new dependency (we keep dispatching from Postgres; Valkey hot queue stays a follow-up).
- Single-statement transaction. The claim + advance happens in one round-trip; no app-level locks.

## L2 — Work units

| WU | Acceptance |
|---|---|
| **W1** Claim semantics | `sweep_once` SELECT becomes a UPDATE-SELECT-WITH-RETURNING that:<br>1) `WITH claimed AS (SELECT id FROM push_sends WHERE status = 'queued' AND next_attempt_at <= now() ORDER BY next_attempt_at LIMIT $1 FOR UPDATE SKIP LOCKED)`<br>2) returns the same rows the existing code expected, but every row is now "claimed" by this worker — a concurrent sweep skips them. Equivalent to `webhook_dispatch`'s post-Phase-29 claim path. |
| **W2** Behavioural test | (no DB test fixture — Postgres-only feature; covered by existing webhook_dispatch claim tests if present; otherwise prod observation is the verification.) |
| **W3** Preflight + close | green; merge develop. |

## What v2.35 does NOT do

- Hot queue on Valkey (deferred; SKIP LOCKED on Postgres is already enough for ≥10× current throughput).
- Multi-instance orchestration (DispatchHandle spawn count, leader election) — single-instance lx64 is still the deployment.
- Self-adaptive tick interval — fixed 30 s tick stays.
