## NEVER rule

- **Zero break of v2.7-v2.10 wire shape.** Same `/v1/push/*` routes, same `device_tokens` table. Dashboard module + Expo plugin are pure additions.
- **Plugin is opt-in.** Hosts that don't add `'@goliapkg/sentori-expo'` to their `app.json` plugins array see no changes. Hosts that DO add it get the full iOS + Android push setup with one entry: `"plugins": ["@goliapkg/sentori-expo"]`.
- **Plugin is idempotent.** Re-running `expo prebuild` should produce the same native files. No accidental duplicate `<service>` entries / duplicate `<key>UIBackgroundModes</key>` entries.
- **No secrets in the plugin.** Provider credentials (APNs p8, FCM service account) stay on the server in encrypted `push_credentials`. The plugin only injects host-side build config (entitlements, manifest entries, gradle deps).
- **Dashboard Push module follows v3 GDS conventions.** PageHeader / Card / DataTable / EmptyState / Alert. Lives under `manage` group with chord `g n`. Default visible (no `hidden: true`).

## What v2.11 does NOT do

- **No device list / send history endpoints.** The dashboard Push module ships credential CRUD only in v2.11. Device list + send history would require new server endpoints + indexes; they're a follow-up in v2.12 or later.
- **No real-time delivery feed.** No SSE / live tail of push_sends. Deferred indefinitely; operators see the receipt via `GET /v1/push/receipts/{id}` on demand.
- **No expo-only push helpers.** The plugin sets up the native config; runtime API stays
  `sentori.push.register(...)` from `@goliapkg/sentori-react-native`.
- **No HCM / MiPush.** v2.12 wraps the Chinese providers.

## Verification recipe (manual smoke after deploy)

```bash
# 1. In an Expo host app:
#    Add to app.json: "plugins": ["@goliapkg/sentori-expo"]
#    (or with options: ["@goliapkg/sentori-expo", { "ios": true, "android": true }])

expo prebuild --clean

# 2. Expected file changes:
#    ios/MyApp/MyApp.entitlements    — aps-environment added
#    ios/MyApp/Info.plist            — UIBackgroundModes: [remote-notification]
#    android/build.gradle            — google-services classpath
#    android/app/build.gradle        — apply google-services, firebase-bom, firebase-messaging
#    android/app/google-services.json — copied from host root if present

# 3. Run the app on a real device; sentori.push.register({...}) flows
#    end-to-end without any host-side config code beyond what the
#    plugin handled.

# 4. In Sentori dashboard → /main/<org>/<proj>/push:
#    Add an APNs credential (provider=apns + p8 PEM + key/team/bundle IDs).
#    Add an FCM credential (provider=fcm + service-account JSON).
#    Verify the rows appear in the DataTable.
```
