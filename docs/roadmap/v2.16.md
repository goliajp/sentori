## NEVER rule

- **Zero SDK change.** Alert rules 是 server-side trigger evaluator + webhook dispatcher,SDK 不参与。
- **Zero server change.** `orgsApi.{list,create,patch,delete}AlertRule` 全自 v1.x alive。`/admin/api/orgs/:slug/alerts/rules`(注:org-scoped not project-scoped)路由不动。
- **Wire shape 不变.** `AlertRule` / `AlertRuleInput` / `AlertChannel` / `AlertTriggerKind` types 不动。

## Lens verdict: not a lens — manage group utility

Per audit doc:alerts 不是 lens — 它是 cross-cutting 的 trigger / notification 配置面。每个 lens(find-bug / find-slow / find-user / hygiene)的某个 measure 都可以是 trigger source(new_issue / regression / event_count / crash_free_drop / 未来新增 trigger),所以 alerts 不该归任一 lens。

放 `manage` group + adminOnly + chord `g k`(thinK alerts,也避免跟其它 chord 冲突)。同 Health / Webhooks / Integrations / Teams / Settings 这种 "管理类" surface 同 group。

## What v2.16 does NOT do

- **No new trigger kinds.** 4 个现有(new_issue / regression / event_count / crash_free_drop)够覆盖 v1.x backlog。新 trigger 单独 v2.17+。
- **No multi-channel.** v2.x 一直 single webhook channel per rule;email / slack-direct / linear-via-integration 通过整合层(integrations / webhooks)走,不在 alert rule 里复制配置。
- **No firings history view.** rule 显示 `lastFiredAt` 但完整 firings timeline 是 v2.17+(需 server endpoint)。
- **No audience flip.** v2.4 verdict 仍 stands — Users overview 已包含。
