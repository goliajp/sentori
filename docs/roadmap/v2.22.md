## NEVER rule

- **Zero wire-shape break.** Same routes, same DB schema. Rate state is in-memory only.
- **Zero customer-side change required.** Defaults sized so legitimate v1.x customers won't hit any layer in normal operation. Update server image, inherit.
- **Per-project isolation absolute.** Project A's rate-limited sends do not slow project B's sends. Tests enforce.
- **Rate-limit deferral MUST NOT burn retry budget.** Same rule as quarantine in v2.21. A send pushed back because a layer was saturated comes back at `retry_count` unchanged.
- **No fairness inversion at L3 saturation.** A higher-traffic project that fills L3 doesn't starve a low-traffic project indefinitely — L1/L2 are checked first, so a noisy project can be capped before it hits L3.

## L2 — Work units

| WU | Acceptance |
|---|---|
| **W1** TokenBucket + RateLimiter | `server/src/push/rate_limit.rs` — `TokenBucket<C: Clock>` with `try_acquire(n) -> bool`, classical refill math. `RateLimiter` composes three layers: L1 per-`ProviderKind` (5 buckets), L2 per-`project_id` (lazy `HashMap`), L3 global `AtomicU32` inflight counter. Defaults (capacity / refill_per_sec): APNs 400/200, FCM 400/200, HCM 200/100, WebPush 200/100, MiPush 100/50; per-project 100/50; global inflight 200. All numbers conservative — well below provider hard limits, sized for typical mid-tier self-host. |
| **W2** `RatePermit` + acquire | `acquire(project, kind) -> Result<RatePermit, RateError>`. RatePermit holds an `Arc<AtomicU32>` for L3 — Drop releases the inflight slot. RateError carries the saturated layer (`ProviderRateLimited(kind)` / `ProjectRateLimited(project)` / `GlobalInflight`) for differentiated deferral semantics. |
| **W3** Providers + dispatch_cron wire | `Providers` gets `pub rate_limiter: Arc<RateLimiter>`, constructed in `Providers::new()` alongside `quarantine`. `dispatch_cron::process_one`, after the quarantine check, calls `rate_limiter.acquire(project_id, kind)`. ## What v2.22 does NOT do

- **No 429 / Retry-After adaptive throttle.** Provider 429 responses already feed smart retry's Retry-After path (v2.20); using a provider's 429 to dial back L1 refill is v2.23 (health gauge phase) — needs the rolling-window infrastructure to be useful.
- **No dashboard exposure.** Rate-limit state is in-memory + visible in tracing logs. UI lives in v2.23.
- **No DB-backed rate state.** v2.22 is process-local; horizontal share is v2.38 (queue upgrade).
- **No admin endpoint to override defaults per project.** Admin surface in v2.23.
- **No multi-instance fairness.** A second sentori-server process gets its own counters. v2.38 fixes via shared Valkey.

## Risk + mitigation

- **Default L3=200 inflight.** Conservative — if a self-host customer needs more, it surfaces as `GlobalInflight` defers, which is recoverable. We'll learn the real ceiling from prod logs and tune.
- **Per-project bucket map growth.** A project deletion leaves an in-memory bucket. Same trade-off as quarantine: acceptable for single-instance + restart cleanup.
- **`try_acquire` precision.** Floating-point math drifts under sustained heavy load; the bucket may over-grant by 1 token at edges. Acceptable — we are guarding against orders-of-magnitude misuse, not single-token precision.
- **L3 fetch_add overflow.** `AtomicU32` overflow is impossible given L3 cap < 2^31.
