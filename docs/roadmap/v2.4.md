## NEVER rule

- **No SDK change.** v2.3 already shipped the `linkBy` identity
  hash path; v2.4 only adds operator-side surfaces over the
  existing fingerprint data. Hosts on v2.3.0 see the new
  endpoints / dashboard pieces without bumping a dependency.
- **`/users/lookup` follow-through is one-hop only.** The
  `identity_merges` schema enforces "one primary per alias" via
  the `(scope_id, alias_fp)` PK; chains of merges are not
  followed. If the operator merges A→B then later wants A→C,
  the second call updates the existing row (`ON CONFLICT DO
  UPDATE`) rather than chaining; this keeps lookup p99 flat at
  one extra index hit.
- **No new event-table writes.** Every v2.4 surface is read-only
  over events / identity_fingerprints / the new
  identity_merges table. Ingest path is untouched.

## Re-deferred to v2.5+

- **Project-level identity-scope carve.** Same reason as v2.3:
  the 0065 migration chose "org-default scope only"; cross-cutting
  carve column lands when a concrete use case surfaces.
- **Salt rotation.** No forcing function.
- **Region / data-residency scopes.** Multi-region routing is a
  deploy-infra project.
