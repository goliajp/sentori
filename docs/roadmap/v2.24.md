Status: **in progress 2026-06-10 — provider health surface + dashboard gauge.** Fifth version of Push Phase 2. Surfaces the v2.23 invalid-rate / total / auto-throttle state in an admin endpoint + dashboard card, so operators can see what the server already knows.

Note: the original sketched v2.24 ("Send inspector + Receipt API + replay + downstream impact") is split. Downstream impact requires the rule-4 wire primitives (`_sentori.msgId` payload, SDK auto-breadcrumb, ack-with-session) that aren't shipped yet (v2.25–v2.27). Receipt API is already in place since v2.7 (`GET /v1/push/receipts/:id`). Replay-last-N-days is a single-action button against existing data (v2.26+). v2.24 ships the smallest coherent slice: make v2.23's gauge visible.

## NEVER rule

- **Zero wire-shape break.** Only new admin endpoint + new dashboard card. Public ingest API untouched.
- **Zero customer-side change required.** SDK + cred + config unchanged.
- **Default-on safe.** The health card shows graceful empty-state when there are no in-window samples (most projects most of the time).
- **No new data sources.** v2.24 reads the v2.23 in-memory HealthState directly. No DB queries beyond what already exists.

## What v2.24 does NOT do

- **No downstream impact.** Requires the v2.25 `_sentori.msgId` wire primitive + v2.26 SDK ack-with-session to do correctly. Hacky "join on user_fingerprint where exists" approximations are misleading; we'd rather show nothing than wrong correlation.
- **No replay-last-N-days.** Bulk replay needs safeguards (idempotency, blast-radius cap, dry-run preview). Singleton retry is already in `admin_retry_push_send` (v2.19); bulk goes with v2.26+.
- **No additional Receipt API fields.** Existing `/v1/push/receipts/:id` is sufficient; if customer ask reveals a missing field we add it then.
- **No alert wiring.** Auto-throttle currently surfaces via `tracing::warn!` only. Adding an Alerts rule binding lives with the v2.27 push-correlation BI work where the rule expression language gets the necessary terms.

## Risk + mitigation

- **In-memory state restart resets.** Health state is per-process and cleared on server restart. After a restart, the gauge will show "no recent sends" until ~20 sends come through. Acceptable; matches the model where the server cold-starts and there's no history to recover.
- **Multi-instance read consistency.** Same as v2.22/v2.23: per-process. v2.38 horizontal queue will introduce shared health state if needed.

## Closeout

- 1 roadmap doc + 1 new admin endpoint handler in `api/push.rs` + 1 route in `router.rs` + 1 API client + 1 React Query key + ~80 lines of UI Card markup in `view.tsx` + ROADMAP row. No SDK / DB / wire touch.
