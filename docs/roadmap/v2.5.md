## NEVER rule

- **Zero SDK change.** Cold-start + TTID + TTFD + slow/frozen
  frame counters all already populate `spans` with `vital.kind=*`
  tags as of v1.x. v2.5 only reshapes the dashboard around the
  data that already lives in `spans`.
- **Zero server change.** `api/vitals.rs` (per-release per-route
  p50/p95 aggregation) is the anchor; v2.5 ships only the web
  redesign.
- **Lens stop-ship rule.** Vitals flips `hidden: false` because
  Phase 1 audit named it the find-slow anchor; no other module
  flips alongside. Audience / cert-monitor / posture / privacy
  stay hidden until their own lenses open.

## What v2.5 does NOT do

- **No `/explore` consumer rewrite.** Phase 8 plan suggested
  reshaping Vitals as a `dim=route + p95_duration` consumer of
  the v2.3 `/explore` grammar. The existing `api/vitals.rs`
  query already does the same shape with vitals-specific
  numbers (p50/p95 TTID + TTFD + slow/frozen) that `/explore`
  doesn't surface today (`p50_duration` / `p95_duration` are
  reserved measures but return null on most dims pending the
  spans-join wiring). Sticking with `vitals.rs` is the simpler
  path; the unified-`/explore` future can swap the backend
  without touching the view layout.
- **No new spans-list drill view.** Phase 8 plan mentioned "if
  traces stays hidden, ship a thin spans-list endpoint for the
  drill." v2.5 routes the drill through Issues instead — the
  operator gets the *errors* on a slow route, which is the
  most-common follow-up anyway. A pure spans-list drill lands
  with the find-slow lens's second iteration (when Traces
  opens under its own lens).
- **No `dim=device_os` lens.** v2.3 added `device_os` to
  `/explore`'s dim list; the Vitals view doesn't pivot on it
  yet. Slice-by-OS comparison is doable today via the
  comparison strip if you label two manually-tagged routes
  `ios:checkout` vs `android:checkout`, but native pivot
  support is a v2.5.x patch.
