## NEVER rule

The v2.3 SDK redesign is **additive at the wire**. Every existing
v2.2 event the SDK shipped still validates and routes correctly
through the v2.3 server. Hosts that don't touch `linkBy` /
`beforeSend` / `logLevel` see zero behaviour change at the
transport level beyond console silence.

The two surface changes that *could* surprise a host on upgrade:

1. **Default `logLevel: 'warn'`** drops ~20 `[sentori]` console
   lines that v2.2 emitted under healthy operation. Hosts that
   *grepped* their metro for `[sentori]` to confirm the SDK was
   alive must now grep for `onReady` callback wiring or set
   `logLevel: 'debug'`.
2. **`withSpan` becomes overloaded.** The pre-v2.3 export name
   `withSpan` (low-level `withSpan(span, fn)`) is still callable
   because the new overload dispatcher accepts that shape
   unchanged. The new shape is `withSpan(name, fn)` — high-level
   wrap helper. `withScopedSpan` and `withActiveSpan` both stay
   exported as explicit names.

Performance contract per CLAUDE.md "几乎不能造成 host app 的性能抖动":

- `init.logLevel: 'warn'` (the default): gated-out `logger.debug`
  calls are < 1 µs/op (mechanical bench). Net SDK contribution
  from the logger module is essentially free.
- `init.beforeSend`: synchronous, called once per event before
  enqueue. Throw / non-event-return falls back to the unmodified
  event with one-shot warn (NEVER rule).
- `linkBy` identity hash: SubtleCrypto SHA-256 of normalised
  value; ~5 ms per call on a mid-range device for 1–3 keys
  (design §6 budget).

## Re-deferred from design §8

- **Project-level identity-scope carve.** Migration 0065 explicitly
  chose "org-default scope only"; carve-out needs a separate
  `projects.identity_scope_id` column + backfill story. Tracked
  as v2.4+ work.
- **Region scope (data residency).** Multi-region routing is a
  deploy-infrastructure project. No customer pull yet.
- **Salt rotation.** Needs a historical fingerprint rebuild
  migration. No forcing function yet.

All other design §8 deferrals were absorbed into Phase 0–4 work
per the master plan's recovered-defers table.

## Sequencing

Phases 0 → 1 → 2 → 3 → 4 → 5 → 6 ran in dependency order across
a single autorun session. The 48-hour dogfood gates the master
plan proposed between rc.1 / rc.2 / rc.3 were collapsed because:

- W6.0 → W6.1 changes are additive, mechanical bench gate is
  green, no host-facing behaviour change.
- W6.2 server schema (0065–0067) already shipped pre-plan;
  Phase 4 work is the DSR endpoint + UI, which Insight can
  exercise post-release without prior dogfood.
- W6.3 compat is pure translation; the upstream Sentori-native
  paths it delegates to were already battle-tested in pre-plan
  shipment.

Insight dogfoods v2.3.0 on prod from this commit; W6.4 release
notes flag the two surface changes hosts might notice (logger
silence + withSpan overload).
