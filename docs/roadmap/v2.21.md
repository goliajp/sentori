## NEVER rule

- **Zero wire-shape break.** Same routes, same DB schema. Quarantine is in-memory only — no new tables, no new fields.
- **Zero customer-side change required.** Update server image, inherit per-provider clients + quarantine. No SDK / cred / config touch.
- **Per-project isolation absolute.** Project A's bad APNs creds must not push project B's APNs sends into quarantine. Tests enforce.
- **Quarantine MUST NOT burn retry budget.** A send deferred because its (project, provider) is quarantined comes back at retry_count unchanged. Quarantine is a "wait", not an "attempt".

## What v2.21 does NOT do

- **No three-layer rate limit.** The L1/L2/L3 token-bucket rate limiters per ironclad rule #2 ("rate non-limiting") are v2.22.
- **No invalid-rate health gauge.** That's v2.23.
- **No DB-backed quarantine.** v2.21 quarantine is process-local; horizontal-scale share is v2.38 (queue upgrade) territory.
- **No dashboard surface.** Quarantine state is visible in server logs (tracing `info!` on trip/release). UI exposure waits for v2.23 health-gauge module which covers all the "is this provider okay?" surfaces at once.
- **No admin endpoint to manually quarantine/release.** Out of scope for v2.21; can be added in v2.23 alongside the dashboard.

## Risk + mitigation

- **Per-provider client memory.** 5 clients × N connections each. reqwest defaults are bounded (`pool_max_idle_per_host` unset = unlimited but driven by activity). Tune `pool_max_idle_per_host = 4` for non-APNs and `8` for APNs as conservative defaults.
- **Quarantine accumulates dead keys.** A `(project, kind)` whose project is deleted leaves an in-memory entry. Acceptable for v2.21 (single instance, restart wipes); v2.38 horizontal queue will need explicit eviction on project delete.
- **Streak counter across restarts.** Quarantine state is in-memory; server restart resets all streaks. Acceptable — the moment the server restarts is also a natural "back to normal" signal; if the provider is still broken, the streak rebuilds within 5 sends.
