## NEVER rule — load-bearing (above every L2 work unit)

> **The SDK must never harm the host app.** v2.1 inherits the
> v2.0 contract verbatim: every public `sentori.metric.*` API and
> every auto-instrument hook is wrapped via `safeFn` / `safeAsync`
> (see `sdk/core/src/safe.ts`). Internal errors silently fail,
> optionally self-report via the existing circuit-breaker; the
> host app never sees a thrown error, a rejected promise, a frame
> drop, a network failure that we caused, or anything else
> attributable to Sentori.
>
> Per `.claude/CLAUDE.md` performance budgets:
> - main-thread cost: **< 1 %** sustained, **< 5 ms** per tick.
> - 60s capture window total network: **< 500 KB**.
> - 60s capture window per-attachment: **< 200 KB**.
>
> **v2.1-specific addition:** auto-instrument is the highest-risk
> surface in this version because it samples continuously in the
> host app's main loop (FPS via rAF, heap snapshot per N seconds,
> route-nav timing on every push). Every auto-instrument pass has
> a **per-tick budget test** as a stop-ship gate in W2. Failing
> this test on the slowest device class in the matrix (currently
> Pixel 5 / iPhone SE 2nd gen as floor) is **stop-ship**. This
> rule beats LLM friendliness, API elegance, and feature coverage.

## Compatibility model

- **Server:** forever backward-compatible with v1 + v2.0 SDK wire
  format. v2.1 adds:
  - a new ingest route `POST /v1/events/<p>/metrics` (orthogonal
    to existing `/v1/events*` — no overlap)
  - new tables `runtime_metrics*`, `endpoint_check`,
    `endpoint_probe` (pure additions, no existing schema migration)
  - v1 / v2.0 SDK requests parse cleanly unchanged. Regression
    suite `server/tests/v1_compat.rs` (from v2.0 W1) is reused
    verbatim — v2.1 adds `server/tests/v20_compat.rs` covering
    v2.0 SDK against v2.1 server.
- **SDK:** v2.1 SDK installs alongside v2.0 SDK without API
  conflicts. `emitMetric` is a new namespace; importers that don't
  reach into it pay zero bundle delta. Auto-instrument is on by
  default but each domain (FPS / heap / cold-start / route / net)
  is independently toggleable via `Sentori.init({ metrics: { fps:
  false } })`.
- **Dashboard:** new top-level routes `/main/metrics` and
  `/main/health` — additive. Empty-state shows "no metrics
  reported yet — upgrade SDK to 2.1+" with copy-paste install
  hint. No existing routes / queries touched.

## L1 — Roadmap statement

> Sentori's v2.0 was about giving developers a way to *say* what
> happened — `captureMessage`, manual spans, the failure-safe
> SDK surface that doesn't lie to the host app.
>
> v2.1 is about giving the SDK a way to *measure* what's happening
> on its own, and giving the dashboard a way to ask "is my service
> alive?" without ever touching the SDK.
>
> Two faces of the same shift: **from passive error reporter to
> active observability platform.**
>
> Concretely:
>
> 1. **Runtime metrics** — RN-first auto-instrument of FPS, JS
>    heap, cold-start, route navigation timing, network bytes.
>    Web SDK auto-instruments web vitals (LCP/CLS/INP) and
>    `performance.memory`. Zero host-app integration cost — same
>    one-line `Sentori.init` already in v1+v2.0 picks up metrics.
>    A BI-style dashboard view answers "did p95 FPS regress after
>    release X" / "which device class drops frames on route /
>    onboarding" without forcing the user to learn a query DSL.
>
> 2. **Endpoint health probing (synthetic / outside-in)** —
>    sentori-server cron-pings user-configured URLs, asserts
>    status / body / latency, opens an issue on consecutive fail
>    and auto-resolves on consecutive pass. Decoupled from the
>    SDK — a host app that hasn't even installed Sentori can
>    still have its endpoints watched. Notification routing
>    reuses the v1.3 channel infra (Linear / Slack / email).
>
> **Why these two together:** metrics writes the timeseries
> infrastructure (partition + rollup + BI query layer) that the
> health module reuses verbatim for its probe history. Splitting
> them across two phases would duplicate that work; bundling
> them in v2.1 is a 30 % infra-cost saving versus v2.1 metrics
> only + v2.2 health.

### Explicitly NOT in v2.1

- **Custom user metric API** (`sentori.metric.counter("checkout.click")` /
  `gauge` / `histogram`). Discussed; the user-facing API surface
  is large (naming convention, tag cardinality rules, type system
  guarantees), and host apps already get 80 % of business-metric
  needs from `Sentori.track(...)` (the v0.7 analytics line). Defer
  to v2.2 once auto-instrument dogfoods through Insight.
- **Multi-region synthetic probe** (probe from 3+ geographic
  origins, majority-vote on outage). v2.1 probes only from lx64.
  Docs ship with a clear "outage detection ≠ outage immunity"
  caveat. Multi-region needs Cloudflare Worker dispatch or extra
  origin VMs — deferred to v2.2 + paid-tier gating.
- **OTLP export.** Already pre-flagged in v2.0 NOT-list; still out.
  Sentori speaks its own protocol per
  [`project_direction`](../../.claude/memory/project_direction.md).
- **CPU per-process / native-thread instrumentation.** RN gives
  us JS-thread CPU cheaply via main-loop sampling; native-thread
  CPU requires platform-specific bridge work (iOS `task_info`,
  Android `proc/self/stat`) and is unjustified in v2.1 — the
  performance signal we care about is JS-thread blocking and
  frame-drop, which FPS already covers.
- **Battery on iOS** is gated by `UIDevice.isBatteryMonitoringEnabled`
  which apps don't usually flip on. v2.1 RN ships an init flag
  `metrics.battery: true` that flips it; default `false` so we
  don't burn host-app battery state. Android: free metric, default
  on.
- **PromQL / Graphite / SQL escape hatch for power users.** v2.1
  BI panel is the only query surface. A power-user escape hatch
  is its own L2 (v2.3+).
- **Plan-gating on probe frequency / metric retention.** v2.1
  ships defaults (probe 60s min interval, metrics 90d raw / 365d
  rolled). Commercial gating is its own commercial-tier L2; v2.1
  exposes the hooks (`org.plan_id` already exists) but doesn't
  enforce.
- **Alert rules for metrics** (e.g. "page me when FPS p95 < 30
  for 5 min"). v2.1 only auto-alerts on endpoint health (where
  the alert semantics are unambiguous — probe failed = page).
  Metric-driven alerting needs a threshold-rule UI + flapping
  protection; defer to v2.2 (reuses v1.2 `rule_eval::spawn_cron`
  infrastructure).

### v0.1 — W1 Metrics ingest + storage

**Doc PR (must land first):** `docs/design/v2-metrics.md` — schema
DDL, partition / rollup / retention strategy, cron interval math
(why 60s rollup not 10s), BI query grammar, capacity envelope
(50 projects × 100 devices × 30s flush → ~170 QPS write, raw 90d
retention sized to one Postgres node).

**Code PRs:**

1. Migration `0068_runtime_metrics.sql`:
   - `runtime_metrics_raw` partitioned by `ts` daily, primary key
     `(project_id, ts, name, tags_hash)`
   - `runtime_metrics_1m` / `_1h` / `_1d` flat tables
     materialized via cron refresh
   - `runtime_metrics_dropped` for accounting (rate-limit overflow,
     malformed values, unknown tag keys)
2. Migration `0069_runtime_metrics_indexes.sql`:
   - `(project_id, name, ts DESC)` covering index on raw partition
   - same on each rollup tier
3. `crates/sentori-server`:
   - `api::metrics::handle` axum handler (token-gated, same as
     `/v1/events`)
   - body limit: `RequestBodyLimitLayer::new(MAX_METRICS_BATCH_BYTES)`
     where `MAX_METRICS_BATCH_BYTES = 1 MB` (one batch = ~10k
     metrics worst case at 100B/metric)
   - schema validation: name regex `^[a-z][a-z0-9_]*\.[a-z0-9_.]+$`,
     value f64 finite, tag count ≤ 16, tag key ≤ 40 chars
   - per-project rate limit: 1000 metrics / sec / project (returns
     `429` with `Retry-After`)
4. `crates/sentori-server/src/metrics_rollup.rs`:
   - `spawn_cron(pool, interval)` matching the
     `rule_eval::spawn_cron` shape
   - 60s tick: aggregate `_raw` from `(now - 70s, now - 10s)` into
     `_1m` (10s safety margin for late-arriving batches)
   - 1h tick: roll `_1m` → `_1h`
   - 24h tick: roll `_1h` → `_1d`
5. `crates/sentori-server/src/main.rs`:
   - Add `metrics_rollup::spawn_cron(p.clone(), 60)` + corresponding
     `tracing::info!("metrics rollup cron spawned (60s tick)")`
6. `server/tests/v20_compat.rs`:
   - Replay v2.0 SDK request fixtures against v2.1 server, assert
     unchanged response shape

**Acceptance:**
- `POST /v1/events/<p>/metrics` accepts a batch, returns 202
- `runtime_metrics_raw` row count == batch size
- 70s later, `runtime_metrics_1m` has the corresponding aggregated
  rows
- v20 compat suite green
- `tracing::info!("metrics rollup cron spawned")` shows in
  `docker logs sentori-server` after deploy

### v0.2 — W2 Metrics SDK + auto-instrument

**SDK PRs (per package):**

1. `sdk/core/src/metric.ts`:
   - `emitMetric(name: string, value: number, tags?: Record<string,
     string>)` — wraps via `safeFn`; pushes to in-memory ring;
     drops with `reportInternal({ kind: "metrics.ring_overflow" })`
     when ring > 10k
   - 30s flush coalesced with existing event flush — single
     network round-trip; metric batch sent as `attachment` of the
     flush envelope (no extra HTTP request)
2. `sdk/core/src/auto-instrument/fps.ts`:
   - rAF loop measures inter-frame Δt; computes rolling 5s p50
     + p95; emits `runtime.fps.p50` + `runtime.fps.p95` every 5s
   - **Per-tick budget test**: assert rAF callback < 0.5 ms on a
     Pixel-5-equivalent CI bench
3. `sdk/core/src/auto-instrument/heap.ts`:
   - 30s sample of `performance.memory?.usedJSHeapSize`; emits
     `runtime.heap.used_bytes`. Web-only; RN ships
     `react-native-mem-usage` polyfill but capability-flag if not
     installed
4. `sdk/core/src/auto-instrument/cold-start.ts`:
   - `Sentori.init()` records `t0`; on first
     `requestAnimationFrame` (RN: `InteractionManager.runAfter
     Interactions`) emit `runtime.cold_start_ms` once per session
5. `sdk/core/src/auto-instrument/route-nav.ts`:
   - For RN: subscribe to react-navigation `state` events, emit
     `runtime.route_nav_ms` with tag `to=<route>` per transition
   - For web (next/remix/vite-router): emit on
     `routeChangeStart` / `routeChangeComplete`
6. `sdk/core/src/auto-instrument/network.ts`:
   - Tap existing fetch + XHR interceptors (already in v1 for
     trace), accumulate bytes-sent + bytes-received per 30s
     window, emit `runtime.network.bytes_sent` +
     `runtime.network.bytes_received`
7. Per-platform package wires (`sdk/react-native`,
   `sdk/javascript`, `sdk/react`, etc.) export an `enableAutoMetrics`
   hook that `Sentori.init` calls based on the `metrics: {...}`
   option
8. **CI perf bench:** new job in `.github/workflows/sdk-perf.yml`
   — pinned device emulator runs FPS instrument for 60s, asserts
   main-thread cost < 1 % sustained

**Acceptance:**
- npm publish: `@goliapkg/sentori-core@2.1.0` + RN + web matrix
