## NEVER rule

- **Zero break of v2.7/v2.8/v2.9 wire shape.** `/v1/push/*` routes + `device_tokens.native_token` shape (FCM registration token) stay byte-compatible. The server's `FcmProvider` already handles real FCM deliveries — v2.10 just hands it real tokens from real devices.
- **`sentori.push.register()` resolves on Android too.** Same JS API, same return type. Host app code authored for v2.9 keeps working on Android with one delta: the `env` field on `device_tokens` is null (FCM has no sandbox/production split).
- **Non-push hosts pay nothing.** Firebase Messaging is a `compileOnly` dependency — class-path available for the SDK's compile + manifest merge, but NOT pulled into the final APK of a host that doesn't itself declare `firebase-messaging`. Runtime FCM availability is verified via `Class.forName` before any Firebase call.
- **Opt-in default holds.** Same as v2.9 iOS: `sentori.push.register()` is host-triggered. Android 13+ adds a runtime `POST_NOTIFICATIONS` permission which the SDK requests on register, not on init.
- **Performance iron rule applies.** FCM callbacks route to a worker thread; the buffer is JS-drained at 1 Hz with AppState-aware pause/resume — same pattern as v2.9.

## What v2.10 does NOT do

- **No Expo config plugin.** v2.11 ships the plugin that auto-injects `google-services.json` setup + the Gradle plugin invocation + the Firebase deps. Until then, recipe walks the manual setup.
- **No Huawei / Xiaomi.** v2.12. HCM has its own SDK (`com.huawei.hms:push`); MiPush uses a different transport. Both lit later.
- **No FCM topic subscriptions.** Out of scope per the design doc. Topic broadcasts deferred to v2.13+.
- **No background data sync via WorkManager.** Host owns that.

## Host-side requirements

For the recipe (v2.10):

1. **Add Firebase to the host app**
   - Download `google-services.json` from Firebase console → place in `android/app/`.
   - Apply Gradle plugin in `android/build.gradle` (project) and `android/app/build.gradle` (app):
     ```gradle
     // android/build.gradle
     buildscript { dependencies { classpath 'com.google.gms:google-services:4.4.x' } }
     // android/app/build.gradle (bottom of file)
     apply plugin: 'com.google.gms.google-services'
     ```
   - Add Firebase deps:
     ```gradle
     dependencies {
       implementation platform('com.google.firebase:firebase-bom:33.x.x')
       implementation 'com.google.firebase:firebase-messaging'
     }
     ```

2. **Notification channel** (Android 8+)
   - Either create a default channel in Application.onCreate, or let the SDK use the default channel ID `"sentori"` (created lazily on first push received).

3. **AppDelegate equivalent — none.** No host-side code required. The SDK declares the `FirebaseMessagingService` in its manifest; manifest merger picks it up.

## Verification recipe (manual smoke after deploy)

```bash
# 1. Upload FCM credentials to Sentori (one-time per project; the
#    server-side JSON service account key is the encrypted secret).
curl -X PUT "https://sentori.golia.jp/admin/api/projects/<id>/push/credentials" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d @- <<EOF
{
  "provider": "fcm",
  "config": { "project_id": "my-fcm-project" },
  "secret": $(cat firebase-admin-sdk.json)
}
EOF

# 2. Build + run the React Native Android app. Host calls
#    await sentori.push.register({ onMessage: (m) => console.log(m) })
#    → Android 13+: permission prompt → FCM token returned → server stores
#    the device → `ipt_<handle>`.

# 3. Send a push:
curl -X POST "https://ingest.sentori.golia.jp/v1/push/send" \
    -H "Authorization: Bearer st_admin_<token>" \
    -H "Content-Type: application/json" \
    -d '{ "to": "ipt_<handle>", "title": "Hi", "body": "from sentori",
          "options": { "priority": "high", "channelId": "sentori" } }'

# 4. Within 30 s the Android device shows the notification. Foreground
#    → onMessage handler fires; background → standard system tray.
```
