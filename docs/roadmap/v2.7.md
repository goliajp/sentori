## NEVER rule

- **Zero SDK change in v2.7.** Every client SDK package (`sentori-core`, `sentori-react-native`, `sentori-javascript`, `sentori-expo`, `sentori-next`, framework wrappers) is unchanged. v2.7 is server-only.
- **Opt-in default.** Even though SDKs don't ship, the eventual SDK config will be `init({ push: { enabled: false } })`. The server already enforces this stance by requiring an explicit per-project credential record before any provider call succeeds.
- **Reuse `ingest.sentori.golia.jp`.** No new subdomain. Push tokens register and sends fan-out through the same ingest host that already carries customer error events.
- **No filesystem credentials.** Provider credentials live in `push_credentials` encrypted with `secrets::seal()`. There is no `/etc/sentori/projects/*` fallback path; the SaaS deployment shape is the only supported shape.
- **Per Sentori 性能铁律.** Server-side dispatch cron uses one `tokio::task` (mirrors `webhook_dispatch::spawn_cron`), no thread-per-send.

## What v2.7 does NOT do

- **No SDK matrix changes.** Customers integrating today still don't have a `sentori.push.register()` call. They can hit `/v1/push/tokens` over raw REST if they're motivated; recipes for the supported pattern arrive in v2.8 (Web Push) and v2.9 / v2.10 (RN iOS / Android).
- **No Web Push / HCM / MiPush provider impl.** The trait + module slots are in place but `send()` returns `NotImplemented`. v2.8 lights Web Push; v2.12 lights HCM + MiPush.
- **No dashboard UI.** The `push` module on the sidebar (chord `g n`, group `manage`) ships with v2.11 — its prerequisites are Expo plugin + cred-CRUD UX. Dashboard's `hidden: true` Push module already exists in spirit because there is no module — added in v2.11.
- **No topic / segment broadcasts.** Per design doc, deferred to v2.13+.
- **No HTTP webhook backfill onto shared client.** Webhooks and integrations keep building per-call `reqwest::Client` for now; backfill is mechanical follow-up.
- **No customer migration playbook.** Sentori currently has no push capability, so there's nothing to migrate from. The Expo-compat endpoint exists to ease customers who today integrate against Expo's push servers.

## Verification recipe (manual smoke after deploy)

```bash
# 1. Set APNs credentials for a project (assumes you have project_id + p8 file)
sentori-cli push creds set-apns -p <project_id> --key-id ABC --team-id DEF --bundle com.example.app --p8 @key.p8

# 2. Register a device (simulated — normally an iOS SDK does this)
TOKEN=ipt_$(uuidgen | tr -d -)
curl https://ingest.sentori.golia.jp/v1/push/tokens \
  -H "Authorization: Bearer st_pk_<project_public_token>" \
  -d '{"provider":"apns","env":"sandbox","nativeToken":"abc...64hex...","metadata":{"bundleId":"com.example.app","osVersion":"17.4"}}'
# → { "id": "ipt_…", "createdAt": "..." }

# 3. Send a notification
curl https://ingest.sentori.golia.jp/v1/push/send \
  -H "Authorization: Bearer st_admin_<project_admin_token>" \
  -d '{"to":"ipt_...","title":"hi","body":"from sentori"}'
# → { "ticket": { "id": "send_...", "status": "queued" } }

# 4. Read receipt
curl https://ingest.sentori.golia.jp/v1/push/receipts/send_... \
  -H "Authorization: Bearer st_admin_<project_admin_token>"
# → { "ticket": { "status": "sent" | "failed", "providerOutcome": "APNS_200" | ..., ... } }
```

(sentori-cli helper for push CRUD ships in v2.12 — until then, set credentials via admin REST.)
