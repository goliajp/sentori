# Sentori v1.0 — Roadmap

Living plan document for v1.0. Layered per the 4-layer method:
L1 (one-line direction) / L2 (version boundary + work units) /
L4 (cold→hot triggers). Hot plans (L3a) live in the working
conversation, not here, so this doc stays stable across iterations.

## L1 — Direction

Sentori v1.0 takes RN error reporting from "you saw the stack" to
"you saw the 30 frames of prop changes before the crash, and you
saw the five other issues sitting on the same user-intent path."

The moat is not redoing stack + screenshot. It is turning the React
fiber tree and the breadcrumb sequence — two RN-only signals — into
visualisable assets the rest of the market cannot reproduce.

## L2 — Workstreams + work units

Two workstreams. A ships first; C unlocks once A is in users' hands.

### Workstream A — Replay scrubber + fiber tree diff

| ID | Unit | Required output |
|---|---|---|
| A1 | SDK wireframe path repair | iOS captureWireframe returns non-null on a known repro app inside this monorepo, without external testers |
| A2 | Ring buffer frame encoding | Frame-to-frame delta compression so 30 frames of fiber tree fit in the existing attachment-size budget |
| A3 | Server-side wireframe storage + query | Attachment store accepts wireframe blobs; new endpoint returns ±N s of frames keyed by event id |
| A4 | Dashboard scrubber UI | Time-axis slider + thumbnail rail + main canvas renders fiber tree at the selected frame |
| A5 | Tree diff renderer | Frame N vs frame N-1: visual highlight on props / state that changed |
| A6 | Issue detail integration | New "Replay" tab on the issue detail surface; nothing else needs to know about wireframes |

### Workstream C — Intent cluster

| ID | Unit | Required output |
|---|---|---|
| C1 | Breadcrumb schema normalisation | Stable action schema + stable hash key per action |
| C2 | Embedding pipeline | Decision on where embeddings run (server / local / hash-prefix hybrid) wired end-to-end |
| C3 | Cluster algorithm choice | HDBSCAN vs agglomerative vs hash-prefix — selected and benchmarked on real breadcrumb data |
| C4 | issue.metadata.cluster_id + reverse index | Server-side schema migration and population pipeline |
| C5 | Issues list cluster view | New tab on Issues; issues fold by cluster_id with path preview |
| C6 | Cluster detail page | Five-issue cluster renders as a path visualisation (sankey or sequence) |

## L4 — Triggers

A trigger is a state predicate, not a date. Cold work promotes to
hot when its trigger fires.

| From | To | Trigger |
|---|---|---|
| (start) | A1 hot | Always — A1 is the gating unit for the whole roadmap |
| A1 hot | A2 hot | Internal RN fixture in this monorepo produces a non-null wireframe payload on every captureException call across three runs |
| A2 hot | A3 hot | 30-frame payload sits inside the existing attachment-size budget after delta encoding |
| A3 hot | A4 hot | Curl against the new endpoint returns frame stream for a known event id |
| A4 hot | A5 hot | Scrubber UI renders a static frame from a real ingested wireframe attachment |
| A5 hot | A6 hot | Diff renderer correctly highlights a known prop change between two adjacent frames |
| A6 hot | C1 hot | "Replay" tab live on app.sentori.golia.jp ≥ 1 week + ≥ 100 ingested events carry full breadcrumbs |
| C1 hot | C2 hot | Action schema applied to ≥ 100 ingested events, stable hash collisions < 1% |
| C2 hot | C3 hot | Embedding pipeline returns vectors for ≥ 50 sample breadcrumb sequences |
| C3 hot | C4 hot | Chosen algorithm produces ≥ 3 clusters of ≥ 2 issues each on the test set |
| C4 hot | C5 hot | cluster_id present on ≥ 80% of issues in the dataset |
| C5 hot | C6 hot | Cluster view tab renders folded issue list |

## Decision points (not L1, but must answer before each L2 unit enters hot)

- **A2** — Frame delta encoding. Use immer patch / json-patch / custom. Affects SDK bundle size and CPU during captureException.
- **C2** — Embedding host. OpenAI is costly and forces network egress; on-device sentence-transformer is memory-heavy on RN; hash-prefix is cheap but loses semantic similarity. Likely answer: hash-prefix v1, embedding v2 if signal is weak.
- **C3** — Surface model. Do users see clusters by URL/path prefix or by semantic similarity? Two different products; pick before C5 designs land.

## What this doc is, what it is not

- It IS the place to find the v1.0 direction, the work units, and
  the trigger predicates that govern checkpoint promotion.
- It is NOT a step list. Hot plans (step-by-step with detection
  commands) live in the active conversation / task list, not here.
- It is NOT a schedule. Trigger predicates replace dates; the only
  question that matters is "is the predicate satisfied yet."

When the boundary above changes — work units added / dropped, a
trigger condition reformulated — update this file in the same PR
as the scope change. Do not let the plan and the code drift.
