## L1 — Roadmap statement

v1.2 admitted W7/W8/W9/native-source into the version but left dashboard UX behind: operators configure integrations via curl, native source bundles via CLI only, notifications without preferences. v1.3 closes that gap.

> **v1.3 turns v1.2's backend-complete features into webapp-complete features.**

The L1 commitment: every backlog item from v1.2 closeout gets full UX. "Backend works + you can curl to it" is no longer an acceptable ship state. Operators must be able to discover, configure, edit, undo, and inspect each feature from the dashboard without referring back to docs.

## L2 — Version boundary

Fixed at this line; no scope creep into v1.3.

### In v1.3

**Settings + integration management (W7 follow-up with proper UX)**:
- **W11** Settings UI shell + Linear adapter card. Establishes the per-org settings page pattern (`/org/<slug>/settings/integrations`). Linear is OAuth so the simplest first card.
- **W11.b** GitHub + GitLab + Jira adapter cards. Apply the W11 pattern; GitHub's card supports both PAT and App auth (lands W13 here), GitLab supports gitlab.com + self-hosted base URL, Jira supports Cloud + Server / DC choice (lands W12 here).

**Per-user preferences (W8 follow-up)**:
- **W14** `/account/notifications` page. Operator sets per-kind mute (status / assignee / comment / merged / labels / priority / regressed), digest cadence (immediate vs hourly vs daily), and channel hint (in-app only vs in-app + email).

**Source bundle decompression streaming (W3 perf follow-up)**:
- **W15** Replace the in-memory `Bundle::from_tar_gz` with a streaming + on-demand-extract design. Backend-only; no new UX surface. Adds an "Uploaded source bundles" admin panel under W11's release-detail page so operators can see what's stored without poking the API.

### Explicitly NOT in v1.3 (deferred to v1.4+)

- Webhook delivery retry UI (current retry is automatic + invisible).
- Cross-org integration sharing / templating.
- A real multi-tenant Jira Cloud OAuth 3LO flow (the v1.3 Jira card stays API-token based; OAuth-3LO is its own sub-project).
- SDK-side native source bundle helper (`@goliapkg/sentori-cli` already does it; SDK auto-bundle is out of scope).

## L3b — Cold plan (per work-unit shape)

Each work unit's L3a hot plan **must include** an explicit UX design step BEFORE backend / frontend impl. Per the webapp-UX-is-delivery memory.

### v0.1 — W11 settings shell + Linear card

**UX design (sketch in commit body)**:
- Entry point: new "Settings" item in the org sidebar → "Integrations" tab. Each adapter is a card with:
  - Header: icon + name + connection status badge ("connected" / "disconnected" / "needs reconnect").
  - Body: connection metadata (workspace name, default repo, last sync at) when connected; CTA + helper text when disconnected.
  - Footer: "Disconnect" or "Reconnect" buttons.
- Empty state: list all adapters with disconnected status; each card's body explains what this integration does + a "Connect" CTA.
- Loading state: skeleton card rows.
- Error state: red banner with `errorBody.hint` from the structured error envelope.
- Confirmation: "Disconnect" prompts a confirm dialog ("This will stop new-issue sync. Existing linked issues stay linked. Proceed?").
- Success feedback: toast after each action (connect/disconnect/reconnect).

**Backend**:
- Extend `GET /admin/api/integrations` to include per-adapter connection status + denormalized metadata (workspace name etc.).
- No new write endpoints (reuse existing `connect` / `callback` / `configure` / `revoke`).

**Tests**:
- Snapshot test on each card state (connected/disconnected/error).
- Server integration test: assert `GET /integrations` returns the new fields.

Verify: load the settings page in dev with no integrations → see four disconnected cards. Connect Linear → card flips to connected + shows workspace name.

### v0.2 — W11.b GitHub + GitLab + Jira cards (with W13 + W12 baked in)

**UX design**:
- GitHub card: choose between "Personal Access Token" and "GitHub App" via a small toggle. PAT shows the existing form (token + default repo); App shows an "Install GitHub App" button that launches the install flow (manifest-based App creation lands in this checkpoint as W13).
- GitLab card: PAT form + a "Self-hosted" expandable section for the base URL (default: gitlab.com).
- Jira card: choose between "Jira Cloud" (current) and "Jira Server / DC" (new in W12). Cloud uses email + API token + site; Server uses PAT + base URL.

**Backend (W12 + W13)**:
- Migration 0055: extend `integrations.kind` if needed (`jira_server` as a separate row? or keep `jira` with config.deployment ∈ {'cloud','server'}? — pick the latter to avoid kind explosion; bake into the existing JiraConfig).
- Extend `JiraAdapter::accept_manual_config` to accept `deployment: 'cloud' | 'server'` + `baseUrl` (for server) instead of `site`.
- Extend `JiraAdapter::create_issue` + `update_status` to route to the right base URL.
- New `GithubAppAdapter` (or refactor `GithubAdapter` to support both modes): JWT-signed app auth, install-token cache.
- `accept_manual_config` validates the App-id + private-key path.

**Tests**:
- New `GithubAppAdapter` unit tests covering JWT sign + install-token exchange (mocked).
- `JiraAdapter` integration test: server-deployment variant routes to a custom URL.

### v0.3 — W14 per-user notification preferences

**UX design**:
- New `/account/notifications` page (account section in user menu).
- Sections:
  - "Channels": toggle "in-app only" vs "in-app + email" (email integration deferred to v1.4 but the toggle persists the choice).
  - "Mute kinds": checkboxes for the 7 activity verbs.
  - "Digest cadence": radio for `immediate | hourly | daily`. Immediate = every event. Hourly/daily = batch into a single digest notification.
- Empty state: explanation + "Save" button.
- Save feedback: toast + the next notification respects new settings.

**Backend**:
- Migration 0056: `notification_preferences(user_id PK, muted_kinds text[], cadence text, channels text[], updated_at)`.
- Hook into `notifications::fan_out`: skip insert if (user, kind) is muted. Cadence digest: deferred to v1.4 (only the per-kind mute is enforced in v1.3; cadence is stored but treated as `immediate`).
- New endpoints:
  - `GET /admin/api/account/notification-preferences`
  - `PUT /admin/api/account/notification-preferences`

**Tests**:
- Server integration test: muted kind doesn't enqueue a notification for a watcher.

### v0.4 — W15 source bundle streaming + uploaded-bundles panel

**UX design**:
- New panel inside the release-detail dashboard view (`/org/<slug>/releases/<id>`): "Uploaded artifacts" expanded by `kind`. Source bundles get a row each with: platform / file count (extracted from tar headers) / upload time / size / "Delete" button.
- The frame source drawer's "No source" hint links back to this panel.

**Backend (W15)**:
- Replace `Bundle::from_tar_gz` building a full HashMap with a streaming design: on lookup, scan tar entries until matching path, return that file's bytes. LRU cache the decompressed bytes per-(release, file) instead of per-release.
- Add admin endpoint `DELETE /admin/api/projects/<p>/releases/<r>/artifacts/<id>` so the panel's delete works.
- Add count metadata on the release_artifacts row (entry count, total uncompressed size) at upload time, so the panel can render without re-extracting.

**Tests**:
- Source-bundle lookup still passes the v1.2 W3.b test (regression).
- New unit test: streaming lookup returns correct file across 100-entry bundle.

## Mandatory UX checklist (per work unit)

Per the [webapp-UX-is-delivery](~/.claude-profile-1/.../feedback_webapp_ux_is_delivery.md) memory, before declaring any v1.3 checkpoint done, verify:

1. **Entry point** — there's a discoverable navigation path to the feature.
2. **Empty state** — first-time user sees an explanation + CTA, not a blank page.
3. **Loading state** — spinner/skeleton during async work.
4. **Error state** — error banner with the structured-error envelope's `hint`.
5. **Success feedback** — toast / inline confirmation after each mutation.
6. **Edit path** — user can change their previous input.
7. **Delete / disconnect path** — destructive actions get a confirm dialog.
8. **No docs required** — an operator can use the feature from the dashboard alone.

A commit that doesn't tick all 8 doesn't ship.
