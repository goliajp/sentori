## L1 — Roadmap statement

v1.0 + v1.1 covered "ingest → store → query → alert". Source code linking is half-wired (sourcemap upload extracts inline source context but UI never surfaces it cleanly) and the built-in issue system is a thin grouping layer (4 statuses, no labels/priority, no audit trail, no comment UI).

> **v1.2 turns sentori's "look at error → look at source → triage" daily loop from 80% half-built into 100% usable.**

The two halves are coupled: when an operator clicks an issue, they should see the source context *and* be able to triage it (label, prioritize, comment, mute) without leaving the page. Today both are broken in different ways.

### In v1.2

**Source code linking (closes the gap from "sourcemap uploaded" to "operator sees source on click")**:
- **W1** Static diagnosis of why "click frame → see source" has never worked end-to-end (no code change; produces a finding doc that drives W2)
- **W2** Fix source-linking UI/backend: separate "inline source drawer" affordance from "open in external repo" deep-link; inline path works without `source_repo_url`; fall-through hint when no sourcemap exists for the release
- **W3** Native source bundles (Swift / Kotlin / Objective-C inline source) — admitted into v1.2 on 2026-05-20 after the user reviewed the W1 finding and decided "release date can slip; native source can't slip the next train". Comprises CLI upload of project source archive, server-side extraction + path resolution against the same uploaded dSYM / proguard mapping, and dashboard wiring so the same FrameSourceDrawer renders native frames inline.

**Issue triage upgrade (adds the missing primitives Linear-class issue tooling needs)**:
- **W4** `issues.priority` enum (`p0|p1|p2|p3`) + `issues.labels text[]` + admin API + dashboard filter/edit UI
- **W5** `activity_log` table (`issue_id, actor_id, verb, payload jsonb, at`) — unified mutation feed; backfill nothing, start writing on next mutation
- **W6** Add `muted` non-terminal status (distinct from `silenced` which is terminal-but-reopenable, distinct from `resolved`) — operator says "I know about it, don't page me, new events do not regress it"
- **W10** Comment UI in dashboard issue detail — the `issue_comments` table exists and is server-write-able since v1.1; only the read-and-write UI is missing

**Issue collaboration loop (admitted 2026-05-20 alongside the W3 expansion)**:
- **W7** External issue tracker link mirror — Linear (complete the v1.1 outbound scaffold with inbound webhook), GitHub Issues, GitLab Issues, Jira Cloud. Sentori stays the source of truth; external systems get one-way outbound push from us, and refresh `external_title` / `external_status` on inbound webhooks. No two-way sync.
- **W8** Watchers + per-user notification feed — `watchers(issue_id, user_id, since)`, `notifications(user_id, kind, payload, read_at, created_at)`, auto-watch reporter/assignee on issue creation, server SSE feed per user, dashboard notification bell + watch toggle.
- **W9** Sentori MCP server — exposes `sentori_issue_list/get/create/assign/comment/transition/link/watch` etc. so external LLM agents can read and triage. Reuses the existing admin token auth.

**Polish (rolled in here so v1.2 ships clean)**:
- **W2.b** Per-frame "no source" hint when `frame.contextLine` is undefined — instead of the frame just looking thinner, render a one-liner explaining the state (no sourcemap / wrong sourcemap / native frame).
- **W3.d** CLI `--from-dir` mode for `upload source-bundle` (walks a directory, picks platform-relevant files, tars + uploads).

### Explicitly NOT in v1.2 (deferred to v1.3+)

- Two-way sync of any external system — sentori stays the source of truth, external systems get one-way push of metadata, with `external_title` / `external_status` refreshed via inbound webhook. Attempts to mirror assignee / labels / priority in either direction belong elsewhere.
- Jira Server / Data Center support (v1.2 covers Jira Cloud only; PAT-based DC support follows the same adapter shape and can land in v1.3 without breaking changes).
- Per-user notification preferences UI (mute kinds, digest cadence). v1.2 ships the SSE feed; operators can already silence the dashboard tab.

### v0.2 — W4 + W5 + W6 (schema three-pack)

These share an issues-table migration window so we don't ship three migrations in a row.

**W4 — Priority + labels**
- Migration adds `priority text NOT NULL DEFAULT 'p3' CHECK (priority IN ('p0','p1','p2','p3'))` and `labels text[] NOT NULL DEFAULT '{}'`.
- Admin API: `PATCH /admin/api/.../issues/:id` accepts `priority` + `labels`; list filter accepts `priority=` and `labels=` (any-of match).
- Dashboard: priority chip in list view + filter dropdown; label chips with inline edit.
- Verify: `bun run check` clean; migration applies idempotent on prod-like data; new event without explicit priority gets `p3`.

**W5 — Activity log**
- Migration creates `activity_log(id bigserial pk, issue_id uuid fk, actor_id uuid null, verb text not null, payload jsonb not null default '{}', at timestamptz not null default now())`. Indexes on `(issue_id, at desc)`.
- Server hooks: every issue mutation (state transition, assignment, label change, priority change, silence, mute, regression) writes one activity row. Comments also write `verb='commented'` rows.
- Dashboard: timeline tab on issue detail showing reverse-chronological feed.
- No backfill — start fresh on next mutation.
- Verify: mutating any field via admin API writes exactly one activity row; comment write produces one row; dashboard timeline renders the rows.

**W6 — `muted` status**
- Migration extends status check constraint to include `muted`. Update CHECK in single migration with W4+W5 if possible (one schema change cycle).
- Semantics decision (commit in migration comment): `muted` is non-terminal — new events do NOT flip it to `regressed`; only manual unmute or auto-aging-out via release boundary clears it. Differs from `silenced` (which auto-regresses on a new event in the same release).
- Server: ingest path checks `muted` and skips regression flip.
- Dashboard: list filter has `muted` as a 5th tab; transition button on detail; tooltip explains difference vs silenced.
- Verify: integration test — issue at `muted` + new matching event → still `muted`, no regression; same scenario with `silenced` → `regressed`.

### v0.5 — W3.a native source bundle upload

**W3.a — schema + server upload endpoint + CLI**
- Migration documents `release_artifacts.kind = 'source_bundle_ios' | 'source_bundle_android'` (the column is already free-form text; this is a doc + check).
- Server endpoint: `POST /admin/api/releases/{release_name}/source-bundles?platform=ios|android` (multipart). Validates archive is a readable tar.gz, stores blob under SENTORI_DATA_DIR/artifacts/{hash}, upserts `release_artifacts` row.
- CLI command: `sentori-cli upload source-bundle --project <uuid> [--release <r>] --platform ios|android <path>`. Accepts either a directory (tars it on the fly, with `--include` glob + sensible defaults — only `*.swift`/`*.m`/`*.mm`/`*.h` for iOS, `*.kt`/`*.java` for Android) or a pre-built `.tar.gz`.
- Dependencies: none (release_artifacts table already accepts arbitrary kinds).
- Verify: integration test uploads a synthetic tar.gz, hits the existing `/releases/.../artifacts` endpoint, sees the source_bundle row with right kind + content_hash. CLI unit test on the directory→tar.gz path.

### v0.6 — W3.b native source lookup

**W3.b — extract + resolve native frames**
- New module `server/src/source_bundle.rs`: open a release's source_bundle artifact (cached LRU), enumerate tar entries, expose `lookup(file_path: &str, line: u32, n: usize) -> Option<FrameSourceWindow>`. Path matching: longest-suffix-match across entries; fall back to basename. Decompressed cache capped at a small N (extracting on miss is cheap).
- Extend `source_for_frame` (`server/src/symbolicate.rs:224`) to fall through to `source_bundle::lookup` when sourcemap reverse-resolution returns None and the frame's `file` ends in a recognised native extension. Platform inferred from extension first, falls back to the parent event's `platform` field.
- Dashboard: zero changes — the same `FrameSourceDrawer` request shape works because the server returns the same `FrameSourceWindow`.
- Dependencies: v0.5.
- Verify: integration test seeds an iOS event with a known dSYM (existing fixture) + uploaded source bundle, hits the `/source` endpoint, asserts ≥3 lines come back.

### v0.7 — W8 watchers + per-user notifications

**W8 — schema + SSE feed**
- Migration: `watchers(issue_id uuid, user_id uuid, since timestamptz, primary key (issue_id, user_id))`; `notifications(id bigserial, user_id uuid, kind text, payload jsonb, read_at timestamptz null, created_at timestamptz default now())`.
- Auto-watch the assignee on patch_issue (W4 already had the snapshot); auto-watch the project's super-admins / org-admins on a new issue's creation in W5's path.
- On every `activity_log::write` for a watched issue, enqueue one notifications row per (user_id) watcher whose user_id != actor. (Skip self-notifications.)
- SSE endpoint `GET /admin/api/notifications:stream` filtered to the caller's user_id; broadcast via a tokio broadcast channel from the activity_log write site.
- Dashboard: notification bell with unread count + dropdown; per-issue "Watch" toggle on detail.
- Verify: integration test asserts watcher + non-actor → notification row + SSE event delivered; muting the issue does NOT auto-watch.

### v0.8 — W9 sentori MCP server

**W9 — read + triage tools exposed via MCP**
- `mcp_server` crate (or module on the existing server binary) listening on stdio + HTTP per the MCP spec. Auth via existing admin token (`sk_…`) for HTTP transport; stdio inherits the caller's env.
- Tools (mirroring the tasks project's MCP API but for error issues):
  * `sentori_issue_list { projectId, status?, priority?, label?, limit? }`
  * `sentori_issue_get { issueId }`
  * `sentori_issue_comment { issueId, body }`
  * `sentori_issue_transition { issueId, status }`
  * `sentori_issue_assign { issueId, userId | unassign }`
  * `sentori_issue_set_priority { issueId, priority }`
  * `sentori_issue_set_labels { issueId, labels[] }`
  * `sentori_issue_link { issueId, kind: github|gitlab|jira|linear, externalId, externalUrl }`
  * `sentori_issue_watch { issueId, watch: true|false }`
- Each tool reuses the admin API handlers under the hood (no parallel logic).
- Verify: a smoke test in `server/tests/mcp_smoke.rs` lists tools, calls one read tool + one write tool, asserts side effects in the DB.

### v0.9 — W2.b + W3.d polish

**W2.b — per-frame no-source hint**
- Below FrameRow's filename line, when `frame.contextLine === undefined`, render a muted single-line explanation ("no sourcemap for this release" / "no source bundle for this iOS frame's release" / "native frame — upload a source bundle").
- Heuristic decides which copy: file extension + release-level sourcemap-status (cached by the existing banner query).
- No new server endpoint; reuse `sourcemap-status` data already on hand.

**W3.d — CLI --from-dir mode**
- Extend `sentori-cli upload source-bundle` to accept a directory; the CLI builds the tar.gz on the fly using a small hand-rolled tar + node:zlib (no new runtime deps). Filter:
  * `--platform ios` → `*.swift`, `*.m`, `*.mm`, `*.h`, `*.hpp` only
  * `--platform android` → `*.kt`, `*.java` only
- Skips `node_modules`, `.git`, `Pods`, `build`, `DerivedData` by default; `--include` overrides.
- Verify: bun test produces a tar.gz with the expected entry set; existing path-to-tar.gz mode keeps working.

### v0.11 — W7.b GitHub Issues

**W7.b — GitHub Issues adapter**
- Auth: GitHub App or PAT — start with PAT for v1.2 (one secret per org, stored in `integrations.config`). Document how to scope the token (`repo` scope on the target repo).
- Outbound: on Sentori `new_issue` create a corresponding GitHub Issue with `title = {error_type}: {message_sample}`, body links back to the Sentori issue; on `resolve` close the GitHub Issue; on `regress` reopen + comment.
- Inbound: GitHub webhook (`issues` event) refreshes the link row when the GH Issue's title/state changes.
- Settings UI: minimal "paste PAT, pick default repo" form.

### v0.12 — W7.c GitLab Issues + W7.d Jira Cloud

**W7.c — GitLab Issues adapter** (mirror of W7.b)
- Auth: GitLab PAT (gitlab.com or self-hosted via configurable base URL).
- Same outbound/inbound shape.

**W7.d — Jira Cloud adapter**
- Auth: Atlassian API token + email (basic auth over HTTPS). Workspace base URL stored in integration config.
- Outbound: `POST /rest/api/3/issue` with the configured project key + issue type ("Bug" default). Status transitions via `/transitions` endpoint (workflow-dependent — we just call the workflow's `Done` / `In Progress` / `To Do` transition by name; misconfigured workflows surface as a user-facing warning).
- Inbound: Jira webhook (jira:issue_updated) refreshes the link row.
