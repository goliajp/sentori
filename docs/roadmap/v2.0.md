## NEVER rule — load-bearing (above every L2 work unit)

> **The SDK must never harm the host app.** Every public `sentori.*`
> API is wrapped via `safeFn` / `safeAsync` (see design spec
> "Failure semantics"). Internal errors silently fail, optionally
> self-report via a circuit-breaker; the host app never sees a
> thrown error, a rejected promise, a frame drop, a network failure
> that we caused, or anything else attributable to Sentori.
>
> Per CLAUDE.md performance budgets: `< 1 %` main-thread,
> `< 5 ms` per tick.
>
> Every W1–W4 PR has a NEVER-rule test for every new public API.
> Failing this test is **stop-ship**. This rule beats LLM
> friendliness, API elegance, and feature coverage.

## Compatibility model

- **Server**: forever backward-compatible with v1 SDK wire format.
  v2 adds optional fields; v1 SDK requests parse cleanly with new
  fields absent. Regression test `server/tests/v1_compat.rs` gates
  this.
- **v1.x SDK packages**: frozen on npm at their last v1.x version.
  Stay subscribed by not upgrading.
- **v2 SDK packages**: clean redesign (`@goliapkg/sentori-react-native@2`).
  Opt-in by upgrading. Aliases / dual exports / SpanHandle /
  Event-type-named exports / `span.finish()` are gone. The
  migration guide ships in W4 as `docs/recipes/v1-to-v2-migration.md`.

## L1 — Roadmap statement

> **Sentori v2.0 turns manual instrumentation into a first-class
> story.** Today the SDK ships eight pieces of manual surface area
> (spans, moments, breadcrumbs, track, metrics, trail, trace context,
> capture-exception) scattered across modules. Customers reach for
> them by reading source. v2.0 surfaces them as a cohesive
> `sentori.*` API set, adds the one missing piece (`captureMessage`
> — issue reporting without forcing an `Error` instance), and ships
> docs + recipes that explain "how do I instrument my own code"
> as one connected narrative instead of eight isolated paragraphs.

The bar is **a developer should be able to write all of the following
in their host app without needing to read the SDK source:**

```ts
// v2 SDK shape — what we ship.

sentori.captureMessage('user denied location permission', {
  level: 'warning',
  tags: { feature: 'maps' },
})

const trace = sentori.startTrace('checkout-flow')
const result = await sentori.withSpan({ name: 'db.query users', parent: trace }, async () => {
  return await db.query(…)  // span auto-ends with status from promise outcome
})
trace.end({ status: 'ok' })

sentori.track('cart.add', { props: { sku, qty } })
sentori.recordMetric('cart.size', cart.length)
sentori.addBreadcrumb({ message: 'clicked Buy', type: 'user' })

const moment = sentori.startMoment('signup', { plan: 'pro' })
moment.checkpoint('email-confirmed')
moment.end()

sentori.setTag('rollout', 'dark-mode-v2')
await sentori.flush()
```

…and the dashboard shows each in the right place: messages join
issues, manual traces appear in Traces alongside auto-instrumented
ones, moments still have their funnel view, track / metrics keep
their analytics view.

### Explicitly NOT in v2.0

- **WASM trace assembly / on-device span tree reconstruction.** Was
  flagged in v1.1 NOT-in; stays out — manual instrumentation works
  at the existing transport layer.
- **OpenTelemetry interop.** Sentori speaks its own protocol per
  [project-direction]; an OTLP export path is its own L2 (v2.1+).
- **`sentori.captureMessage` async result inspection.** The function
  is fire-and-forget like `captureException`. If a callsite needs to
  know the event id, that's a separate work item (probably never).
- **Custom event kinds beyond `'message'`.** Resist the urge to add
  `'log'` / `'audit'` / `'security'` — the kind axis is for dashboard
  triage shape, not for free-form taxonomy. Use tags for that.
- **Renaming `track` to `captureEvent`.** Discussed; rejected. `track`
  is the established analytics-line verb and renaming churns every
  existing dogfood / customer integration.

### v0.1 — W1 `captureMessage` end-to-end

**What:** Add the `captureMessage(message, level?, extras?)` function
to every SDK; extend `EventKind` server-side to accept `'message'`;
extend Event schema with a `level: 'fatal' | 'error' | 'warning' | 'info' | 'debug'`
column that's required for `kind: 'message'` and ignored for others.
Dashboard's Issues list renders message-kind events without expecting
a stack trace; Issue Detail has a "no stack — manual report" panel
shape.

**Requirements:**

- Naming aligned with W6 policy: `sentori.captureMessage(...)` (flat
  namespace; alongside `sentori.captureException`).
- Issue grouping: messages group by `(release, message-hash, tags)`
  default. Stack-based grouping doesn't apply.
- Sampling: `SamplingConfig.messages: number | null` (default 1.0).
  `captureMessage` drops events when `Math.random() > rate`.
- Privacy: same redaction as captureException — message string goes
  through the existing PII scrubber.
- Storage: messages live in the same `events` table; the new `level`
  column is nullable; existing rows get NULL.
- Dashboard: Issue card shape changes when there's no stack — leading
  with the message body, level chip on the right, no "view stack"
  button.

**Resources:**

- SDK: `sdk/core/src/types.ts` (EventKind, MessageLevel, Event); new
  `sdk/core/src/capture-message.ts` (or fold into existing capture.ts
  per-SDK); per-framework re-exports.
- Server: `server/src/event.rs` (EventKind enum + Event struct); new
  migration adding `events.level` column + index for filtering;
  `server/src/api/events.rs` ingestion path accepts new shape.
- Dashboard: `web/src/modules/issues/view.tsx` + Issue Detail. Add
  level filter to URL state per [project-v05-direction].

**UX design step (required before impl):**

- How does an Issue card look when message-kind?
- How does Issue Detail render with no stack — what fills the
  vertical space the stack panel used to take?
- Level chip color + position (info/debug muted, error/warning
  accent, fatal danger).

### v0.2 — W2 Manual span / trace surface integration

**What:** Re-export the existing `core.startSpan` / `startMoment` /
`withSpan` / `setActiveSpan` from every framework SDK's `sentori.*`
namespace. Add `sentori.startTrace(name, opts?)` as a `startSpan({
parent: null, name, ...opts })` shortcut so the "I want to open a
new trace from scratch" intent is explicit. Extend `SamplingConfig`
with `messages` (W1) and ensure manual spans honour the existing
`traces` rate.

**Requirements:**

- `startTrace(name, opts?)` returns a `SpanHandle` (same shape as
  `startSpan`, no new type to learn).
- Re-exports are stable per-SDK; signatures match `core` exactly so
  there's no surface drift across packages.
- `sentori.withSpan(span, fn)` works on RN (synchronous module-scoped
  fallback already in place per `trace-context.ts:18–25`); document
  the Node-only AsyncLocalStorage path separately.
- Web SDK manual span surface is identical to RN's. No framework
  bias.
- `SamplingConfig.messages: number | null`. Default 1.0.
  `SamplingConfig.traces` already exists; manual spans honour it.

**Resources:**

- `sdk/react-native/src/index.ts` + every other framework SDK index —
  add the re-exports. Mechanical.
- `sdk/core/src/spans.ts` — possibly extend `StartSpanOptions` with
  better defaults for the "manual root" case (auto-tag with
  `source: 'manual'`?).
- Tests per SDK confirm the new re-exports exist + types align.

**UX design step (required before impl):**

- No new dashboard view required (manual traces show in existing
  Traces module). But: how does the Traces module distinguish
  manual-rooted traces from auto-instrumented? Tag-based filter
  (`source: 'manual'`)? Icon?

### v0.3 — W3 Instrumentation cohesion + cross-API links

**What:** `track` / `recordMetric` / `addBreadcrumb` move into the
`sentori.*` namespace where they aren't already (some SDKs only had
JS-side helpers, not the namespace). Behaviour upgrade: `track`
auto-emits a `breadcrumb` of type `'track'` with the event name +
props subset, so when a captureException fires near a tracked event
the breadcrumb trail shows the customer journey. `recordMetric`
gains a `parent` option that ties the metric point to a span — the
dashboard can correlate.

**Requirements:**

- Behavioural change to `track`: auto-breadcrumb is opt-in via
  `init({ capture: { trackAutoBreadcrumb: true } })`. Defaults
  **off** to preserve existing customer breadcrumb shape; doc the
  recommended `true` for new integrations.
- `recordMetric(name, value, tags?, { parent?: SpanContextLike })` —
  parent stored as `tags.span_id` so existing transport doesn't
  change shape, dashboard joins on tag.
- Document the layering: "moment > trace > span > breadcrumb >
  track > metric — five granularities for five different signal
  shapes."

**Resources:**

- `sdk/react-native/src/track.ts` — add the auto-breadcrumb branch
  + the new init option.
- `sdk/react-native/src/metrics.ts` — add `parent` option.
- `sdk/core/src/types.ts` — add `trackAutoBreadcrumb` to
  `CommonInitOptions`.
- Dashboard: span detail panel gets a "related metrics" row when
  any metric ts is within the span + tags match.

**UX design step (required before impl):**

- "Related metrics" row layout in span detail.
- Track-as-breadcrumb visual treatment in Issue Detail (distinguish
  from user/network/route breadcrumbs).
