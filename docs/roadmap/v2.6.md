## Why these two together

The find-threat lens answers a single operator question — "is anything
trying to spoof us, MITM our installs, or otherwise behave like an
attacker?" Two surfaces feed that answer:

1. **Cert monitor** — "is a certificate appearing for one of our
   domains that we didn't issue?" CT-log polling catches mis-issued
   or rogue certs before they're in active use; the operator's job
   is to scan the observation feed, click into `crt.sh/<id>` for
   anything unfamiliar, and trace which issuer / serial it is.
2. **Posture** — "are our SDK callers reporting evidence of
   tampering or downgrade?" Pin anomalies (TLS pin mismatches) +
   trust scores (low-score installs report disproportionately on
   `reportSecurity()` channels) surface the in-the-field signals
   that real users are encountering an attack.

Both are passive monitors that the operator scans daily — they sit
in the same workflow rung. Splitting them across releases would just
delay one without value.

## BI-five answers per module

(Per [[project-v22-lens]] — every flipped module owes 5 answers.)

**Cert monitor**

1. **dim** — one row per CT observation (`domain × cert serial`)
2. **measure** — first seen relative time, issuer name, not-before /
   not-after validity window
3. **drill** — external link `crt.sh/?id=<certId>` for the raw cert
4. **compare** — n/a (each observation is identified by serial; cross-cert delta is the operator's reading of the feed, not a side-by-side UI)
5. **cache** — 10-min server-side poll cron; client refetches on
   mount + manual refresh

**Posture** (4-tab surface)

1. **dim** — server name (Pin tab) / install id (Trust tab) /
   weight name (Explorer tab) / identity hash (Federation tab)
2. **measure** — report count + install count (Pin) /
   trust score 0–100 (Trust) / simulated re-score delta (Explorer) /
   project hit count (Federation)
3. **drill** — Pin / Trust → no drill (the row IS the answer);
   Explorer → in-browser sim only; Federation → cross-project
   user view
4. **compare** — Explorer's weight slider IS the compare axis (before
   / after re-score, in-browser via Rust→WASM)
5. **cache** — 60-second refetch on Pin + Trust tabs (`refetchInterval`
   + `placeholderData`); Federation + Explorer are user-triggered

## What v2.6 does NOT do

- **Privacy stays hidden.** It's the engineering-hygiene lens
  anchor; the find-threat lens is about *attackers*, hygiene is
  about *us getting our house in order* (sourcemap coverage,
  PII redaction QA). Different operator workflow.
- **No new server endpoint.** Both modules already have their
  admin endpoints (`pinAnomalies` / `trustScores` /
  `listCertWatchDomains` / `listCertObservations` /
  `addCertWatchDomain` / `deleteCertWatchDomain`).
- **No `/explore` consumer rewrite.** Both views have native
  endpoints with shapes that don't fit the unified-grammar bench
  (`pin_anomalies` aggregates server-name buckets;
  `trust_scores` is a windowed top-N over `security_reports`).
  Unifying onto `/explore` is a long-tail rewrite, not v2.6 work.
- **No moments verdict.** The hidden-modules audit left
  `moments` straddling find-user / engineering-hygiene; v2.6
  doesn't resolve that. It'll be decided when whichever lens
  opens next picks it up.
