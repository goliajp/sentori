## L3b — Cold plan (per checkpoint)

Each work unit's L3a hot plan **must include** an explicit UX design step BEFORE backend / frontend impl. Per the [webapp-UX-is-delivery] memory.

### v0.1 — W16 Email notification channel

**UX design**:
- Entry point: `/account/notifications` (the W14 page) — when the operator checks "email" channel and saves, future notifications also email them.
- A "Send a test email" button beside the email channel checkbox to validate SMTP is wired.
- Email body: HTML + plain text. Subject like `[sentori] $verb on $error_type`. Body: who did it, link to the issue, payload preview.
- Email unsubscribe: a per-user "off" toggle that flips channels to `[in_app]`.

**Backend**:
- Reuse `crate::mailer` (`lettre` already in tree). Build a `notification_email` template module.
- Hook into `notifications::fan_out`: after persisting the `notifications` row, check the user's `channels` — if `email` is there, dispatch an email via the existing mailer transport.
- New `notifications_email_log` table to track delivery (delivered / failed / bounced) so the v0.2 digest worker has audit data.
- New admin endpoint `POST /admin/api/account/notification-preferences:test-email`.

**Tests**:
- Server: integration test that toggles channel='email', triggers an activity, asserts the email log row appears with `delivered`.
- Server: test-email endpoint returns 200 + log row.

Verify: trigger an activity_log write, see the email log row.

### v0.2 — W17 Hourly + daily digest workers

**UX design**:
- Account page: when cadence is "hourly" or "daily", show the next-scheduled-digest-time (e.g. "next at 14:00 UTC").
- Email body is a digest: groups notifications by issue, shows up to 20 issues, "5 more — see dashboard" link.

**Backend**:
- New `digest_runs` table to track scheduling per (user, cadence).
- Background task in the server binary using `tokio::time::interval` (top of every hour for hourly; daily at a configurable UTC time, default 09:00). Picks up users whose cadence ∈ {hourly,daily} and whose unread notifications since last digest > 0.
- Sends one digest email; updates `digest_runs.last_sent_at`.

**Tests**:
- Manually trigger digest via `POST /admin/api/digests:run-now` (admin-only diagnostic endpoint).
- Assert that immediate notifications STILL email (cadence='hourly' doesn't suppress immediate channel='email' if the operator selected it — wait, that conflicts. Let me re-think: cadence=hourly should mean "no immediate, batch only".)
- Decision: cadence ≠ 'immediate' suppresses per-event emails; only digest emails fire.

### v0.3 — W18 Per-issue mute + W19 auto-watch first-opener

**W18 UX**:
- Issue detail page: a "Mute notifications" toggle next to "Watch" / "Unwatch". When muted, notifications for this issue are suppressed for this user (regardless of global preferences).
- "Muted" indicator visible on issue list rail when current user has muted.

**W19 UX**:
- Silent — operator who first navigates to an issue's detail view auto-becomes a watcher, same as v1.2's "assigned-to" auto-watch. Visible only via the Watch button showing "✓ Watching" on subsequent visits.

**Backend**:
- New `issue_user_mutes (user_id, issue_id, since)` table.
- Notification fan_out joins it: skip if (user, issue) is muted, even when the kind isn't globally muted.
- `GET /issues/{id}/events` (or wherever issue-detail loads): after the issue is fetched, also call `notifications::add_watcher(pool, issue_id, user_id)` (idempotent). Best-effort.

### v0.4 — W20 Jira OAuth 3LO + W21 GitHub App Valkey cache

**W20 — Jira Cloud OAuth 3LO**:
- Atlassian's OAuth 2.0 (3LO) — requires Atlassian to provision a "Connect App" with the operator's tenant. The flow:
  1. Operator clicks "Connect Jira (OAuth)" on the integrations card.
  2. Sentori redirects to `https://auth.atlassian.com/authorize?audience=api.atlassian.com&client_id=...&scope=...&redirect_uri=...&state=...&response_type=code&prompt=consent`.
  3. Atlassian redirects back with `?code=...`.
  4. Sentori POSTs to `https://auth.atlassian.com/oauth/token` for an access token + refresh token.
  5. Sentori queries `/oauth/token/accessible-resources` to discover the workspace site URL + cloud id, stores in config.
- New JiraConfig variant `OAuth { access_token, refresh_token, expires_at, cloud_id, site_url, project_key, issue_type }`.
- Token refresh: when access_token expires, use refresh_token to mint a new one. Cache in process; bonus: use Valkey if available.
- Card UI: now THREE modes (Cloud-OAuth / Cloud-API-token / Server-PAT).

**W21 — GitHub App install-token Valkey cache**:
- Refactor `INSTALL_TOKEN_CACHE` from `OnceLock<Mutex<HashMap>>` to a Valkey-backed cache when `VALKEY_URL` is set. Key: `gh-app-install-token:{installation_id}`, value: JSON {token, expires_at}, TTL 55 min.

### v0.5 — W22 Webhook delivery retry UI

**UX**:
- Settings → Webhooks page (or a section inside the Integrations area).
- Lists pending + failed webhook deliveries per integration. Columns: integration / event kind / attempted_at / status / last_error.
- "Retry" button per row triggers a manual re-delivery.

**Backend**:
- The existing `webhook_deliveries` + `webhook_dispatch` modules already track retries; we just need a list/retry endpoint pair and the UI.

### v0.6 — W24 Per-org label catalog + colors + p0 SLA timer + labels in fingerprint

**Schema**:
- New `org_labels (id, org_id, name, color, sla_priority_hours)` — registered labels per org. The existing `issues.labels text[]` referenced by NAME stays the same.
- New `issue_priority_sla_breaches (issue_id, breached_at)` — tracks p0/p1 issues that exceeded the SLA timer without resolution.
- New `issues.fingerprint_labels text[]` (or rolled into existing fingerprint hash) — opt-in: labels participate in grouping. Default off; flip per-project setting.

**UX**:
- `/org/<slug>/settings/labels` — new page. CRUD: create / rename / recolor / delete labels. Lists how many issues use each.
- Issue detail's `TriageRow` label chips now show colors from the catalog.
- Per-project setting "Include labels in fingerprint?" toggle.

### v0.7 — W25 True streaming + W26 Multi-bundle + W27 Per-event sourcemap-status

**W25 — True streaming**:
- Replace v1.3's "decompress + scan tar" lookup with a single-pass `tar::Archive::entries()` scan, returning the first matching path's bytes without buffering the rest. Already mostly done — main improvement is dropping the "read whole bytes file first" call.
- Benchmark: a 50 MB tar.gz with the target file at position 80% takes O(40 MB decompress) per lookup. Same as v1.3.

Hmm — actually for true streaming, replace `tokio::fs::read(blob_path)` with `tokio::fs::File::open` + `tokio_util::compat` adapter for `tar::Archive<R: Read>`. Read on demand.

**W26 — Multi-bundle**:
- Schema: drop unique constraint on (release_id, name) and accept multiple `source_bundle_<platform>` rows per release. Each is identified by an optional "module label" the operator supplies on upload.
- CLI: `--module <label>` flag on `upload source-bundle`.
- Dashboard: release detail panel shows N bundles per platform, grouped by module label.
- Lookup: try all bundles for the (release, platform) until one resolves.

**W27 — Per-event sourcemap-status query**:
- FrameRow's "no source" heuristic replaced by a server-side check: "does THIS release have the right kind of artifact?" The dashboard fetches via the existing W2 `sourcemap-status` endpoint, but per-release variant.
- New: `GET /admin/api/projects/{p}/releases/{r}/source-coverage` → `{ hasJsSourcemap, hasIosBundle, hasAndroidBundle }`.
- Banner copy + per-frame hint now match the specific release's state.

### v0.8 — W23 Cross-org integration sharing / templating

This is the architecture lift. The user opted to keep it in v1.4 explicitly.

**UX design**:
- "Templates" tab on the Integrations settings page. Lists integration configs the operator can save as a template (e.g. "GitHub for my main org"), then re-apply to other orgs.
- "Use template" button on the per-org adapter card — pops a list of templates the operator has access to.
- Templates are per-USER (the operator who created them); they can mark as "share with my org admins" for visibility to teammates.

**Schema**:
- New `integration_templates (id, owner_user_id, kind, config jsonb, name, shared_with_org_id (nullable), created_at, updated_at)`.
- The config is the same shape stored in `integrations.config` minus the per-org bits.

**Backend**:
- CRUD endpoints under `/admin/api/account/integration-templates`.
- "Apply template" action: POSTs the template's config to the existing `/integrations/<kind>/configure` endpoint.

### v0.9 — W28 SDK-side build-time native source-bundle helper

**UX design**:
- `@goliapkg/sentori-react-native` ships a `postinstall.js` hook + a metro/babel transform that:
  - During `expo build` / `react-native bundle --platform ios|android --release`, identifies the source tree paths.
  - Builds a tar.gz of the relevant files (Swift / Kotlin / OC).
  - Calls `sentori-cli upload source-bundle --from-dir <ios|android>/sources --platform ios|android --release $RELEASE` automatically.
- Configurable via `sentori.config.json` in the app root: `{ uploadSourceBundle: true, sentoriToken: process.env.SENTORI_TOKEN }`.

**Effort**: requires real RN build integration; high risk of "RN released" / "Metro changed" breakage. Out-of-band tests are necessary.

## Mandatory UX checklist (per work unit)

Same as v1.3. Re-stated here so contributors don't have to chase the memory file:

1. Entry point   — discoverable nav path
2. Empty state   — first-time user sees explanation + CTA
3. Loading state — spinner/skeleton on async
4. Error state   — banner with structured-error hint
5. Success feedback — toast / inline confirm
6. Edit path     — user can change their input
7. Delete path   — destructive action gets confirm
8. No docs required — operator can use it from dashboard alone

A commit that doesn't tick all 8 doesn't ship.
