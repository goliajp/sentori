## Why

Migrating off `expo-notifications` is one of the most common
sources of friction for teams adopting Sentori. The upstream SDK
ties the app to Expo's exp.host push service for the actual
delivery; switching to Sentori means switching transports, which
historically meant rewriting client-side registration code on top
of `sentori.push.register()`.

This release reframes that as a one-line import swap. Drop-in
shim covers the P0 + P1 APIs the typical app uses; everything else
either has a same-day workaround documented or queues up as a
follow-up minor where the native module grows another method.

## What v2.18 does NOT do

- **No new native module exports.** Everything routes through the
  existing v2.7+ surface (`pushGetStatus` / `pushRequestPermission` /
  `pushRegister` / `pushUnregister` / `pushDrainState`). The
  unsupported list lives there until we add native methods for
  badge / channels / actions / cold-start / dismissal.
- **No exp.host proxy.** `getExpoPushTokenAsync` returns the raw
  native token wrapped in the upstream envelope shape — your
  server-side code must POST to Sentori, not exp.host. Recipe is
  explicit.
- **No local notification scheduling.** `scheduleNotificationAsync`
  throws. Server-side `delay` field on the send is a candidate for
  v2.19+.
