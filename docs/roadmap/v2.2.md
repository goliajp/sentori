## NEVER rule ‚Äî load-bearing (above every L2 work unit)

> v2.2 is a **read-side** refactor. The SDK contract from v2.0/v2.1
> is unchanged; ingest pipeline is unchanged; storage is unchanged.
> Specifically:
>
> - No new SDK API. No SDK behaviour change. No new wire format.
>   The performance budget for the SDK (`< 1 %` main-thread, `< 5
>   ms` per tick) stays trivially honoured because there is no SDK
>   delta at all.
> - The dashboard NEVER blocks on `/explore`. Every consumer treats
>   the endpoint as a slow-read (P95 target 200 ms, hard timeout 5
>   s) and falls back to a skeleton + clear error state on miss.
> - `/explore` is **safe to call from an LLM agent** by design: it
>   accepts only enum-constrained `dim` / `measure` / `filter`
>   values, never raw SQL. Adding a new dim / measure means adding
>   a Rust match arm in `explore.rs` ‚Äî never threading a string
>   through to a query builder.
>
> **v2.2-specific addition:** "one lens at a time." The dashboard
> still ships dozens of feature surfaces; v2.2 only re-opens the
> **find-bug** ones (Issues, Releases). Find-slow / find-user /
> find-threat are explicitly v2.3+. Touching a hidden module to
> "polish it a bit" without opening its lens is **stop-ship** for
> a v2.2 PR ‚Äî it dilutes the scope this doc is closing.

## Compatibility model

- **Server:** v2.2 adds ONE new admin endpoint (`POST
  /admin/api/projects/{p}/explore`). Every existing endpoint ‚Äî
  `/v1/events*`, `/admin/api/projects/<p>/issues*`,
  `/admin/api/projects/<p>/runtime-metrics/query`, the
  `/endpoint-checks` suite ‚Äî is untouched. The v2.1 regression
  suite still applies verbatim; no new compat tests needed.
- **SDK:** no change. v2.1 SDK matrix (npm) covers v2.2 dashboards
  without a republish. Hosts that haven't moved off v2.0 still
  work ‚Äî v2.2 doesn't gate on anything new the SDK reports.
- **Dashboard:** existing routes (`/main/issues`, `/main/releases`,
  `/main/runtime`, `/main/health`) keep their URLs. v2.2 swaps the
  query layer behind some of them and adds **picker UI**
  (`dim`/`measure`/`window`) above the list ‚Äî additive. A user who
  doesn't touch the pickers sees a list that's pretty much the
  same shape as before. The URL gains new optional query params
  (`?dim=‚Ä¶&measure=‚Ä¶`); old links without them resolve to the
  default preset.

## L1 ‚Äî Roadmap statement

> v2.1 made Sentori a measurer (auto-instrument runtime metrics
> + synthetic uptime probes). v2.2 is about **using the data we
> already collect** ‚Äî three years of events, issues, releases ‚Äî
> to answer the operator's actual Monday-morning question:
>
> > "Which release introduced the most new pain in the last seven
> >  days, and which user-facing issues are blocking the largest
> >  cohort right now?"
>
> The old dashboard had 16 modules and the operator only ever
> opened Issues. The fix isn't to add a 17th module; it's to
> redesign the modules we have around a single mental model.
>
> v2.2 adopts the **find-bug lens** as the first such redesign:
>
> 1. **`/explore` query endpoint** ‚Äî one HTTP route, a whitelist
>    of `dim √ó measure √ó filter`. UI and LLM agents are both
>    consumers; both call the same shape and read the same JSON
>    back. There is no separate "agent API" ‚Äî the UI's URL state
>    *is* the agent's query.
>
> 2. **Module = preset of `/explore`** ‚Äî Issues, Releases, and
>    (later) Users are rebuilt as thin wrappers over `/explore`
>    with a fixed `dim` per module + user-mutable `measures` +
>    `filters` driven from URL state. Cross-module drills are
>    just "open the other module with a different preset."
>
> 3. **Lens discipline** ‚Äî every other monitor module
>    (`metrics`, `vitals`, `moments`, `audience`, `cert-monitor`,
>    `posture`, `privacy`, `live-debug`, `alerts`, `traces`)
>    stays `hidden: true` in the registry. Their data still
>    flows in; the dashboard just doesn't advertise them until
>    a future lens (find-slow / find-user / find-threat / etc.)
>    reopens them with a designed surface.
>
> The cost of pulling a module out of `hidden: true` without a
> lens is dashboard sprawl ‚Äî exactly the failure mode v2.2 is
> correcting. **Don't.**

### Explicitly NOT in v2.2

- **Re-opening hidden modules.** `traces`, `metrics`, `vitals`,
  `moments`, `audience`, `cert-monitor`, `posture`, `privacy`,
  `live-debug`, `alerts` stay `hidden: true`. Their lens
  (find-slow / find-user / find-threat / engineering-hygiene)
  isn't being designed in v2.2. A "while we're here" un-hide is
  exactly the scope creep this version is meant to push back on.
- **New dims on `/explore`.** `issueEq` / `device.os` /
  `issue.priority` / `severity` would be useful (per-issue
  sparkline in W3 wants `issueEq`), but each one expands the
  agent-callable surface and needs its own thought about SQL
  fanout. Defer to v2.3 dim-grammar expansion. Sparklines on the
  Issues list wait until then.
- **New measures.** `crash_free_rate` (Sentry has it; legitimate
  ask) needs a session schema we don't have. `new_issue_count`
  (events in window where `first_seen ‚â• windowStart`) is cheap
  but no v2.2 module needs it. Defer.
- **Cross-org `/explore`.** v2.2 endpoint scopes to one project
  per call. Superadmin / cross-org analytics is its own L2 (v2.3+).
- **Users module rework.** The `users` module is already visible
  and was opened under the **v2.3 / v2.4 identity-lookup** plan
  (`docs/design/sdk-v2.3-redesign.md` ¬ß5). It's NOT a v2.2 lens
  surface; touching it here would mix lenses.
- **Cmd+K commands for slice presets** ("Open issues in release
  X"). Worth doing, but the chord registry is still single-letter
  (v2.1.2). Multi-token commands are a UX iteration on CmdK, not
  a v2.2 deliverable.
- **`/explore` write path** ‚Äî saved views, alert-on-result, etc.
  v2.2 is strictly a **read** endpoint. Saved views land with
  whatever lens needs them first.
- **Performance work on `/explore`.** v2.2 endpoint runs the
  release agg as three separate SQL queries merged in Rust; that
  was the simplest shape. If a real project's row counts make
  this slow, the fix is a single multi-CTE query (or
  pre-aggregation), but no profiling has surfaced this as a
  bottleneck. Defer until it bites.
