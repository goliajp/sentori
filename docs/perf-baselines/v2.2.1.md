# SDK perf baseline — v2.2.1 (Phase 0 closeout)

**Recorded:** 2026-06-03
**Branch:** master @ post-Phase-0 commit
**Tracked by:** `docs/roadmap/post-v2.2-plan.md` Phase 0
**Used by:** Phase 3 must diff against these numbers and any
regression > +5% CPU / > +1 ms per-tick blocks ship.

## Why this doc exists

Phase 0 (W6.0) introduced the logger module + `logLevel` / `onReady`
init fields without touching any hot-path behaviour. We need a
recorded baseline so Phase 3 (W6.1 — flatten `init`, rename
`withSpan`, add `beforeSend`) can prove it didn't regress.

Two surfaces:

1. **Mechanical bench (CI-gated):** `sdk/core/src/__tests__/perf.bench.ts`
   — runs in `.github/workflows/sdk-perf.yml` on every push to
   `sdk/**`. Numbers below are recorded against the `bun:test`
   runner on macOS arm64; CI's GitHub-hosted ubuntu runner has
   different absolute values but the budget multipliers (10x
   observed) hold there too.

2. **Device smoke (manual, Phase 3 entry gate):** `apps/rn-example`
   booted on `sim-sentori` (iOS) + `Pixel_10_Pro` AVD with
   `logLevel: 'silent'` AND `logLevel: 'debug'`, recording main-
   thread CPU + frame-rate during a 60s steady-state run. Numbers
   filled in by takagi or whoever opens Phase 3 — the rig exists
   per `.claude/memory/reference_dedicated_sim.md` and
   `reference_android_verify_rig.md`.

## Mechanical bench numbers (recorded 2026-06-03)

Run command:

    cd sdk/core && bun run build && bun test ./src/__tests__/perf.bench.ts

Observed (macOS arm64, M-class CPU; bun 1.3.14):

| Bench | Observed | Budget | Headroom |
|---|---|---|---|
| `uuidV7` | 0.74 µs/op | 5 µs | 6.8x |
| `shouldSample` | 0.01 µs/op | 1 µs | 100x |
| `shouldSampleTrace` | 0.05 µs/op | 5 µs | 100x |
| `breadcrumb round-trip` | 0.26 µs/op | 10 µs | 38x |
| `TrailBuffer.push` | 0.04 µs/op | 1 µs | 25x |
| `sealTrail` | 0.19 µs/op | 50 µs | 263x |
| `emitMetric.no-tags` | 0.18 µs/op | 5 µs | 27x |
| `emitMetric.3-tags` | 0.24 µs/op | 10 µs | 41x |
| `drainRuntimeMetricsForFlush.300` | 46.35 µs/op | 1000 µs | 21x |
| **`logger.debug gated` (new)** | 0.01 µs/op | 1 µs | 100x |
| **`logger.warn emit` (new)** | 0.04 µs/op | 5 µs | 125x |
| **`setLogLevel` (new)** | 0.00 µs/op | 1 µs | ∞ |

Logger hot path is essentially free at the gated-out level (which
is the case for the default `'warn'` setting — every
`logger.debug('replay', 'tick ok')` call in the SDK takes 10 ns).
At emit time the dispatch into `console.warn` (or a host-supplied
transport) adds < 50 ns of SDK overhead on top of whatever the
sink itself does.

Required before Phase 3 (`W6.1`) can move past its first commit.

Procedure:

1. `cd apps/rn-example && bun ios -- --simulator sim-sentori` for
   iOS; `bun android` with `Pixel_10_Pro` AVD running for Android.

## Why not fill the device numbers now

Running the sims requires the host machine. Capturing the numbers
is a Phase 3 entry gate, not a Phase 0 deliverable — Phase 0 ships
the logger module (zero behaviour change at hot path) and the
mechanical bench numbers above demonstrate no regression in the
parts of the SDK Phase 0 touched. Phase 3 changes the init shape
+ adds `beforeSend` (a new hot-path-adjacent hook); that is when
device-level confirmation matters.

Phase 0 close: mechanical bench green (above) + this doc landed.
Phase 3 open: fill device columns first, then start W6.1
implementation.
