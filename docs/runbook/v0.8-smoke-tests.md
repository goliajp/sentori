# v0.8 smoke tests

End-to-end verification for every feature that landed in v0.8.0 →
v0.8.4. Run these after deploying server + after publishing each SDK
that the host app is on. Each section is independent — skip what
isn't relevant to the host.

Assumptions throughout:
- Server reachable as `https://api.sentori.golia.jp` (production) or
  `http://localhost:18080` (dev). Set `SENTORI_API` accordingly.
- Project token (ingest) lives in `SENTORI_TOKEN`.
- Admin token lives in `SENTORI_ADMIN_TOKEN`.
- Project UUID lives in `SENTORI_PROJECT_ID`.
- Dashboard URL: `https://app.sentori.golia.jp` (or local equivalent).

---

## 1. Native crash end-to-end (v0.8.5)

**Goal**: an iOS `NSException` or Android uncaught Java/Kotlin throw
fired in a release-mode host app lands in the dashboard with
symbolicated frames within ~60 s of the next launch.

**Pre-req**: host app is built with `@goliapkg/sentori-react-native >=
0.7.6` and `initSentori({...})` runs at boot.

### iOS

1. **Run the build that's wired to dSYM upload**. If you haven't done
   that yet, follow `docs/sdk-react-native.md` § Native debug symbols.

2. **In the app** (a debug button or a hidden dev menu):

   ```ts
   import { triggerNativeCrash } from '@goliapkg/sentori-react-native'
   triggerNativeCrash()
   ```

   The app will close (real `NSException`).

3. **Relaunch the app**. SDK reads
   `<Documents>/sentori/pending/*.json` on init, posts each event to
   `/v1/events`, deletes the file.

4. **Verify on server**:

   ```bash
   curl -s "$SENTORI_API/v1/events/_recent?kind=error&limit=5" \
     -H "Authorization: Bearer $SENTORI_TOKEN" | jq '.[]
     | select(.platform == "ios")
     | { id, error_type: .error.type, ts: .timestamp }'
   ```

   Expect one row with `error.type == "SentoriTestException"`.

5. **Verify on dashboard**: navigate to the issue, open the Stack
   tab. Each frame should show `function (file:line:col)` — if frames
   are raw hex addresses, the dSYM didn't upload for this build's
   debug-id (see § Native debug symbols).

### Android

Same flow; the dev-only crash triggers a `RuntimeException` and the
mapping path mirrors iOS. Use `--release` matching `init({ release })`
on the mapping upload.

### Failure modes to check

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| No event arrives after relaunch | Token mismatch / offline queue full | Check `SENTORI_TOKEN` matches `init({ token })`. Tail server logs for `event accepted` |
| Event arrives but stack frames are raw hex | dSYM / mapping not uploaded, or wrong `--release` | `sentori-cli upload dsym <path>` with matching release |
| Event arrives, dashboard says `releaseHasMap: true` but still raw | Debug-id mismatch (CI rebuilt with same version string) | Inspect `dwarfdump --uuid` and re-upload the right slice |

## 2. Analytics dimensions (v0.8.0)

Verify the SDK is shipping the new device fields and the server is
storing + showing them.

### Trigger

In the host app, fire a synthetic non-fatal error:

```ts
import { captureException } from '@goliapkg/sentori-react-native'
captureException(new Error('smoke-v0.8.0-analytics'))
```

### Verify

Server side:

```bash
curl -s "$SENTORI_API/v1/events/_recent?limit=1" \
  -H "Authorization: Bearer $SENTORI_TOKEN" | jq '.[0]
  | { os: .device.os, locale: .device.locale,
      networkType: .device.networkType, geo: .geo }'
```

Expected (RN host):
- `device.locale` populated — `"en_US"` or similar from
  `NativeModules.SettingsManager` (iOS) / `I18nManager` (Android).
- `device.networkType` populated when `@react-native-community/netinfo`
  is installed — `"wifi"` / `"4g"` / etc. When the peer isn't
  installed, this field is absent (not an error).
- `geo` populated server-side when `SENTORI_GEOIP_DB_PATH` points at
  a valid `.mmdb`. Field is `{ country, region?, city? }`; only
  country is guaranteed (DB-IP Lite is country-only). When the db
  isn't configured this field is absent.

Dashboard: open the issue, see the Context pane — locale + networkType
+ geo rows render only when non-null.

### Route dwell time

For RN apps using `useTraceNavigation`, navigate between two screens
in the app, then fire `captureException`. Verify:

```bash
curl -s "$SENTORI_API/v1/events/_recent?limit=1" \
  -H "Authorization: Bearer $SENTORI_TOKEN" | \
jq '.[0].breadcrumbs[] | select(.type == "navigation") | .data'
```

Expected: `{ "dwellMsPrev": <number> }` on the entering breadcrumb.
Also check the `react.navigation` span (in the matching trace) carries
a `nav.dwell_ms` tag.

### GeoIP db (auto-bundled since v0.8)

The Docker image ships DB-IP Lite IP-to-Country at `/app/data/geo.mmdb`
out of the box — `Dockerfile.server`'s `geoip` stage downloads it at
build time. No operator action needed; `event.geo` populates on every
ingested event with a non-private client IP.

To refresh in CI, pass `--build-arg DBIP_VERSION=YYYY-MM` to
`docker build` (default: current month, falls back to previous month
if not published yet).

To swap in a GeoLite2 City db for region + city precision:

```bash
SENTORI_GEOIP_DB_PATH=/path/to/GeoLite2-City.mmdb
```

To disable enrichment entirely (e.g. in a privacy-conscious deployment):

```bash
SENTORI_GEOIP_DB_PATH=/nonexistent   # any unreadable path
```

## 3. User feedback (v0.8.2)

```ts
import { sentori } from '@goliapkg/sentori-react-native'

await sentori.sendUserFeedback({
  title: 'smoke-test',
  body: 'verifying v0.8.2 flow',
  email: 'tester@example.com',
})
```

Server side:

```bash
curl -s "$SENTORI_API/admin/api/projects/$SENTORI_PROJECT_ID/user-reports" \
  -H "Authorization: Bearer $SENTORI_ADMIN_TOKEN" | \
jq '.[0] | { title, body, email, receivedAt }'
```

Dashboard: open any issue, switch to the **User reports** tab. If you
passed `eventId` in the call, the report shows up tied to that
event's issue. Reports without `eventId` are visible only via the
admin endpoint for now (the Inbox UI is deferred).

## 4. Custom metrics (v0.8.3)

```ts
import { sentori } from '@goliapkg/sentori-react-native'

sentori.recordMetric('smoke.checkout.attempts', 1, { method: 'card' })
sentori.recordMetric('smoke.checkout.latency_ms', 123)
sentori.recordMetric('smoke.checkout.attempts', 1, { method: 'wallet' })

// Force a flush — by default points wait up to 30 s.
await sentori.flushMetrics()
```

Server side:

```bash
curl -s "$SENTORI_API/admin/api/projects/$SENTORI_PROJECT_ID/metric-names" \
  -H "Authorization: Bearer $SENTORI_ADMIN_TOKEN" | jq '.'

curl -s "$SENTORI_API/admin/api/projects/$SENTORI_PROJECT_ID/metrics?name=smoke.checkout.attempts" \
  -H "Authorization: Bearer $SENTORI_ADMIN_TOKEN" | jq '.[0:3]'
```

Expected: two distinct metric names + several recent points each. Tag
filter is per-point in the JSON.

Dashboard: **Monitor → Metrics**. Both metric names appear in the left
rail; clicking each shows the recent-points table + a sparkline.

## 5. Cert monitor (v0.8.4)

### Add a domain to the watchlist

Dashboard route: **Monitor → Cert monitor**. Type a domain you control
(e.g. `sentori.golia.jp`) and click **Watch**.

Or via API:

```bash
curl -s -X POST \
  "$SENTORI_API/admin/api/projects/$SENTORI_PROJECT_ID/cert-monitor/domains" \
  -H "Authorization: Bearer $SENTORI_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"domain": "sentori.golia.jp"}'
```

### Wait for the first poll

The background task ticks every 10 min, with a 30 s warm-up at server
startup. After the first tick:

```bash
curl -s "$SENTORI_API/admin/api/projects/$SENTORI_PROJECT_ID/cert-monitor/observations" \
  -H "Authorization: Bearer $SENTORI_ADMIN_TOKEN" | jq 'length'
```

Should be `> 0` (every domain has historical certs in crt.sh).

### Verify the notification

Trigger a fresh issuance: renew an LE cert on the watched domain (or
issue a new one for any subdomain). Within ~10-15 min after the next
poll tick:

- `cert_observations` table has a new row
- Subject **`[Sentori] New cert observed for <domain>`** lands in the
  inbox of every email on the project's `notification_recipients`
  list with `on_new_issue = TRUE`

### Failure modes

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| Empty `observations` after 15 min | server can't reach crt.sh / pool not configured | tail server log for `cert-monitor: domain poll failed` |
| Observations row but no email | notification_recipients empty for project / SMTP misconfig | re-check recipients + `notifier` log |
| Duplicate observation rows | this shouldn't happen — UNIQUE (project_id, cert_id) drops re-polls | report; means the migration didn't apply |
