# v1.0 fresh deploy — re-seed + superadmin + OAuth wiring

What to do once you (the operator) decide to drop the existing
dogfood data, seed `lihao@golia.jp` as superadmin, create the
`qualcomm-insight` project, and turn on Google + GitHub sign-in.

> **Note** — this runbook predates the v2.4 single-domain
> consolidation. Subdomain URLs in env-var examples and log lines
> have been refreshed to the current `sentori.golia.jp/main` shape;
> the OAuth-callback path stays `/auth/oauth/{provider}/callback`
> as before. The Caddyfile sample in §"Caddy / front-door routing"
> below is **superseded** — for the actual production block see
> [`docs/design/single-domain-routing.md`](../design/single-domain-routing.md).

## 1. Compose env — add three new sections

In your `goliajp/devops/sentori/compose.yml` (or wherever the
sentori-server service is defined), extend `environment` with:

```yaml
environment:
  # Existing entries stay.
  SENTORI_DEV_TOKEN: "st_pk_dev0000000000000000000000"
  SENTORI_ADMIN_PASSWORD: "<redacted>"
  SENTORI_BASE_URL: "https://sentori.golia.jp"
  DATABASE_URL: "postgres://..."

  # v1.0 — seed the operator superadmin. Server creates this user
  # at boot if missing; idempotent on subsequent restarts.
  SENTORI_SUPERADMIN_EMAIL: "lihao@golia.jp"
  SENTORI_SUPERADMIN_DISPLAY_NAME: "Lihao"

  # v1.0 — first real project. Server creates it at boot under
  # the named org (creates the org owned by the first superadmin
  # if it doesn't exist yet). No-op if the project already exists.
  SENTORI_SEED_PROJECT_NAME: "qualcomm-insight"
  SENTORI_SEED_PROJECT_ORG_SLUG: "qualcomm"
  SENTORI_SEED_PROJECT_ORG_NAME: "Qualcomm"

  # v1.0 — OAuth credentials. Configure both apps' callback to:
  #   https://sentori.golia.jp/auth/oauth/{github,google}/callback
  # (v2.4: was https://app.sentori.golia.jp/auth/oauth/...; the
  #  host moved, the path stayed the same)
  SENTORI_GITHUB_CLIENT_ID: "..."
  SENTORI_GITHUB_CLIENT_SECRET: "..."
  SENTORI_GOOGLE_CLIENT_ID: "..."
  SENTORI_GOOGLE_CLIENT_SECRET: "..."
```

## 2. Wipe + restart (destructive)

The existing DB has dogfood data. Drop the volume to start clean:

```bash
docker compose down -v
docker compose up -d
docker compose logs sentori-server | grep -E "seeded|superadmin"
```

Expected log lines:

```
postgres connected, migrations applied, dev project seeded
superadmin seeded     email=lihao@golia.jp user_id=<uuid>
superadmin reset-password link (valid 48 h)  link=https://sentori.golia.jp/reset-password/<token>
seed project ensured  org_slug=qualcomm project_name=qualcomm-insight
```

Open that reset link in a browser → set your password → sign in
at `https://sentori.golia.jp/login`. You should see a new
"Superadmin" item in the sidebar's Operator group.

## 3. OAuth app registration (once each)

You only need to do this if you want sign-in-with-GitHub / Google
to work for everyone. The dashboard hides the buttons automatically
when the env vars are missing.

### GitHub

1. <https://github.com/settings/developers> → OAuth Apps → New OAuth App
2. Application name: `Sentori` (anything you want)
3. Homepage URL: `https://sentori.golia.jp`
4. Authorization callback URL: `https://sentori.golia.jp/auth/oauth/github/callback`
5. Save → copy Client ID + generate a Client Secret
6. Paste into the compose env as
   `SENTORI_GITHUB_CLIENT_ID` / `SENTORI_GITHUB_CLIENT_SECRET`
7. `docker compose up -d` to roll the server with the new env

### Google

1. <https://console.cloud.google.com/apis/credentials> → Create Credentials → OAuth client ID
2. Application type: Web application
3. Name: `Sentori`
4. Authorized JavaScript origins: `https://sentori.golia.jp`
5. Authorized redirect URIs: `https://sentori.golia.jp/auth/oauth/google/callback`
6. Save → copy Client ID + Client Secret
7. Paste into compose env, roll the server

The first time `lihao@golia.jp` signs in with one of these
providers, the OAuth subject is account-linked to the seeded user
row — you don't lose superadmin.

## 4. Verify

After step 2 (and optionally step 3):

```bash
# Server is up, migrations applied, seed ran clean
curl -s https://sentori.golia.jp/healthz | head -1

# /auth/oauth/providers reports whichever ones you configured
curl -s https://sentori.golia.jp/api/auth/oauth/providers
# → {"github":true,"google":true}  (or false until step 3)

# Dashboard shows the Superadmin link in the sidebar for lihao
open https://sentori.golia.jp/login
```

Once you're signed in as the superadmin:

- `/superadmin/users` lists every account
- `/superadmin/orgs` lists every org + member/project counts
- `/superadmin/projects` lists every project + event-30d count
- Per-row "grant / revoke" superadmin toggle works (your own row is
  disabled so you can't lock yourself out of god-mode mid-session)
- The Settings → Members / Teams / Projects forms work on the
  `qualcomm` org since you're its owner via the seed

## Caddy / front-door routing

The Caddyfile on the public ingress (t01) needs **three** sentori
host blocks, plus a non-obvious `/auth/*` rewrite on `app.sentori`.

```caddy
app.sentori.golia.jp {
    encode gzip
    handle /api/*    { reverse_proxy lx64:18080 }
    handle /admin/*  { reverse_proxy lx64:18080 }
    # OAuth callback URLs registered at GitHub / Google point at
    # /auth/oauth/{provider}/callback (no /api/). Server mounts them
    # under /api/auth/oauth/*. Rewrite the prefix before forwarding.
    handle /auth/* {
        uri replace /auth/ /api/auth/
        reverse_proxy lx64:18080
    }
    handle           { reverse_proxy lx64:18800 }
}

api.sentori.golia.jp     { reverse_proxy lx64:18080 }
ingest.sentori.golia.jp  { reverse_proxy lx64:18080 }
```

Skip the `/auth/*` block and any OAuth login attempt will bounce back
to `/login` silently — the callback HTML falls through to the SPA
container, the SPA has no route for it, and the auth state never
persists.

## What this doesn't do

- Doesn't issue Insight a project token automatically. After step 2,
  sign in, go to **Organize → Integrate** in the sidebar (or
  `/org/{slug}/integrate?project=…`), `+ mint token` with a label,
  copy the secret on the one-time reveal, and hand it to the Insight
  team. The same view shows the SDK install + init snippet with the
  token already substituted in.
- Doesn't auto-invite anyone. Inviting teammates to the org is
  `Settings → Members → + invite member` once the org exists.
- Doesn't pre-fill OAuth secrets. `gh secret set
  SENTORI_{GITHUB,GOOGLE}_CLIENT_{ID,SECRET}` after step 3 so the
  deploy template plugs them into `/apps/sentori/.env` on every
  redeploy. The next workflow run picks them up; in the meantime
  patch `.env` directly + `docker compose up -d server`.

## What's destructive about step 2

`docker compose down -v` drops the postgres volume. Everything goes:

- all existing events / issues / traces
- all existing attachments (screenshots / wireframes / state)
- all existing users / orgs / teams / projects (including the
  built-in `dev` org/project — migration 0007 re-seeds those)
- all existing source maps / dsyms / mappings

Make a `pg_dump` first if any of that matters to you.
