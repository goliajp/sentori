## Walker fix — anchor at content view

```kotlin
val decor = activity.window?.decorView
val root = decor?.findViewById<View>(android.R.id.content)
```

PhoneWindow injects `StatusBarBackground` and `NavigationBarBackground` View children directly inside the decorView (siblings of the content FrameLayout) to paint the system bar tints. After rc.3's deep-walk fix these started getting traversed; on devices with edge-to-edge insets, those views report `width = displayWidth` and absolute window coordinates, which then painted as full-width bars in the rendered SVG.

Anchoring at `android.R.id.content` skips them. The viewport metadata (`width`, `height` in the payload) now reflect the content FrameLayout's measurement, which is what the React tree paints into anyway.

**Side effect**: nodes are now expressed relative to the content view's origin. For most apps this is identical to decorView origin (both at 0,0). For apps with non-trivial action bars or system-window-overlap they'd shift by `statusBarHeight`. Verify your CSS overlay positions if you read the wireframe coordinates programmatically.

## Drawable coverage

```kotlin
is StateListDrawable -> {
  extractDrawableColor(drawable.current)
}
is BitmapDrawable -> null  // walker branches to kind=image
is LayerDrawable -> { /* recurse — covers RippleDrawable, CompositeBackgroundDrawable */ }
else -> extractByReflection(drawable)  // getBackgroundColor() / getColor()
```

Cumulative type coverage as of rc.8:

| Drawable | Branch | Colour extraction |
|---|---|---|
| `ColorDrawable` | direct | `.color` |
| `GradientDrawable` | direct | `.color?.defaultColor` |
| `StateListDrawable` (Pressable / TouchableOpacity) | direct | `.current` → recurse |
| `BitmapDrawable` (raster background) | direct | none; emits `kind: image` |
| `RippleDrawable` (modern Pressable feedback) | LayerDrawable | recurse layers |
| `CompositeBackgroundDrawable` (RN 0.74+ `backgroundColor + borderRadius`) | LayerDrawable | recurse → reflection |
| `BackgroundDrawable` (inner RN layer holding the paint colour) | reflection | `getBackgroundColor()` |
| Other custom Drawables | reflection | `getBackgroundColor()` / `getColor()` |

## Dashboard SVG clip

```jsx
<svg overflow="hidden" style={{ overflow: 'hidden' }} viewBox={`0 0 ${w} ${h}`}>
  <defs>
    <clipPath id="wf-viewport-clip">
      <rect height={h} width={w} x={0} y={0} />
    </clipPath>
  </defs>
  <g clipPath="url(#wf-viewport-clip)">
    {nodes.map(...)}
  </g>
</svg>
```

Both the `<svg overflow="hidden">` attribute and the inline `style.overflow='hidden'` guard against Safari's historical lax clipping behaviour; the `<clipPath>` is the spec-compliant per-element clip. Even if a stray walker emission with `x + w > viewport.width` slips through, no pixel paints outside the device frame in the viewer.

Applied to both `ReplayPlayer.tsx` (issue-detail inline player) and `replay-tab.tsx` (dedicated Replay tab) paths.
