# Insight followup — rc.6 closes the body-size ceiling

Date: 2026-05-18 (~06:35 JST)
SDK: `@goliapkg/sentori-react-native@1.0.0-rc.6` (publish to npm `@next`)
Server: `1.0.0-rc.1` builds with the new ingest body limit on the attachment endpoint.

## TL;DR

Your rc.4/rc.5 re-verify caught the right thing: the client chain is fully correct, but **the prod ingest server cut the 1.5 MB replay POST off at the door**. The global axum body limit was 1 MB; dense-UI replay payloads (82 nodes × 60 frames) blow through it.

rc.6 ships two coordinated changes:

1. **Server** (`server/src/router.rs`) — per-route body limit on `POST /v1/events/{eid}/attachments/{kind}` raised to **16 MB** (vs. the 1 MB global default). The 1 MB cap stays on every other ingest endpoint.
2. **SDK** (`sdk/react-native/src/transport.ts`) — uploads now surface the HTTP status in dev logs so future failures are diagnosable at a glance. Your `replay upload returned null status=?` request lands as a separate `[sentori] attachment upload non-2xx eventId=… kind=replay status=NNN` line right above the existing one.

## Server fix

```rust
const MAX_INGEST_ATTACHMENT_BYTES: usize = 16 * 1024 * 1024;

.route(
    "/v1/events/{event_id}/attachments/{kind}",
    post(api::attachments::upload).layer((
        DefaultBodyLimit::disable(),
        RequestBodyLimitLayer::new(MAX_INGEST_ATTACHMENT_BYTES),
    )),
)
```

Per-route layer pattern (same shape as the existing dsym / mapping / sourcemap admin uploads at 256 MB). Keeps every other ingest endpoint at the 1 MB ceiling so abusive payloads still bounce there. 16 MB ≈ 10× your observed dense-UI payload (1.5 MB) — leaves headroom for replay-heavy screens without inviting silly clients.

Body limits this size are normal for diagnostics ingest stacks; Sentry's analog accepts up to 100 MB per attachment. We hold a tighter cap because the v0.9 SDK doesn't yet chunk and 16 MB is plenty for the typical replay window.

## SDK fix

`transport.ts` `uploadAttachment` was returning `null` on any non-2xx without surfacing the status to logcat. The breadcrumb already carried `http_NNN` but that's not visible until the next event ships. New dev-only logs at each failure shape:

```
[sentori] attachment upload non-2xx eventId=… kind=replay status=413
[sentori] attachment upload bad-response-body eventId=… kind=… status=…
[sentori] attachment upload fetch threw eventId=… kind=… reason=fetch_TypeError
```

The existing `[sentori] replay upload returned null` line continues to fire after these — read the two together. Same pattern applies to screenshot / sessionTrail uploads.

## What you should see on rc.6 (after the server-side fix is also deployed)

```
[sentori] captureException eventId=… breadcrumbs=N wantScreenshot=true wantSessionTrail=true wantReplay=true
[sentori] screenshot blob ok, uploading mediaType=image/webp base64Bytes=...
[sentori] enqueue eventId=… attachments=3 kinds=screenshot,sessionTrail,replay
```

Dashboard issue-detail Session Replay tab should render with the brand colours rc.5 captures plus the new hairline grid pattern underneath (more on that below) so even pre-rc.5 events read as "captured area, just no colour data" rather than "loading failed".

## Dashboard side — grid pattern + decoupled fallback opacity

Independent of the rc.6 SDK but shipping at the same time:

- The wireframe SVG now paints a hairline grid (`var(--rule)` @ 0.8 stroke-opacity, 1-px non-scaling stroke) underneath every frame. Always-visible structural indicator.
- Fallback rect fill (when `node.color` is missing — rc.3 era events) now renders at **0.12** opacity instead of 0.9; SDK-coloured rects stay at 0.9. Grid reads through the fallback, brand colours stay saturated.

Captures from rc.5 onward show coloured CTAs at the higher opacity; older captures' uncoloured rects now resolve as "structural ghost over grid", not "flat white block".

## Caveats

- **Server fix needs to deploy before rc.6 SDK is useful.** CI is running; deploy should land within a few minutes of this commit. If you `bun add @goliapkg/sentori-react-native@1.0.0-rc.6` before the server deploys, you'll still see `status=413` in the logcat — the new diagnostic is correctly telling you what's happening, but the prod ingest is the bottleneck until the server image cycles.
- **Old captured wireframe NDJSONs** stored with `node.color` absent for rect kind don't retroactively gain colour. Only events captured under rc.5+ Android natives have colour data; events captured under rc.3 native (your 2026-05-17 dataset) will keep rendering as fallback-translucent over grid.
