Verify rig: Samsung Galaxy S22 (SM-S9010), Android 16, real device over USB; local mock ingest on `localhost:8080` (adb reverse).

## TL;DR

rc.4 fixes the **`captureException → enqueue` doesn't include `replay` kind** finding from your 2026-05-18 ~05:15 JST report. After your install, the captureException log line will include `wantReplay=true` and the enqueue log will read `kinds=screenshot,sessionTrail,replay`.

## Root cause — UTF-8 NDJSON tripped Hermes' spec-compliant `btoa`

The pre-rc.4 replay attachment upload path was:

```ts
async function captureAndAttachReplay(event, ndjson) {
  try {
    const base64 = globalThis.btoa(ndjson)   // ← unsafe
    const meta = await uploadAttachment(event.id, 'replay', { base64, ... })
    if (meta) event.attachments.push(meta)
  } catch { /* best-effort */ }              // ← silent
}
```

`globalThis.btoa` is **Latin-1 only** (WHATWG spec, Hermes matches). Any code point > 0xFF throws `InvalidCharacterError` synchronously. The catch swallowed the throw, the upload never fired, and `event.attachments` stayed empty.

This is why rc.2's intermittent success worked at all: rc.2's Android walker bailed at the first zero-size wrapper, so the NDJSON contained only top-level View boxes (no text, no UTF-8). rc.3 fixed the walker — and immediately surfaced TextView content. Your Insight app has Japanese strings (and even an English app would hit this on em-dash / smart quotes / emoji); the deep walk produced UTF-8 NDJSON; `btoa` threw; attachment silently disappeared.

Same code path runs on iOS, but your iOS dev-panel verify event was on an English-only screen at the moment of capture — pure ASCII — so it worked by coincidence.

## Fix

`sdk/react-native/src/base64.ts`:

```ts
export function base64Utf8(s: string): string {
  if (typeof globalThis.btoa === 'function') {
    return globalThis.btoa(unescape(encodeURIComponent(s)))
  }
  return Buffer.from(s, 'utf8').toString('base64')
}
```

`unescape(encodeURIComponent(s))` rewrites the UTF-8 byte sequence into a Latin-1-equivalent string that `btoa` can chew. Same pattern the sessionTrail path was already using (which is why your sessionTrail attachments worked on rc.3 even with deep walk).

All three JSON-attachment sites in `capture.ts` now route through this helper:
- `captureAndAttachReplay`
- `captureAndAttachStateSnapshots` (had the same latent bug — would surface on any JP state)
- `captureAndAttachSessionTrail` (already safe inline; folded into helper for consistency)

## Diagnostics rc.4 adds (so the next failure has a visible shape)

The pre-rc.4 silent catch is gone. The new dev-only log surface:

| Log line | Means |
|---|---|
| `[sentori] captureException ... wantReplay=true` | replay subsystem is running at capture time (rc.4 added `wantReplay` to the existing line — see your "log line missing wantReplay" point) |
| `[sentori] replay drain empty (no frames buffered at captureException)` | replay was on but ring drained zero frames. **Different bug** from "encoding failed". |
| `[sentori] replay attachment threw eventId=... ndjsonBytes=N e=...` | base64 step or fetch threw; the throw is now logged with the ndjson size for triage |
| `[sentori] replay upload returned null eventId=... ndjsonBytes=N` | the server responded non-201. Check the ingest /events/<id>/attachments/replay POST response. |
| `[sentori] enqueue ... kinds=screenshot,sessionTrail,replay` | success — all three landed |

Pre-rc.4 you only ever saw the captureException line and then the enqueue line; the failure was invisible.
