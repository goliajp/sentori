# Sentori → Insight: rc.3 — Android replay walker recurses through zero-size wrappers

Date: 2026-05-18
Cuts: `@goliapkg/sentori-react-native@1.0.0-rc.3` (publishes to npm `@next` after S22 verify lands)
Triggered by: 2026-05-18 ~04:42 JST post-clean-rebuild section in
`feedback-to-sentori-20260517-android-findings.md`

## Diagnosis

Your three findings, separated:

### (A) `captureWireframe` intermittently returns null on rc.2 — confirmed bug, fixed

Root cause: `SentoriReplayCapture.kt walk()` did

```kotlin
val w = view.width
val h = view.height
if (w <= 0 || h <= 0) return    // ← returned ENTIRELY, including no recursion
```

For an RN Fabric tree, intermediate shadow-tree wrappers occasionally
measure to zero size (lazy layout, paint phase, ScrollView content
not yet laid out). The walker treated that as "skip everything below"
when it should have been "skip emitting this node but keep walking the
subtree". Result: the captured wireframe had only the decorView +
2-3 padding wrappers, the entire React tree under them silently
dropped.

iOS already handled this correctly — the iOS walk() comment literally
reads `invisible container — skip emitting but recurse`. The bug was
Android-only and undocumented.

### (B) "Native returned null" warning fired too narrowly on rc.2 — fixed by adding a thin-result warning

The rc.2 diagnostic only fires when `snapshot === null` or
`snapshot === ''`. A non-empty JSON payload with `nodes: []` (the
shape produced when walk() returned without recursing) looks like a
success to JS — pushes to ring, no warning. You saw this as
"rc.2 把诊断警告砍了" but the warning code is intact; it just doesn't
match the failure mode you hit.

### (C) Suggested rc.3 diagnostic surface — adopted, with shape changes

* **Stride-based "thin result" warning** (1st, 10th, 100th, ... ticks
  where `nodes < 6` — the threshold that catches "walker bailed
  early" without spamming on tiny screens). Coexists with the
  null/empty warning, doesn't replace it.
* **First-good-tick log**:
  `[sentori] replay tick: first ok — nodes=N sizeBytes=B` — devs see
  a single positive proof that the capture is healthy.
* **`probeWireframe()`** extended with:
  - `lastDepthMax` — max recursion depth reached by walk(). RN apps
    typically hit 20-40; if probe shows 2-3 the walker is bailing
    early, regardless of node count.
  - `lastSizeBytes` — byte length of the last serialised payload.
    ~50 bytes / node is typical; 1 KB for 800 nodes would be a flag.
  - `totalTicks` / `totalEmptyResultTicks` — lifetime counters.
    Failure rate = `totalEmptyResultTicks / totalTicks`.
* All four fields are now on the JS-side `probeNativeWireframe()`
  return type too; no need to read `raw` to get them.

## Bonus

Your GOL-589 (`java.lang.IllegalStateException ReactViewGroup
contains null child`) keeps firing — that's your RN-side bug not
ours, and we expect to keep capturing it cleanly. The bonus value-
add: the rc.3 replay walker is now structurally tolerant of zero-
size wrappers, which is the same shape of brittleness that produces
the `null child at index 3` exception on your end. Worth a look at
the affected ViewGroup hierarchy when you triage GOL-589 — the same
"View was removed mid-traversal" race that bites your dispatch path
might be hitting the SDK walker too, just we're now more robust to
it.
