# Insight followup — rc.5 puts colour back into Android wireframes

Date: 2026-05-18 (~06:10 JST)
SDK: `@goliapkg/sentori-react-native@1.0.0-rc.5` (publish to npm `@next`)
Verify rig: same Samsung Galaxy S22 SM-S9010 / Android 16 over USB; local mock ingest on `localhost:8080` with adb reverse.

## TL;DR

rc.5 closes a v0.9.6-era omission in the Android walker: the rect kind was emitting `null` for `node.color`, so the dashboard wireframe rendered every coloured ViewGroup as a neutral grey block. iOS already had this; Android now matches. Insight should see brand colours (blue / orange / red CTAs etc.) populate the Session Replay viewport after a full APK rebuild on rc.5.

**This is a native change — `bun add @goliapkg/sentori-react-native@1.0.0-rc.5` alone is not enough. You need a full prebuild + APK rebuild for the new Kotlin to take effect.** iOS is unchanged.

## Root cause

The Android walker file `SentoriReplayCapture.kt` had this branch:

```kotlin
view.background != null -> {
  node.put("kind", "rect")
  // Background drawables don't always expose color directly.
  // Skip color for non-ColorDrawable; renderer falls back to neutral.
  kindEmitted = true
}
```

The "skip and let the renderer fall back to neutral" decision was load-bearing in v0.9.6 when the renderer fallback worked at 0.06 alpha (invisible anyway) and nobody noticed. After rc.3's walker fix surfaced the deep ViewGroup tree, rc.4's encoding fix landed the replay attachment, and the rc.4/rc.5 dashboard palette bumped fill-opacity to 0.9 — the absence of `node.color` became visible: every rect rendered as a flat white block, regardless of the host app's actual brand palette.

## Fix

`sdk/react-native/android/src/main/java/com/sentori/SentoriReplayCapture.kt`:

```kotlin
view.background != null -> {
  node.put("kind", "rect")
  val color = extractDrawableColor(view.background)
  if (color != null && (color shr 24 and 0xff) != 0) {
    node.put("color", colorToHex(color))
  }
  kindEmitted = true
}

private fun extractDrawableColor(drawable: Drawable?): Int? {
  return when (drawable) {
    null -> null
    is ColorDrawable -> drawable.color
    is GradientDrawable -> drawable.color?.defaultColor
    is RippleDrawable -> {
      for (i in 0 until drawable.numberOfLayers) {
        val inner = drawable.getDrawable(i)
        val c = extractDrawableColor(inner)
        if (c != null && (c shr 24 and 0xff) != 0) return c
      }
      null
    }
    else -> null
  }
}
```

Covers the three RN-common cases:
- `ColorDrawable` → flat `<View style={{ backgroundColor: '#xxx' }}>` without rounding.
- `GradientDrawable` → a `<View>` with `backgroundColor` + `borderRadius` or `borderWidth` — RN's most common shape since RN 0.74. `.color.defaultColor` is the ColorStateList default; gradients with multiple stops surface a representative.
- `RippleDrawable` → `<Pressable>` wrappers; recurses into the inner painted layer.

`StateListDrawable`, `LayerDrawable` (other than RippleDrawable), and `BitmapDrawable` fall through to neutral. The alpha-0 filter mirrors the iOS `!= "#00000000"` check so fully-transparent backgrounds keep falling back.

## Dashboard side — fill-opacity bumped to 0.9

Along with rc.5, `web/src/lib/wireframe-palette.ts` bumps:
- `WIREFRAME_RECT_OPACITY` 0.75 → **0.9** (overlap still composes in the remaining 10% headroom)
- `WIREFRAME_IMAGE_OPACITY` 0.9 → **1.0** (media regions read as solid blocks)

`WIREFRAME_TEXT_FILL` stays as `var(--ink)` — wireframes are structural diagrams; content readability beats colour fidelity.

This part is dashboard-only and ships independently of any SDK version.

## What's still NOT covered

- **Older events stay grey.** rc.5 only affects newly captured wireframes. The 2026-05-17 events you have already in your dashboard were captured under rc.3 native and have no `node.color` data for rect kinds — the dashboard can't render what isn't there.
- `StateListDrawable` (Material checkbox / switch wrappers) and `LayerDrawable` outside of RippleDrawable — these still fall back to neutral. If you see specific grey blocks that should be coloured, paste the wireframe NDJSON node and we'll trace it.
- Text colour fidelity stays ink-only intentionally; if you want SDK-emitted text colours to win, that's a one-line dashboard change but please flag if you'd actually use it.
