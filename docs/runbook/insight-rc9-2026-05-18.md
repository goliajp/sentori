# Insight followup — rc.9 ships replay encoding v2 (keyframe + delta)

Date: 2026-05-18 (~09:40 JST)
SDK: `@goliapkg/sentori-react-native@1.0.0-rc.9` (publish to npm `@next`)
Dashboard: same deploy cycle.

## TL;DR

Replay attachment switches from "one full snapshot per tick" to **keyframe + delta** NDJSON. Same 60 s pre-error window, ~50 % fewer wire bytes, **4 Hz native capture (was 1 Hz)** so the dashboard player feels like motion rather than a slideshow. **Breaking wire format** — there is no v1↔v2 compat shim baked into the SDK; the dashboard parser auto-detects (rc.8 NDJSON still plays as flat keyframes).

Full schema: `docs/replay-encoding-v2.md`.

## Why the rework

Insight's rc.6 verify showed a 1.5 MB / 60 s NDJSON for ~80 nodes — comfortably under the new 16 MB server cap but already at the lower end of "could this be faster?" Three things motivated v2:

1. **Throughput headroom**. Half the bytes leaves room for richer per-node data (motion, animation) without crossing the cap.
2. **Visual smoothness**. 1 Hz capture renders as discrete steps. 4 Hz + cross-fade interpolation renders as motion — better debugging UX, more like sentry replay or rrweb.
3. **Scrubbing**. Time-axis seek (`0.0 s / 60.0 s`) reads correctly; "frame 23 of 42" required mental math.

## Sample-rate change

| | rc.8 | rc.9 |
|---|---|---|
| Native tick interval | 1000 ms | **250 ms (4 Hz)** |
| Source samples / 60 s | 60 | 240 |
| Playback render | step (one state) | **rAF @ ~60 fps, cross-fade** |
| `replay.hz` option | yes | yes (default raised) |
| `replay.keyframeMs` option | n/a | new — default 4000 ms |

Pass `replay.hz: 2` for cheaper capture (motion smoothness drops); `replay.hz: 8` for higher fidelity on motion-heavy apps (bytes still under rc.8 baseline because of delta compression).

## Wire format

NDJSON, one of two line shapes per emit:

```jsonc
// keyframe
{"ts": 1779000000000, "kind": "key", "width": 1080, "height": 2340, "nodes": [...]}

// delta against the previous emit's reconstructed state
{"ts": 1779000000250, "kind": "delta",
  "added":   [...],                         // new fingerprint
  "changed": [...],                         // same fingerprint, different kind/text/color
  "removed": [{"x":105,"y":618,"w":870,"h":57}]
}
```

Fingerprint is `${x|0},${y|0},${w|0},${h|0}` — same approach the dashboard's old diff overlay used. No backwards compat — rc.9 wire format is *only* what rc.9+ writes.

## Encoder behaviour

A keyframe is emitted when:

1. Replay subsystem starts (cold or after `stopReplay()`).
2. `ts - lastKeyframeTs >= 4000 ms` (default; tunable).
3. The delta would be `≥ 40 %` of the current node count *and* there are `≥ 10` nodes — big screen transitions prefer fresh keyframes over near-rewrite deltas.

Static-UI ticks (no add/change/remove) drop the line entirely — cheaper than the rc.8 byte-equality dedup. Ring buffer evicts by time window (last 60 s), with a 1000-item hard cap.

## Dashboard player rewrite

`/Users/doracawl/workspace/goliajp/sentori/web/src/lib/replay-reconstruct.ts` is the new parse + reconstruct layer:

- `parseLines(text)` — drops malformed lines, returns `Line[]`.
- `class ReplayTimeline` — `reconstructAt(absTs)` memoises the last `(ts, state)` pair so forward scrubbing pays for one or two deltas per render rather than rebuilding from the most-recent keyframe.

`ReplayPlayer` and `ReplayTab` both:

- Seek axis is time (ms). Scrubber reads `0.40 s / 4.20 s`.
- rAF loop drives `playheadRel` from wall-clock so playback speed matches real time regardless of render rate.
- Each rendered frame finds the two bracketing captures, reconstructs both, and cross-fades via stacked `<g opacity>` layers. Matched nodes with identical colour render at full opacity (cumulative); colour-changed nodes visibly cross-fade; added/removed nodes fade in/out.
- The frame-list panel switches to a **keyframe-only** list (clickable seek targets for big state transitions); deltas show up as the animation between keyframes.
- A capture/keyframe tick strip sits beneath the scrubber as visual cadence cue.

Position interpolation (linear-tween x/y for moving nodes) is **not in rc.9**. Without stable node IDs, two fingerprints that differ only by position can't reliably correlate. Cross-fade is the honest visualisation for this data shape. Stable IDs are tracked as future work.

- 11 SDK unit tests covering: first-tick keyframe, static-UI no-op skip, single-node-changed delta shape, added-vs-removed delta correctness, keyframe-overdue forces a fresh key, big-transition heuristic, full reconstruction round-trip, byte budget (60 s × 4 Hz dense UI < 200 KB).

## Old (rc.8 and earlier) events

The dashboard parser's `asV2OrUpgradeV1` shim recognises v1 NDJSON (no top-level `kind`) and treats each line as a standalone keyframe. Archived 2026-05-17 events still play; they just don't have delta entries so the cross-fade never kicks in (renders as a step-frame slideshow). New events captured under rc.9 get the full motion treatment.

## What you need to do on Insight

```
bun add @goliapkg/sentori-react-native@1.0.0-rc.9
bun install
bun android   # full APK rebuild — replay.ts changes are JS-only but
              # rc.5..rc.8 native changes are still required for
              # Drawable colour extraction and content-view anchor.
```

Optional in your `initSentori({ capture: { replay: ... } })`:

```ts
capture: {
  replay: {
    mode: 'wireframe',
    hz: 4,           // default; bump to 8 for motion-heavy screens
    keyframeMs: 4000 // default; lower for finer reconstruction resync,
                     // higher for marginally smaller payloads
  }
}
```

Trigger any captureException. Open the issue's Session Replay tab on the dashboard. You should see:

- Time scrubber reading `0.00 s / X.XX s` rather than frame indices.
- Cross-fade between captures — UI changes smoothly rather than stepping.
- Keyframe tick marks on the strip below the scrubber.

Per-attachment size at typical 4 Hz settings: 100–200 KB for 60 s windows on dense UIs. Well under the 16 MB server cap.
